## Decision

**PASS — the admitted v0.7 Commercial Completion scope is implemented and the
working tree is release-candidate ready.**

One accepted Quote can become a governed Work Order, complete field execution,
issue an immutable Invoice snapshot, collect hosted payment, apply the trusted
provider result, and reach a paid Invoice. Refund confirmation reverses the
allocation and reopens the receivable.

## Delivered scope

- official `runory.invoice` Module with Invoice, lines, and payment allocations;
- governed `invoice.issue_from_work_order` and `invoice.void` Commands;
- atomic `invoice.apply_payment` and `invoice.apply_refund` capability
  Providers used by existing Payment Commands;
- integer-minor-unit amount, balance, currency, overpayment, open-request,
  void, version, and idempotency guards;
- Invoice navigation, list/detail, timeline, issue/void actions, hosted payment
  panel, and refund-aware state;
- outstanding and overdue Invoice dashboard visibility;
- read-only generic API/UI boundaries for financial objects;
- canonical Quote-to-paid-Invoice journey acceptance.

## Proportionality review

No new runtime framework was introduced. The implementation uses the v0.6
Command Contract and Provider mechanism specifically for the two atomic
cross-context writes that require it.

The following attractive but non-essential work was rejected from v0.7:

- Contract/e-signature;
- customer portal and general document sharing;
- tax, accounting, credit-note, and reconciliation subsystems;
- recurring/project/consolidated billing;
- general operational inbox or BI;
- inventory and advanced scheduling;
- Agent-generated official Module/Pack/Template productization.

The retained design is three financial objects, two user-facing Commands, and
two semantic Providers. Each retained part supports an accepted lifecycle,
audit, immutable snapshot, or payment/refund invariant; no speculative
abstraction remains.

## Residual boundaries

- Invoice issue from the standard UI is intentionally available only for a
  completed Quote-origin Work Order.
- Non-Quote Work Orders can use the command with an explicit amount and
  currency, but v0.7 does not build a free-form Invoice editor.
- Customer-facing document access remains a later v1.0 journey requirement.
- Currency presentation currently follows the existing Payment UI convention;
  jurisdiction-specific invoice formatting and currency-exponent policy are
  outside v0.7.
- Hosted production-provider validation remains deployment evidence, not a
  repository release blocker; signed webhook HTTP and provider-boundary tests
  pass locally.

## Release action

tag after final diff review. This evidence does not itself claim that a tag or
