## 1. Purpose

本文档是 Runory Cloud `v0.1.0` 的发布范围、验收门槛和签发记录。它用于回答：

1. `v0.1.0` 具体承诺什么。
2. 哪些能力必须完成，哪些明确不阻塞发布。
3. 每项结论需要什么证据。
4. 谁批准最终 Tag 与 Production release。

本文档不重新定义架构；详细设计以引用的 decision baseline 为准。若发布实现需要改变边界，先更新对应设计文档和 ADR，再修改本发布定义。

## 2. Release Thesis

`v0.1.0` 代表 Runory 从概念验证进入可供受控真实用户使用的 Cloud Early Access：

> Runory 已具备安全承载 Organization/Workspace、持续制造和发布 Module/Pack/Template、通过 Agent 受控扩展业务能力，并在不破坏用户数据和 Workspace Extension 的前提下持续升级的第一套完整 Cloud 基线。

它不是功能完整的 1.0，也不是面向所有企业的大规模 GA。

## 3. Target Users and Channel

目标用户：

- Runory 内部平台研发与运营人员。
- 受邀请的早期 SMB 试用客户。
- 参与 CRM Lite 验证的设计伙伴。

发布渠道：

- 邀请制 Early Access。
- 默认使用 `early_access` Entitlement。
- 不公开承诺生产级 SLA。
- 不开放第三方 Module 发布。

## 4. Canonical v0.1 User Journey

以下路径必须在 Production-like 环境端到端通过：

```text
用户输入邮箱并完成 OTP
→ 创建 Organization + default Workspace
→ 进入生产级 Cloud UI
→ 从 Stable Catalog 安装 CRM Lite Pack
→ 创建和查看 Customer / Contact 数据
→ Agent 提议 Customer custom field
→ 用户查看 Diff 并批准 Apply
→ UI 显示新字段，Audit 记录操作
→ SDK toolchain 构建并提交 runory.customer 新版本
→ 平台验证并发布该 immutable artifact
→ Workspace 生成 Extension Compatibility Report
→ Workspace Admin 批准升级
→ 数据与 Extension 保持可用
```

如果这条路径无法完整执行，`v0.1.0` 不得签发。

## 5. Release Scope

### 5.1 SaaS Core

- Passwordless Email OTP 与服务端 Session。
- User/AuthIdentity、Organization、Workspace。
- Organization/Workspace Membership 与固定 RBAC。
- Invitation、Owner invariant 与即时权限撤销。
- HTTP/MCP/Agent/Job 的统一 RequestContext 与 tenant isolation。
- Append-only Audit、Workspace API Key 和基础安全控制。
- Versioned Platform Migration。
- `early_access` Entitlement 与防滥用基础配额。
- Workspace export、archive、restore、purge 基础。
- 托管数据库 backup 与恢复 runbook。

### 5.2 Platform Runtime

- Metadata-driven Object、Field、View、Form。
- Schema-driven UI。
- Module/Pack install runtime。
- Managed Workspace Extension plan、preview、apply、audit、rollback point。
- Module 与 Extension ownership boundary。
- Governed API shared by UI and Agent。

### 5.3 Catalog & Release Control Plane

- Official/Internal immutable artifact Registry。
- Module/Pack/Template versions。
- Structured Validation Run 与 Sandbox evidence。
- Internal/Beta/Stable release channel。
- Pack dependency resolver 与 frozen lock。
- Workspace install/upgrade compatibility report。
- Allowlisted rollout、pause 与 failure isolation。
- Platform Catalog Console 基础页面。
- Workspace Module Center 基础页面。
- Stable promotion 的人工 Release Manager approval。

### 5.4 Internal SDK Toolchain

- `@runory/sdk` typed canonical Manifest authoring。
- `runory validate/test/build --json` 或等价 CI commands。
- 使用真实 Runtime adapters 的 install/upgrade/Extension fixture harness。
- Reproducible artifact、provenance、checksum 与 secret/path safety。
- `publish --channel internal` candidate adapter，无 Stable bypass。
- Runory Module Skill。

### 5.5 Reference Business Capability

- `runory.customer` Module。
- `runory.contact` Module。
- `crm-lite-pack`。
- `small-business-crm` Template。
- Customer CRUD。
- Contact CRUD 与 Customer relation。
- Dashboard、navigation、empty/loading/error states。
- Customer custom field Extension scenario。

## 6. Explicit Non-goals

以下项目不阻塞 `v0.1.0`：

- Team。
- Custom roles、field ACL、record ACL。
- OIDC、SAML、SCIM。
- 第三方 Marketplace 与 publisher onboarding。
- Cloud code editor 或任意 SQL editor。
- Service Account。
- Advanced Workflow Runtime。
- Seat Billing、usage overage Billing、Add-on。
- Public self-service paid signup。
- Private/VPC/On-premise production delivery。
- Data residency、per-tenant database、customer-managed encryption key。
- SOC 2 / ISO 27001 certification。
- 正式 SLA 和多区域灾备。

Stripe Billing 可以在 `v0.1.0` 后启用；在邀请制免费 Early Access 下，Entitlement、Quota 和 Usage 基础必须存在，但付费闭环不作为 Tag blocker。

## 7. Release Blockers

任何一项成立都阻止 Tag 或 Production release：

1. Production 仍依赖 trusted identity headers 或开发 bootstrap 登录。
2. 存在已知跨 Organization/Workspace 数据读取或写入路径。
3. 已知 Session、OTP、API Key 以明文存储。
4. Platform role 可以隐式访问客户业务数据。
5. Stable Catalog artifact 可以被覆盖或无 checksum 安装。
6. Pack 安装未使用 frozen dependency lock。
7. Workspace upgrade 没有 compatibility preflight。
8. Migration 失败会继续扩大 rollout 或破坏其他 Workspace。
9. 无法从空数据库重放 Platform Migrations。
10. 未完成一次真实 backup restore drill。
11. Canonical v0.1 User Journey 失败。
12. 存在未接受的 P0/P1 数据安全或数据丢失风险。
13. Customer 1.1 Catalog artifact 仍依靠手工拼包，未经过 SDK validate/test/build pipeline。

## 10. Compatibility Promise

`v0.1.0` 在 1.0 之前允许快速演进，但作出以下承诺：

- 不静默丢失用户业务数据。
- Published Catalog Version 不可变。
- Platform Migration 保持可追踪和可重放。
- Workspace Extension compatibility 在 upgrade 前可见。
- Breaking Module upgrade 不自动执行。
- API 和 Manifest 可以演进，但 breaking change 必须有版本和迁移说明。

## 12. Go / No-go Process

### Release Candidate Freeze

- 确定候选 commit。
- 冻结 Platform/Core contracts 和 v0.1 Catalog artifacts。
- 只接受 blocker fix。
- 生成 release evidence bundle。

### Go/no-go Review

按顺序检查：

1. Release Blockers。
2. Acceptance Matrix。
3. Required E2E Scenarios。
4. Restore drill 与 rollback/runbook。
5. Known issues 与 waivers。
6. Sign-off。

### Tag and Deploy

```text
approved release commit
→ git tag v0.1.0
→ build immutable Cloud artifact
→ deploy Production
→ run smoke + canonical journey
→ publish GitHub Release and release notes
```

如果 Tag 后 Production smoke 失败：停止新用户邀请与 Catalog rollout，按 runbook 恢复 last-known-good deployment；不要删除已创建数据或覆盖发布 artifact。

## 15. Post-release Early Access Plan

- 仅邀请受控客户，逐步扩大 cohort。
- 每日检查 auth、tenant denial、API error、migration 和 Catalog rollout 指标。
- 所有 Module upgrade 先 Internal/Beta，再进入 Stable。
- P0 security/data issue 立即暂停新邀请和 rollout。
- 收集真实 Workspace 对 Module/Pack/Template 与 Agent 的使用反馈。
- `v0.1.x` 只做兼容修复、可靠性和小型增强；breaking platform change 进入后续 minor release 并提供 migration。
