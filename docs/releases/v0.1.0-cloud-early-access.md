## 1. Purpose

This document is the release scope, acceptance gate, and sign-off record for Runory Cloud `v0.1.0`. It answers:

1. What `v0.1.0` specifically commits to.
2. Which capabilities must be complete, and which explicitly do not block release.
3. What evidence is required for each conclusion.
4. Who approves the final Tag and Production release.

This document does not redefine architecture; detailed design is governed by the referenced decision baselines. If release implementation needs to change boundaries, first update the corresponding design document and ADR, then update this release definition.

## 2. Release Thesis

`v0.1.0` represents Runory moving from proof of concept into Cloud Early Access usable by controlled real users:

> Runory has the first complete Cloud baseline to safely host Organizations/Workspaces, continuously manufacture and release Modules/Packs/Templates, extend business capability through governed Agents, and continuously upgrade without breaking user data or Workspace Extensions.

It is not a feature-complete 1.0 and not large-scale GA for all enterprises.

## 3. Target Users and Channel

Target users:

- Runory internal platform engineering and operations personnel.
- Invited early SMB trial customers.
- Design partners participating in CRM Lite validation.

Release channel:

- Invite-only Early Access.
- Uses `early_access` Entitlement by default.
- No public production-grade SLA commitment.
- Third-party Module publishing is not open.

## 4. Canonical v0.1 User Journey

The following path must pass end to end in a production-like environment:

```text
User enters email and completes OTP
→ Organization + default Workspace created
→ Enter production-grade Cloud UI
→ Install CRM Lite Pack from Stable Catalog
→ Create and view Customer / Contact data
→ Agent proposes Customer custom field
→ User reviews Diff and approves Apply
→ UI shows new field, Audit records operation
→ SDK toolchain builds and submits a new runory.customer version
→ Platform validates and publishes the immutable artifact
→ Workspace generates Extension Compatibility Report
→ Workspace Admin approves upgrade
→ Data and Extension remain usable
```

If this path cannot be fully executed, `v0.1.0` must not be signed off.

## 5. Release Scope

### 5.1 SaaS Core

- Passwordless Email OTP and server-side Session.
- User/AuthIdentity, Organization, Workspace.
- Organization/Workspace Membership and fixed RBAC.
- Invitation, Owner invariant, and immediate permission revocation.
- Unified RequestContext and tenant isolation for HTTP/MCP/Agent/Job.
- Append-only Audit, Workspace API Key, and baseline security controls.
- Versioned Platform Migration.
- `early_access` Entitlement and baseline anti-abuse quotas.
- Workspace export, archive, restore, and purge foundation.
- Managed database backup and recovery runbook.

### 5.2 Platform Runtime

- Metadata-driven Object, Field, View, Form.
- Schema-driven UI.
- Module/Pack install runtime.
- Managed Workspace Extension plan, preview, apply, audit, rollback point.
- Module and Extension ownership boundary.
- Governed API shared by UI and Agent.

### 5.3 Catalog & Release Control Plane

- Official/Internal immutable artifact Registry.
- Module/Pack/Template versions.
- Structured Validation Run and Sandbox evidence.
- Internal/Beta/Stable release channel.
- Pack dependency resolver and frozen lock.
- Workspace install/upgrade compatibility report.
- Allowlisted rollout, pause, and failure isolation.
- Baseline Platform Catalog Console pages.
- Baseline Workspace Module Center pages.
- Human Release Manager approval for Stable promotion.

### 5.4 Internal SDK Toolchain

- `@runory/sdk` typed canonical Manifest authoring.
- `runory validate/test/build --json` or equivalent CI commands.
- install/upgrade/Extension fixture harness using real Runtime adapters.
- Reproducible artifact, provenance, checksum, and secret/path safety.
- `publish --channel internal` candidate adapter, with no Stable bypass.
- Runory Module Skill.

### 5.5 Reference Business Capability

- `runory.customer` Module.
- `runory.contact` Module.
- `crm-lite-pack`.
- `small-business-crm` Template.
- Customer CRUD.
- Contact CRUD and Customer relation.
- Dashboard, navigation, empty/loading/error states.
- Customer custom field Extension scenario.

## 6. Explicit Non-goals

The following items do not block `v0.1.0`:

- Team.
- Custom roles, field ACL, record ACL.
- OIDC, SAML, SCIM.
- Third-party Marketplace and publisher onboarding.
- Cloud code editor or arbitrary SQL editor.
- Service Account.
- Advanced Workflow Runtime.
- Seat Billing, usage overage Billing, Add-on.
- Public self-service paid signup.
- Private/VPC/On-premise production delivery.
- Data residency, per-tenant database, customer-managed encryption key.
- SOC 2 / ISO 27001 certification.
- Formal SLA and multi-region disaster recovery.

Stripe Billing can be enabled after `v0.1.0`; under invite-only free Early Access, Entitlement, Quota, and Usage foundations must exist, but the paid loop is not a Tag blocker.

## 7. Release Blockers

Any of the following blocks Tag or Production release:

1. Production still depends on trusted identity headers or development bootstrap login.
2. Any known cross-Organization/Workspace data read or write path exists.
3. Session, OTP, or API Key is known to be stored in plaintext.
4. Platform role can implicitly access customer business data.
5. Stable Catalog artifact can be overwritten or installed without checksum.
6. Pack installation does not use frozen dependency lock.
7. Workspace upgrade lacks compatibility preflight.
8. Migration failure continues expanding rollout or breaks other Workspaces.
9. Platform Migrations cannot replay from an empty database.
10. One real backup restore drill has not been completed.
11. Canonical v0.1 User Journey fails.
12. Any unaccepted P0/P1 data security or data-loss risk exists.
13. Customer 1.1 Catalog artifact still relies on manual packaging and has not gone through SDK validate/test/build pipeline.

## 10. Compatibility Promise

`v0.1.0` may evolve quickly before 1.0, but makes the following promises:

- No silent loss of user business data.
- Published Catalog Version is immutable.
- Platform Migration remains traceable and replayable.
- Workspace Extension compatibility is visible before upgrade.
- Breaking Module upgrade does not auto-execute.
- API and Manifest may evolve, but breaking changes must have version and migration notes.

## 12. Go / No-go Process

### Release Candidate Freeze

- Determine candidate commit.
- Freeze Platform/Core contracts and v0.1 Catalog artifacts.
- Accept blocker fixes only.
- Generate release evidence bundle.

### Go/no-go Review

Check in order:

1. Release Blockers.
2. Acceptance Matrix.
3. Required E2E Scenarios.
4. Restore drill and rollback/runbook.
5. Known issues and waivers.
6. Sign-off.

### Tag and Deploy

```text
approved release commit
→ git tag v0.1.0
→ build immutable Cloud artifact
→ deploy Production
→ run smoke + canonical journey
→ publish GitHub Release and release notes
```

If Production smoke fails after Tag: stop new user invitations and Catalog rollout, restore the last-known-good deployment according to runbook; do not delete created data or overwrite release artifacts.

## 15. Post-release Early Access Plan

- Invite controlled customers only and expand cohort gradually.
- Check auth, tenant denial, API error, migration, and Catalog rollout metrics daily.
- All Module upgrades go Internal/Beta before Stable.
- P0 security/data issue immediately pauses new invitations and rollouts.
- Collect real Workspace feedback on Module/Pack/Template and Agent usage.
- `v0.1.x` only ships compatible fixes, reliability improvements, and small enhancements; breaking platform change goes into a later minor release with migration.
