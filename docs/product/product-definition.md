# Runory Product Definition

Status: Draft v0.3
Date: 2026-06-22
Change: Cloud-first pivot — see [../04-architecture-pivot-cloud-first.md](../04-architecture-pivot-cloud-first.md)

## 1. Canonical Definition

Runory is a **Cloud-first**, Agent-native SMB business operations platform — a composable, extensible business capability platform close to WordPress for the SMB era, but with an Agent performing installation, configuration, extension, validation, and continuous operation inside a governed structure.

Runory serves small and medium businesses that want business software to start quickly in the Cloud, adapt to their operations through a Built-in Agent, and remain upgrade-safe—with an optional path to Private or Local deployment for advanced needs.

## 2. What Runory Is

Runory is:

* a **Cloud-first SMB business platform** with Portable Runtime architecture;
* a composable system of Official Modules, Business Packs, and Workspace Templates;
* a **Managed Workspace Extension** layer for user-specific customization;
* a **Built-in Agent** as the default configuration and operations layer;
* a deterministic Platform Core behind governed Agent Operation APIs;
* a schema-driven Dynamic UI shell;
* an MCP / SDK channel for developers and private deployment (not the default SMB entry).

Core product shape:

```text
Runory Cloud
+ Platform Core
+ Modules / Packs / Templates
+ Managed Workspace Extensions
+ Built-in Agent
+ Schema-driven UI
```

Early official business packs should grow as a composable portfolio, not as one monolithic suite. The current product direction is documented in [Business Pack Portfolio Strategy](./business-pack-portfolio-strategy.md).

## 3. What Runory Is Not

Runory is not:

* an AI Coding tool;
* a traditional low-code visual app builder;
* a general ERP;
* a Local-first developer tool where users modify module source;
* a temporary software project generated by an Agent;
* a vibe coding workspace where production code is freely rewritten for each user;
* a product that requires Git, MCP setup, or local installation for ordinary SMB users.

## 4. Product Paradigm

Runory is a **configuration-first, extension-first, Cloud-first** business platform.

Agents help users describe needs, recommend Packs, inspect schemas, propose plans, and apply approved changes through governed APIs. They do not directly rewrite production code, official module source, or database schema.

Default stance:

```text
Cloud Workspace first
Declarative configuration second
Managed Workspace Extension third
Generated code only as a last resort (and never for official modules)
```

Default user journey:

```text
Verify email with OTP
→ Create Organization + Workspace
→ Select Template / Install Pack
→ Import data
→ Agent configures and operates
→ Business runs
```

## 5. Difference From Existing Categories

Compared with traditional SaaS, Runory allows workspace-specific adaptation through Agent-driven Extensions without waiting for a vendor roadmap—and supports Pack/Template composition rather than a fixed product SKU.

Compared with AI Coding tools, Runory does not ask the Agent to build a custom app from scratch. It exposes trusted Modules, Packs, and governed Extension APIs.

Compared with low-code platforms, Runory is not primarily a drag-and-drop app builder. It is an Agent-native business runtime with metadata-driven objects, auditable operations, and upgrade-safe extensions.

Compared with WordPress, Runory targets business operations—not websites—with stronger governance (Diff, Audit, Rollback, field ownership, workflow runtime) and Agent as the primary operator for SMB users.

Compared with ERP, Runory starts from composable SMB workflows and Cloud operation instead of a broad, centralized enterprise suite.

## 6. Three Product Decisions

### Configuration Platform vs Generative Platform

Runory uses declarative configuration and Managed Workspace Extensions as the default mechanism. Generated code is only a last resort and must not be the normal customization path.

### Official Module Mutability

Official modules are read-only from the workspace perspective. User-specific needs are expressed through Managed Workspace Extensions—not by forking or editing module source.

### UI Freedom

Runory UI is dynamic but constrained. Views are composed from validated schemas, Module UI Slots, Template layouts, and approved components. Agents propose schema changes; Runory validates and renders.

## 7. Cloud To Local Evolution

Runory is **Cloud-first**, but architecture must preserve **Portable Runtime** so advanced customers can deploy privately or locally.

The intended evolution is:

> Cloud is the default entry point; Private / Local is the advanced deployment form.

Recommended paths:

```text
Runory Cloud (default)
→ Members / Billing upgrade
→ Export Workspace
→ Private Cloud / VPC / On-premise (advanced)
```

Local / Private deployment is **not** the MVP default. It is an Enterprise / Advanced Deployment Mode for compliance, offline use, and customers with IT teams.

Early Cloud should not depend on bidirectional sync with Local. The preferred advanced path is **controlled export/import**:

```text
Cloud Workspace
→ Export Package (config + modules + extensions + schema)
→ Private / Local Runtime Import
```

Detailed architecture notes: [../architecture/cloud-to-local-workspace.md](../architecture/cloud-to-local-workspace.md).

Team is not a current product requirement. It is reserved as a future Organization-scoped permission group when customers repeatedly assign the same people across multiple Workspaces. Team will not become a tenant, billing, or business-data ownership boundary.

SaaS Core product boundaries: [../07-saas-core-boundaries.md](../07-saas-core-boundaries.md).

## 8. Layer Summary

```text
Module     = technical install unit
Pack       = commercial delivery unit
Template   = workspace experience entry
Extension  = workspace-specific customization (managed, auditable)
```

```text
Effective App = Core + Official Modules + Pack/Template overlays + Workspace Extensions + User Data
```
