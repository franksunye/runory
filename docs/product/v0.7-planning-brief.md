## Purpose

This document records the v0.7 discovery discussion. The accepted implementation
scope is now [v0.7 Commercial Completion Execution Plan](./v0.7-commercial-completion-execution-plan.md).

v0.6 ended with a stable Foundation. v0.7 must deepen one commercial operating
model rather than introduce another:

```text
customer request / Voice
→ identify and triage
→ Quote / authorization
→ Work Order
→ plan, dispatch, and Visit
→ mobile Forms, evidence, and service report
→ completion
→ Invoice / payment
→ aftercare and warranty follow-up
```

This is the canonical Reactive Repair / Callout journey through 1.0.

## Existing v0.7 Thesis

The earlier roadmap proposed:

1. advanced scheduling and resource management;
2. inventory and materials baseline;
3. multi-site, asset, and service-history depth;
4. customer portal and communication depth;
5. data migration, roles, and implementation readiness.

These themes are retained only when they close a measured gap in the canonical
journey. They are not authorization to add a recurring-service,
project/installation, inventory-management, or portal business of their own.

Recurring Service (`V06-P02`) is explicitly beyond 1.0 because it is a second
commercial shape. The real-customer validation proposal (`V06-P06`) belongs to
v0.9 repeatability, using multiple customers with the same reactive-repair
shape. Agent discovery and public SDK generation belong to v0.8.

## Pareto Selection Rule

A v0.7 capability is admitted only when it satisfies at least one of:

- repeated evidence from real FSM customers or implementation work;
- a material sales, onboarding, or daily-operation blocker;
- a dependency required by two or more selected v0.7 workflows;
- prevention of a demonstrated destructive rearchitecture.

The following are insufficient by themselves:

- parity with a large incumbent;
- an attractive but unvalidated abstraction;
- a single provider or industry edge case;
- code that is technically possible because the Foundation now supports it.

## First Decisions to Make

Before writing the v0.7 execution plan, decide:

1. the weakest links in the canonical reactive-repair journey;
2. the top one or two closure outcomes, not five parallel themes;
3. which inherited v0.6 candidates directly support those outcomes;
4. which roadmap themes are deferred, split, or cancelled;
5. the canonical end-to-end acceptance journey and production-shaped dataset;
6. whether v0.7 ships as one release or a small sequence of independently
   valuable increments.

## Recommended Discussion Starting Point

Start with one bounded candidate:

### Canonical Journey Completion

```text
request/Voice-to-customer identification and triage
Quote/authorization-to-Work-Order continuity
dispatch, Visit, field evidence, review, and completion depth
minimum Invoice/payment allocation and customer document access
actionable exception/follow-up inbox for this journey
```

Use acceptance evidence to rank these links, then admit only the smallest set
needed to make the single journey commercially complete. A feature that serves
another operating model is deferred even when technically reusable.

## Decision

v0.7 selects **Commercial Completion**, bounded to the minimum Invoice and
payment-allocation closure for one completed Reactive Repair / Callout job.

Customer portal, Contract, inventory, recurring service, and Agent-generated
official Module/Pack/Template productization are not part of v0.7. The Agent MPT
theme remains the recommended v0.8 multiplier after the canonical product
journey is commercially complete.

## Non-goals for the Discussion

- reopening completed v0.6 Foundation architecture;
- implementing v0.8 Agent discovery or SDK generation early;
- recurring Service Plans, generated route work, pause/skip/renewal, or
  recurring billing;
- project/installation stage management as a second reference solution;
- building a universal integration, BI, offline, or route-optimization
  platform;
- treating every earlier roadmap bullet as debt that must be repaid.
