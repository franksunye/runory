## Purpose

This document starts the v0.7 discussion. It is not an implementation plan and
does not commit every inherited candidate.

v0.6 ended with a stable Foundation. v0.7 should now select the smallest set of
advanced FSM capabilities that materially improves customer adoption, daily
operation, or implementation readiness.

## Existing v0.7 Thesis

The active roadmap currently proposes:

1. advanced scheduling and resource management;
2. inventory and materials baseline;
3. multi-site, asset, and service-history depth;
4. customer portal and communication depth;
5. data migration, roles, and implementation readiness.

This is the starting hypothesis, not a mandate to build five large subsystems.

## Inherited v0.6 Candidates

| Candidate | Natural comparison in v0.7 | Main question |
| --- | --- | --- |
| Lead and customer operations (`V06-P01`) | Customer/operational workflow | Does Lead depth repeatedly block FSM customer acquisition or can current CRM/Quote intake remain sufficient? |
| Recurring service (`V06-P02`) | Multi-site, asset, and service history | Is recurring generation more valuable than broader asset hierarchy for the next real customer? |
| Contract, Invoice, and receivables (`V06-P03`) | Customer portal/commercial workflow | Which document-to-payment gap most often blocks commercial completion? |
| Operational inbox and reporting (`V06-P04`) | Cross-cutting support | Which selected workflow needs an actionable queue or metric to be usable every day? |
| Customer access (`V06-P05`) | Customer portal and communication | What is the smallest secure self-service loop with repeated customer demand? |
| Safe Module-upgrade analysis (`V06-D01`) | Implementation readiness | Is a real productized upgrade mutation entering v0.7.4? |
| Conditional architecture inventory (`V06-D02`) | Architecture support | Does an admitted capability create a boundary the current inventory cannot enforce? |

The three-customer validation proposal (`V06-P06`) belongs to v0.9
repeatability. Agent discovery and public SDK generation belong to v0.8.

## Pareto Selection Rule

A v0.7 capability is admitted only when it satisfies at least one of:

- repeated evidence from real FSM customers or implementation work;
- a material sales, onboarding, or daily-operation blocker;
- a dependency required by two or more selected v0.7 workflows;
- prevention of a demonstrated destructive rearchitecture.

The following are insufficient by themselves:

- parity with a large incumbent;
- an attractive but unvalidated abstraction;
- a single provider or industry edge case;
- code that is technically possible because the Foundation now supports it.

## First Decisions to Make

Before writing the v0.7 execution plan, decide:

1. the primary v0.7 customer/job-to-be-done;
2. the top one or two product outcomes, not five parallel themes;
3. which inherited v0.6 candidates directly support those outcomes;
4. which roadmap themes are deferred, split, or cancelled;
5. the canonical end-to-end acceptance journey and production-shaped dataset;
6. whether v0.7 ships as one release or a small sequence of independently
   valuable increments.

## Recommended Discussion Starting Point

Start with two competing bundles:

### Bundle A — Recurring Service Operations

```text
recurring Service Plan generation
asset/service-history depth
advanced technician availability and scheduling
actionable exception inbox
minimum customer appointment visibility
```

### Bundle B — Commercial Completion

```text
Lead/customer follow-up
Contract and Invoice baseline
receivables and payment allocation
customer document/payment access
actionable commercial inbox
```

Choose one as the primary v0.7 outcome using customer evidence. The other
remains a candidate; combining both by default would recreate the oversized
v0.6.1–v0.6.5 roadmap.

## Non-goals for the Discussion

- reopening completed v0.6 Foundation architecture;
- implementing v0.8 Agent discovery or SDK generation early;
- building a universal integration, BI, offline, or route-optimization
  platform;
- treating every earlier roadmap bullet as debt that must be repaid.
