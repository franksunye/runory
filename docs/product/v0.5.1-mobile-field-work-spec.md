# Runory Cloud v0.5.1 Mobile Field Work, Forms And My Work Specification

Status: Proposed for scope lock

Date: 2026-07-03

Parent plan: [v0.5 Commercial FSM Execution And Planning Plan](./v0.5-commercial-fsm-execution-plan.md)

v0.5.1 master specification: [v0.5.1 Commercial FSM Productization, Command Kernel And Optional Workflow Technical Specification](./v0.5.1-commercial-fsm-productization-technical-spec.md)

This document is the required mobile/PWA sub-specification of the v0.5.1 master specification. If a requirement here conflicts with the master specification, the master specification owns business state, command, migration, demo, and release acceptance; this document owns mobile presentation, PWA safety, browser capability, and mobile performance details.

## 1. Release Goal

v0.5.1 turns the v0.5 execution runtime into a usable daily work entry for field personnel and supervisors:

> A technician can install a lightweight Runory PWA, open assigned work quickly on a phone, complete configured forms/checklists, capture evidence and acknowledgment, and submit governed work without learning the desktop administration product.

This iteration also closes two commercial-readiness gaps found repeatedly in customer requirements:

1. customer, service-site, and asset pages need a coherent service history;
2. a Quote needs a formal printable/exportable output, not only database records.

v0.5.1 does not introduce a separate mobile business system. It exposes the same users, permissions, work items, commands, forms, records, and audit history through a focused mobile shell.

## 2. Evidence And Reuse Decisions

The implementation should reuse lessons, not copy code wholesale, from three nearby products:

| Reference | Useful evidence | Runory decision | Do not copy blindly |
|---|---|---|---|
| XLink Business 3.0 Web | mobile-first max-width shell, bottom tabs, safe-area handling, kept-alive visited tabs, conservative API caching | use a role-focused app shell and stable tab state | XLink-specific screens, session adapter, and domain vocabulary |
| StockWise frontend | install flow, manifest variants, service-worker lifecycle, update recovery, Web Vitals/performance discipline | add explicit install/update/recovery and performance gates | authenticated navigation caching and its accumulated product-specific recovery complexity |
| VisuTry Mobile | very small PWA, touch-first layout, camera/upload journey, separation of platform-independent core from browser adapters | keep the entry lightweight and isolate browser capabilities | separate deployment/repository before Runory has a genuine platform-divergence need |

## 3. Locked Product Boundary

### 3.1 One Product, Two Shells

```text
Runory Cloud
├── Desktop shell: configuration, Planning, management, broad record work
└── Mobile PWA shell: assigned work, schedule, execution, evidence, quick context

Both shells
└── same-origin session + same API + same commands + same permissions + same data
```

v0.5.1 MUST implement the mobile entry inside `apps/cloud`; it MUST NOT create a second backend, second object model, second authentication system, or duplicated workflow/form implementation.

Initial URL contract:

```text
/m                                  mobile entry / workspace selection
/m/w/{workspaceId}                  Today / My Work
/m/w/{workspaceId}/schedule         personal schedule
/m/w/{workspaceId}/work-orders      contextual work-order search/list
/m/w/{workspaceId}/work/{workItemId} assigned work execution
/m/w/{workspaceId}/visits/{visitId} visit context and actions
/m/account                           account, installation, language, diagnostics
```

The PWA manifest uses `start_url: /m` and `scope: /m/`. The same-origin design preserves the existing secure Session and tenant boundary. A separate `m.runory.*` origin is deferred until there is a measured deployment or isolation need.

### 3.2 Role-Focused Navigation

Default tabs:

```text
Today / My Work
Schedule
Work Orders
Me
```

The tabs are product jobs, not module/object enumeration. Administrators still use the desktop shell for Forms, Processes, Members & Access, Packs, and diagnostics. Mobile MAY deep-link to a customer, site, asset, Quote, or Work Order, but those objects do not automatically become bottom tabs.

### 3.3 Online Execution, PWA Resilience

v0.5.1 is installable and weak-network tolerant; it is not offline-first.

```text
Available without a successful data request:
  app icon, static shell, offline/retry screen, last safe navigation target

Requires an authenticated online request:
  login/session validation
  current My Work and schedule
  permission/assignment resolution
  final form submission
  approval, completion, reassignment, reschedule, cancel, reopen
  attachment authorization and authoritative download
```

The UI MUST never claim a governed action succeeded until the server command succeeds. Offline mutation queues, background business synchronization, conflict merging, and complete offline datasets are deferred to v0.7.

## 4. v0.5.1 P0 User Journeys

### 4.1 Install And Resume

1. User opens `/m` in iOS Safari or Android Chrome.
2. The product explains installation only when useful and shows platform-correct guidance.
3. Installed PWA opens in standalone mode and restores a safe workspace route.
4. Expired sessions return to login without showing cached customer/work data.
5. A newly deployed shell updates predictably and provides a recovery action if assets mismatch.

### 4.2 Technician Executes Assigned Work

1. Technician opens Today/My Work.
2. Queue shows subject, site, appointment time, due state, and required action.
3. Technician opens the Visit and sees customer/site contact context, asset, instructions, and service history allowed by permission.
4. Technician executes named commands such as Start Travel and Arrive.
5. Required form/checklist renders from the same Forms 2.0 definition used elsewhere.
6. Technician records values, failures, notes, photos, files, and customer acknowledgment.
7. Submit validates on the server and advances the work item exactly once.
8. A stale assignment or state returns an understandable recovery path instead of overwriting newer work.

### 4.3 Supervisor Reviews Returned Work

1. Supervisor sees review work in My Work.
2. Submission shows form version, evidence, prior revisions, actor, and timestamps.
3. Supervisor accepts or returns it with a required reason.
4. Return creates a new work item and submission revision; it does not edit prior evidence.
5. Technician receives the returned item with the reason and prior context.

### 4.4 Customer/Service History

Customer, service site, asset, Work Order, and Visit detail surfaces show a permission-filtered timeline containing:

```text
opportunity and quote milestones
accepted commercial revision
work-order creation and lifecycle events
visit schedule and assignment changes
form/checklist submissions
evidence and service-deliverable status
completion, cancellation, return, and reopening
```

Email, SMS, telephone, portal activity, and generalized omnichannel history are not P0. The timeline is a projection/query, not a new `history` business object.

### 4.5 Formal Quote Output

Users can preview, print, or export a Quote using one provider-neutral layout with workspace branding and terminology.

The output MUST contain:

```text
workspace/provider identity
quote number and revision
customer/contact and service site
line description, quantity, unit, price, discount, tax, totals, currency
validity, terms, notes
state and generated-at timestamp
source revision/snapshot identifier
Draft watermark when the Quote is not approved/sent/accepted
```

The output contract is established in v0.5.1; v0.5.2 supplies authoritative pricing, immutable revisions, and acceptance guarantees. Email delivery, remote customer portal acceptance, contract generation, and external e-signature remain later milestones.

## 5. Mobile Technical Specification

### 5.1 Repository Boundaries

Recommended structure:

```text
apps/cloud/src/app/m/                       mobile route group and manifest entry
apps/cloud/src/components/mobile/           shell and mobile-only presentation
apps/cloud/src/components/forms/            shared Forms renderer
apps/cloud/src/lib/mobile/                   install, connectivity, camera/file adapters
packages/contracts/                         shared DTOs and command contracts
packages/platform-core/                     authoritative business behavior
```

Mobile components MAY adapt presentation and navigation. They MUST NOT reimplement command rules, derive independent lifecycle state, or call business tables directly.

Browser capabilities such as installation state, camera/file input, connectivity hints, Share API, and service-worker messaging SHOULD sit behind small adapters so a later native shell can reuse domain clients without pretending React DOM components are portable.

### 5.2 PWA Manifest

Required manifest properties:

```text
id: /m
start_url: /m
scope: /m/
display: standalone
name / short_name localized
theme_color / background_color
192, 512, and maskable icons
portrait-primary preference, without blocking rotation
```

The install experience MUST work without making installation a prerequisite for use.

### 5.3 Service Worker Policy

Cache classes are deliberately narrow:

| Resource | Strategy |
|---|---|
| versioned JS/CSS/fonts/icons | cache-first, content-hash/version bounded |
| manifest and offline page | stale-while-revalidate or explicit versioned precache |
| API, auth, command responses | network-only, `no-store` |
| RSC/Flight/data requests | network-only unless a future security review approves a safe shell-only design |
| authenticated HTML | network-only |
| customer files, photos, signatures, generated Quote documents | network-only, authorization on every request |

The service worker MUST NOT cache tenant-specific or user-specific business responses. Deployment MUST set `sw.js` to `no-cache, no-store, must-revalidate`, clean old named caches, and expose a safe update/reload path.

### 5.4 Data And Refresh Behavior

- My Work and Schedule use cursor/time-window queries rather than downloading all records.
- Returning from background or regaining network triggers deduplicated revalidation.
- Mutations use the command ID, expected version, and idempotency behavior defined by the v0.5 technical specification.
- Optimistic UI MAY show “submitting”; it MUST NOT show final success before acknowledgment.
- Online form drafts SHOULD persist server-side. Device-only draft persistence is not a release gate and MUST undergo a PII/security review before use.
- Files show upload progress and explicit retry. Retrying an attachment association MUST be idempotent.

### 5.5 Camera, Evidence, And Signature

- File input supports direct camera capture where the browser permits it and ordinary gallery/file selection as fallback.
- The UI shows file type, size, progress, retry, and removal before submission.
- Evidence is associated with the submission revision and Visit/Work Item subject; it is not stored as an untyped URL inside a form answer.
- Server verifies content type, size, authorization, and integrity metadata.
- A drawn or typed acknowledgment records signer label, actor/capturer, timestamp, form version, and evidence hash. It is not marketed as regulated remote e-signature.

### 5.6 Mobile Interaction Rules

```text
minimum primary touch target: 44 x 44 CSS px
safe-area aware top and bottom chrome
dynamic viewport units; no layout hidden behind browser chrome/keyboard
one primary action per execution step
sticky submit/actions only when they do not cover form errors
camera/file controls usable without drag-and-drop
visible online/offline/submitting/failed state
screen-reader labels and keyboard path for all actions
```

Visited primary tabs MAY remain mounted to preserve scroll and draft state, but background screens MUST be inert and MUST NOT continue unnecessary polling.

### 5.7 Performance Budgets

Measured on a representative mid-range mobile device and throttled mobile network:

```text
LCP p75                          <= 2.5 s
INP p75                          <= 200 ms
CLS p75                          <= 0.1
mobile-shell initial JS          <= 220 KB gzip, excluding lazy form/map/upload features
cached static shell visible      <= 1.0 s target
My Work useful data              <= 3.0 s target including authenticated API
no map/editor/management bundle in initial field-worker route
```

The release report MUST state device, network, dataset, build, and measurement method. Budgets are gates, not marketing claims detached from a load profile.

## 6. API Additions

The mobile shell consumes the command and My Work APIs in the parent technical specification. v0.5.1 additionally locks these read/output contracts:

```http
GET /api/workspaces/{workspaceId}/my-work?assignee=me&from=...&to=...&cursor=...
GET /api/workspaces/{workspaceId}/planning/entries?resource=me&from=...&to=...
GET /api/workspaces/{workspaceId}/timeline?subjectType=company&subjectId=...&cursor=...
GET /api/workspaces/{workspaceId}/timeline?subjectType=service_site&subjectId=...&cursor=...
GET /api/workspaces/{workspaceId}/timeline?subjectType=asset&subjectId=...&cursor=...
GET /api/workspaces/{workspaceId}/quotes/{quoteId}/document?format=html|pdf&revision=...
```

Timeline entries use stable event IDs, event types, occurrence time, subject links, actor when authorized, and localized presentation data. The API MUST enforce relation-level visibility; it MUST NOT return the entire customer graph and rely on the client to hide it.

Quote document generation MUST resolve one explicit revision and record an audit event. It MUST NOT calculate different totals from the Quote command runtime.

## 7. Security And Privacy Gates

- reuse the current secure same-origin Session; never store bearer tokens in local storage;
- validate workspace membership on every mobile API and attachment request;
- exclude authenticated business data from browser/service-worker caches;
- clear ephemeral UI state on logout/workspace removal;
- prevent cross-workspace deep links from resolving by identifier alone;
- apply upload type/size limits and malware scanning policy before production pilot;
- avoid displaying customer phone/address/evidence in OS notification bodies by default;
- audit evidence download, form submission, Quote output generation, and governed commands;
- support workspace export through the existing controlled export path, not ad hoc mobile downloads.

## 8. Delivery Slices

### Slice A — PWA Shell And Safety

- `/m` routes, manifest, icons, install guidance;
- safe-area layout, bottom navigation, workspace/session entry;
- conservative service worker, offline/retry and update recovery;
- automated cache-policy and cross-tenant tests.

### Slice B — My Work And Field Forms

- Visit/work-item detail and contextual commands;
- Forms 2.0 mobile renderer, checklist, evidence, acknowledgment;

### Slice C — Context And Deliverables

- formal Quote preview/print/PDF output contract;
- role-filtered deep links and audit labels.

### Slice D — Performance And Pilot

- bundle split and query-window tuning;
- iOS/Android installed-PWA checks;
- weak-network and deployment-update recovery;
- XLink-derived field scenario and neutral HVAC/pool/solar-style form variants.

## 9. Acceptance Gates

v0.5.1 is not complete until all are true:

1. iOS Safari and Android Chrome can use the mobile entry without installation.
2. Installed PWA opens at `/m`, preserves a safe workspace route, and handles expired Session correctly.
3. A field technician can complete the canonical Visit from a phone viewport, including photo evidence and acknowledgment.
4. Required fields/evidence block completion on the server, not only in the UI.
5. Return/resubmit preserves immutable prior revisions.
6. Airplane/offline state never reports a business command as successful and provides a clear retry path.
7. Service worker caches contain no authenticated API, attachment, customer HTML, or Quote document responses.
8. Customer/site/asset history shows the same underlying events as desktop contextual views.
9. Quote print/PDF resolves a specific revision and matches server-calculated values.
10. Mobile and desktop actions produce the same command, permission, audit, and idempotency outcomes.
11. Cross-workspace deep links and cached-shell scenarios pass isolation regression tests.
12. Performance budgets in Section 5.7 are measured and met or explicitly waived before the pilot.

## 10. Explicit Deferrals

```text
full offline business-data synchronization
offline photo/form mutation queue and merge conflict UI
native iOS/Android binaries
live technician GPS tracking
push notification workflow as a release dependency
route optimization
customer portal
two-way messaging and AI phone/email agents
remote regulated e-signature
invoice/payment/accounting integration
```

The architecture leaves room for them; v0.5.1 earns the right to add them by first proving one fast, safe, mobile field-work path.
