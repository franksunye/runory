## 1. Version Thesis

`v0.2.x` proved that Runory can contain multiple official business packs and shared modules.

`v0.3` should prove that this breadth can become a coherent daily product experience.

The core shift is:

```text
from pack breadth
to runtime maturity, dynamic object experience, navigation clarity, and installable product quality
```

The screenshots and local `v0.2` trial feedback show that Runory is now hitting the right kind of pressure:

- More modules make navigation and information architecture harder.
- More objects expose the need for common relation display.
- More packs make onboarding, demo data, and installation state visible product problems.
- More route files under `apps/cloud/src/app/w/[workspaceId]` expose the need for a dynamic object shell.
- More workspace scenarios require a clear organization/workspace data boundary.

These are `v0.3` problems, not random cleanup items.

## 3. Architecture Decisions

### 3.1 Organization And Workspace Boundary

Decision:

```text
organization = account, billing, governance, members, policy
workspace = business execution space, installed packs, business records, automations, workflows
```

In `v0.3`, pack data is isolated by workspace by default.

If two workspaces both install CRM, they each get their own companies, contacts, deals, tasks, and demo data. This keeps early product behavior understandable and prevents accidental cross-team data exposure.

Not in `v0.3`:

```text
implicit cross-workspace CRM sharing
automatic shared company/contact directory
cross-workspace dedupe
global customer master data
```

Future option:

```text
organization shared data space
explicit shared directory module
workspace subscription to shared objects
cross-workspace reference permission model
```

This can be considered after `v0.4` once real usage proves the need.

### 3.2 Dynamic Object Shell Boundary

Decision:

Most object list/detail/create/edit pages should be rendered by a shared dynamic object shell.

Hand-written routes should be reserved for product experiences that combine multiple objects, workflow states, or opinionated business journeys.

Recommended boundary:

```text
dynamic route shell:
  company
  contact
  deal
  task
  work order
  service site
  asset
  quote
  ticket
  form submission

hand-written product page:
  workspace dashboard
  pack install/onboarding
  workflow builder
  automation builder
  agent proposal review
  report/workbench composition
  public landing page editor
```

The shell should be "product-grade generic", not a raw metadata table.

Required components:

```text
ObjectRouteShell
ObjectListPage
ObjectDetailPage
ObjectCreatePage
ObjectEditDrawer
ObjectFieldRenderer
ObjectRelationRenderer
ObjectActivityPanel
ObjectEmptyState
ObjectErrorState
```

### 3.3 Navigation Model

Decision:

Workspace navigation should be pack-aware and object-aware.

Recommended hierarchy:

```text
Workspace
  Dashboard
  Business
    CRM
      Companies
      Contacts
      Deals
      Tasks
    Field Service
      Work Orders
      Sites
      Assets
      Visits
    Sales
      Quotes
      Price Books
  Workbench
  Reports
  Automations
  Audit
  Settings
```

The exact labels can evolve, but the principle is stable:

- Daily business objects should be easy to find.
- Technical/platform functions should not dominate the daily workspace.
- The left nav must support collapse, scroll, grouping, and active breadcrumbs.

### 3.4 Pack Install And Demo Data Model

Decision:

Pack installation and demo data loading are separate steps.

Recommended states:

```text
available
recommended
installing
installed
demo_available
demo_loading
demo_loaded
update_available
disabled
```

This allows a workspace to install a pack without polluting real data, and later load demo data for trial or training.

### 3.5 Agent Operation Runtime Boundary

Decision:

Agent operation runtime is part of Runory Cloud, not an external afterthought.

In `v0.3`, Agent operations should be able to safely propose changes to:

```text
object fields
views
relations
workbench widgets
pack configuration
workflow definitions
automation definitions
```

Required operation lifecycle:

```text
draft
validate
preview
approve
apply
verify
rollback, when supported
history
re-run with changes
```

The UI must describe changes in business language. JSON can remain available for debugging, but it cannot be the primary approval experience.

### 3.6 Visual System And Localization Boundary

Decision:

`v0.3` should build enough design system and localization infrastructure to prevent future pack pages from drifting.

Required foundations:

```text
typography scale
spacing scale
button variants
form components
table components
card components
badge/status components
toast/notification patterns
modal/drawer patterns
responsive rules
accessibility checks
translation key convention
pack terminology localization
object field label localization
validation message localization
Agent proposal language consistency
```

`v0.3` does not need full public launch localization polish, but it must avoid hardcoding a Chinese-only product shell that would make `v0.4` expensive.

## 4. Iteration Sequence

### 4.1 v0.3.0 - Workspace IA And Navigation Shell

Goal:

Make the multi-pack workspace understandable before adding more runtime power.

Required outcomes:

```text
collapsible workspace sidebar
scrollable sidebar
pack/module grouping
second-level navigation support
module ordering policy
default icon policy
visual system token baseline
responsive shell baseline
global dashboard to workspace dashboard return path
workspace switcher clarity
organization/workspace data boundary copy
```

Implementation notes:

- Introduce or harden `WorkspaceShell`.
- Introduce a declarative `WorkspaceNavItem` model.
- Navigation should be built from installed pack/module metadata plus platform-owned workspace entries.
- Establish shared typography, spacing, surface, button, badge, and menu variants for workspace-level UI.
- Module order should be deterministic:
  1. platform workspace entries
  2. installed pack display order
  3. module display order inside pack
  4. alphabetical fallback
- If a module has no icon, use a stable default by object category rather than random visual variety.

Acceptance:

- A workspace with CRM, FSM, Sales Quote, Marketing Capture, and GEO/AEO seed modules remains navigable.
- Sidebar can collapse and expand without losing active route context.
- Sidebar scroll works on small screens and module-heavy workspaces.
- User can return from a workspace to "My Workspaces" or global dashboard.
- Product copy clearly states that each workspace owns its own pack data.
- Workspace shell passes a first responsive check for desktop, tablet-width, and narrow layouts.

### 4.2 v0.3.1 - Dynamic Object Runtime Shell

Goal:

Reduce per-object route duplication and make object pages customizable by metadata.

Required outcomes:

```text
generic object list route
generic object detail route
generic object create/edit flow
schema-driven field rendering
view-definition-driven columns
server-side pagination
server-side filtering/search
empty/loading/error states
object-level action bar
shared form/table/card primitives
pack-aware labels and localized field labels
localized validation messages
route migration plan for existing default module pages
```

Implementation notes:

- Consolidate repeated pages under a shared route or shared page shell.
- Keep explicit app routes only where Next.js routing requires stable entry points.
- Do not let a generated page feel like a database admin table.
- Default list page should support title, subtitle, primary action, search, filters, table columns, and empty state from metadata.
- Default detail page should support summary header, grouped fields, related records, activity, and edit action.
- Object list pages should not load all records eagerly once server-side pagination/filtering is available.
- Field labels, validation messages, and object terminology should use localization keys even if only English and Chinese are initially present.

Acceptance:

- At least CRM company/contact/deal/task and FSM work-order/site/asset use the shared object shell.
- Adding a new object definition does not require copying a new set of list/detail/create files.
- Existing business routes still deep-link correctly.
- Agent customization of object fields appears automatically in the dynamic shell.
- Object lists support pagination and server-side search/filtering for demo-scale and early real-scale data.
- Object forms use shared field, validation, toast, drawer/modal, and error components.

### 4.3 v0.3.2 - Relations And Cross-Object Context

Goal:

Make linked business data visible in normal user language.

Required outcomes:

```text
relation definition contract
relation display resolver
lookup field renderer
related records panel
backlink display
relationship empty states
cross-pack relation validation
relation-aware demo data
localized relation labels
```

Priority relation examples:

```text
deal -> company
deal -> primary contact
contact -> company
task -> related object
work order -> company
work order -> service site
work order -> asset
quote -> deal
quote -> work order
submission -> contact/deal
```

Acceptance:

- A deal detail page shows linked company and contact.
- A company detail page shows related contacts, deals, quotes, work orders, and tasks when relevant packs are installed.
- Missing relation targets show a clear recoverable state.
- Demo data exercises cross-pack relations instead of isolated sample records.
- Relation labels render correctly in English and Chinese terminology.

### 4.4 v0.3.3 - Workbench Reports And Widget Configuration

Goal:

Move the workspace home from static cards to a configurable business cockpit.

Required outcomes:

```text
metric widget
trend widget
breakdown / pie widget
list widget
activity widget
widget configuration panel
pack-aware widget grouping
empty/error/loading/freshness states
workbench data loading strategy
cache invalidation policy
safe default layout for many installed packs
```

Implementation notes:

- Keep the zone model simple unless real usage proves otherwise.
- Recommended default zones remain:

```text
metrics
trends
lists
activity
```

- Users can hide/reorder/configure widgets inside these zones.
- Do not build a full BI/report designer in `v0.3`.
- Dashboard data fetching should support partial widget failure and avoid blocking the whole page on one slow widget.

Acceptance:

- CRM can show pipeline value trend and deal stage breakdown.
- FSM can show work order trend and status breakdown.
- Sales Quote can show quote value trend and approval state breakdown.
- Widget failure does not break the whole dashboard.
- Workbench remains readable after installing at least four packs.
- Workbench data loading remains stable with all official demo packs installed.
- Widget configuration labels and empty states are localized.

### 4.5 v0.3.4 - Pack Onboarding And Demo Data Flow

Goal:

Make a new workspace feel guided and alive.

Required outcomes:

```text
pack catalog entry inside workspace
recommended packs
all packs list
installed packs list
pack detail page
install pack action
load demo data action
demo data loaded indicator
demo reset/reload policy
pack update indicator, if available
onboarding checklist
localized pack descriptions
install diagnostics
demo data diagnostics
```

Product behavior:

- Installing a pack creates modules, navigation, default views, and workbench contributions.
- Loading demo data is a separate explicit action.
- Demo data should be friendly enough for a new user to understand the pack in minutes.
- Official packs should have at least one guided "try this next" path after installation.
- Pack install and demo data actions should produce supportable diagnostics when they fail.

Acceptance:

- A new workspace can install CRM Lite, FSM, Sales Quote, Marketing Capture, and GEO/AEO seed capabilities from a consistent UI.
- User can choose whether to load demo data.
- Demo data status is visible per pack.
- Installed pack pages link to key objects and first recommended actions.
- Pack cards and onboarding checklists are usable in English by default and Chinese as a supported locale.
- Failed install/demo data load states show clear user-facing messages and request IDs or diagnostic IDs.

### 4.6 v0.3.5 - Agent Operations, Workflow, And Automation MVP

Goal:

Start proving Runory as an operable runtime, not only a configurable data workspace.

Required outcomes:

```text
operation catalog
Agent plan / preview / apply lifecycle
business-language diff
risk labels
preflight checks
post-apply verification
operation history
rollback support, where feasible
re-run with changes
workflow definition
workflow instance
step state
manual approval step
system action step
workflow audit trail
automation trigger
automation condition
automation action
dry run / preview
pause / disable
failure state
automation safety guardrails
```

Priority scenarios:

```text
quote approval
work order lifecycle
ticket escalation
marketing content approval
after-sales warranty review
form submission to contact/deal/task
overdue work order reminder
```

Acceptance:

- Agent can propose a safe workflow or automation change and route it through validate, preview, approve, apply, and verify.
- Agent operation diffs are readable without inspecting raw JSON.
- Operation history links to changed objects, views, workflow definitions, or automations.
- A quote approval workflow can pause for human approval and continue.
- A work order lifecycle can move through scheduled, in progress, completed, and reviewed.
- A marketing content approval or after-sales warranty review workflow can be represented by the same workflow model, even if the UI is minimal.
- A simple automation can be previewed before activation.
- Workflow and automation runs are visible from the relevant business object and audit surfaces.

- English is the default product language, and Chinese remains complete enough for local demos.
- The product is ready for English-first `v0.4` preparation.

## 6. Out Of Scope For 0.3

The following should not block `v0.3`:

```text
full cross-workspace shared master data
full BI/report designer
arbitrary custom dashboard zones
public paid billing / Stripe
complete marketing automation suite
full omnichannel support center
full enterprise role builder
complete translation QA for every public marketing page
```

They can be revisited after the product proves daily usage quality.

## 7. Exit Criteria

`v0.3` is complete when:

- A multi-pack workspace is easy to navigate.
- Most business objects use the shared dynamic object shell.
- Cross-object relations are visible and navigable.
- Workbench widgets support core business chart/list/activity patterns.
- Pack installation and demo data loading feel intentional.
- Agent operations support validate, preview, approve, apply, verify, history, and rollback where feasible.
- At least one workflow and one automation scenario are usable end to end.
- Soft delete, restore, export/import validation, unpublish, and automation guardrails have clear policies.
- Pagination, search/filtering, workbench loading, responsive behavior, and bundle health are acceptable for early external users.
- English is the default product language, and Chinese remains complete across core product surfaces.
- Audit, permissions, diagnostics, and error states are strong enough for early external users.

After this, `v0.4` can focus on English-first public launch readiness, MCP/Skill/Agent operation 1.0, website/docs, deployment hardening, and free-version onboarding.
