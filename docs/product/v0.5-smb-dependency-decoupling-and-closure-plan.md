## 1. Executive Decision

Runory should make SMB business management feel simple by default while preserving the platform depth needed for larger customers, AI agents, vertical packs, and third-party extensions.

The product rule is:

> SMB default execution must depend only on the business kernel: object model, governed commands, permissions, assignments/schedules where relevant, forms/evidence where relevant, audit/domain history, and clear role-focused UI. Platform engines such as Workflow, Automation, Catalog Rollout, MCP, Entitlements, Outbox, and Extension are enhancement layers, not hidden prerequisites for normal SMB operation.

This is not a decision to delete advanced capabilities. It is a decision to prevent advanced capabilities from becoming accidental product dependencies.

## 2. Why This Matters Before v1.0

Runory has already built many powerful primitives:

```text
metadata runtime
module and pack installer
workflow V2
automation
forms V2
permission groups
dashboard widgets
timeline/audit
outbox
api keys
entitlements
catalog release/rollout
workspace extensions
MCP/agent-facing command direction
mobile/PWA shell
```

The risk is not that these primitives are bad. The risk is that a non-technical SMB customer must understand them before completing normal work.

For v0.5/v0.5.1, the customer should experience:

```text
My Work
Customers
Quotes
Planning
Work Orders / Visits
Forms inside the work
History / timeline
```

They should not need to understand:

```text
workflow instances
form binding keys
automation rules
catalog versions
outbox messages
MCP tools
extension manifests
entitlement gates
raw object schemas
```

## 3. Dependency Layers

Runory should classify each capability into one of four layers.

| Layer | Meaning | SMB default visibility | Examples |
|---|---|---|---|
| Business kernel | Required for trusted business execution | visible through business language | commands, governed fields, records, permissions, assignment, schedule, field form execution, audit |
| Product surface | Required for usability, but not a separate engine | visible as role-focused pages | My Work, Planning, contextual actions, readable lists/detail, mobile field work |
| Optional enhancement | Useful when configured; not required for default SMB operation | hidden until enabled or configured | Workflow V2, Automation, dashboards, advanced forms admin, timeline enrichment |
| Platform/enterprise layer | Supports SaaS operations, integrations, AI, extension, or larger deployments | management/admin only | Catalog rollout, Outbox, API keys, MCP, extensions, entitlements, lifecycle export/delete |

The implementation goal is not to remove optional or platform layers. The goal is to ensure the Business kernel can run without them.

## 5. Architecture Rules

### 5.1 Commands Are The Kernel

Every governed business transition must be represented by a named command.

Examples:

```text
quote.submit_for_approval
quote.approve
quote.accept
quote.convert_to_work_order
assignment.assign
schedule.plan
visit.arrive
visit.submit_work
work_order.complete
```

No Workflow, Automation, MCP tool, form submission handler, generic CRUD endpoint, seed script, migration, or browser test may directly mutate governed lifecycle state.

### 5.2 Optional Engines May Produce Obligations, Not Business Truth

Workflow and Automation may create obligations, reminders, projections, or suggested next actions. They must not become independent owners of quote, work order, visit, assignment, schedule, or form-submission state.

The allowed relationship is:

```text
Workflow / Automation / Agent
  → calls named command
  → command validates permission, assignment, version, state
  → command writes state + event + audit + projections
```

The disallowed relationship is:

```text
Workflow / Automation / Agent
  → set_field / update_record / SQL
  → lifecycle state changes without command invariant
```

### 5.3 Product Surfaces Hide Engine Boundaries

SMB pages must use business language:

```text
My Work
Customers
Quotes
Planning
Work Execution
Service History
```

Technical surfaces belong under Management/Admin:

```text
Forms library
Workflow/process definitions
Automations
Outbox
Migration
Catalog
Extensions
API keys
Entitlements
Audit diagnostics
```

If a technician, dispatcher, or salesperson must understand one of these technical surfaces to complete a normal job, the product boundary has failed.

### 5.4 Default Pack Configuration Must Be Useful

Official packs must ship with sane default roles, views, commands, forms, and labels. Customization handles variation; it must not compensate for incomplete official pack design.

For v0.5.1, the official CRM + Sales Quote + FSM demo must work with:

```text
no custom workflow required
no custom automation required
no custom field editing required
no raw JSON form editing required by daily users
no MCP/agent operation required
no external integration required
```

Workflow-backed examples may exist, but they must be additive.

## 6. v0.5 Closure Goal

The v0.5 closure goal is:

> Convert the current v0.5 implementation from “many capable primitives exist” into “one SMB-friendly business loop works by default, while advanced engines remain optional and safely bounded.”

This closure is narrower than v0.5.1 full commercial productization. It should create the baseline from which v0.5.1 implementation can proceed without architectural drift.

## 7. v0.5 Closure Scope

### 7.1 P0 Closure Items

These must be resolved before declaring v0.5 engineering complete:

| ID | Item | Acceptance |
|---|---|---|
| CLOSURE-01 | Define SMB dependency matrix in docs | This document is linked from v0.5/v0.5.1 planning docs |
| CLOSURE-02 | Prove command-only FSM path | At least one Quote/Work Order/Visit or Work Order/Visit journey runs without active workflow instance |
| CLOSURE-03 | Remove direct lifecycle SQL from tests/demo | No v0.5/v0.5.1 test crosses governed state through SQL |
| CLOSURE-04 | Make workflow optional in pack semantics | Pack install and core demo do not require workflow definitions to exist |
| CLOSURE-05 | Keep Workflow V1 retired path explicit | V1 is not used for active execution; legacy data is read-only or migrated/conflicted |
| CLOSURE-06 | Mark technical pages as management surfaces | Forms, Workflow, Automation, Outbox, Migration, API Keys, Extensions, Catalog are not daily-role destinations |
| CLOSURE-07 | Establish default role templates | Sales, Manager, Dispatcher, Technician, Supervisor work without custom permission modeling |
| CLOSURE-08 | Ensure demo data readability | P0 pages show business labels, not raw IDs/keys |
| CLOSURE-09 | Ensure forms execute contextually | Field users complete required forms inside assigned work; Forms admin is not part of daily flow |
| CLOSURE-10 | Document optional engine rules | Workflow, Automation, Agent, Outbox, Entitlements, Extensions cannot own business truth |

### 7.2 P1 Closure Items

## 8. Implementation Slices

### Slice A — Documentation And Architecture Guardrails

Deliverables:

- link this document from v0.5/v0.5.1 planning;
- add architecture review note that optional engines cannot be business truth owners;
- add or update tests that detect direct governed lifecycle writes outside commands;
- add pack-level test proving workflow definitions are optional for core execution.

Exit:

```text
docs linked
guard tests fail on direct lifecycle writes
one command-only journey fixture exists
```

### Slice B — Command-Only Core Path

Deliverables:

- identify the shortest meaningful FSM path that must run without Workflow V2;
- ensure every edge has a command;
- ensure command events/audit/projections are written;
- ensure My Work/Planning can surface command-generated obligations where needed.

Suggested path:

```text
create Work Order
→ triage/plan
→ assign technician
→ schedule Visit
→ technician starts/arrives
→ submit field form/evidence
→ complete Visit
→ complete Work Order
```

Exit:

```text
no workflow instance required
no direct SQL status mutation
audit/domain history visible
browser or API acceptance fixture passes
```

### Slice C — Optional Workflow Overlay

Deliverables:

- keep workflow-backed Quote approval or service-review example;
- ensure workflow-generated Work Items complete through commands;
- retire V1 active path;
- remove product wording that implies workflow is mandatory.

Exit:

```text
workflow-backed scenario passes
command-only scenario passes
V1 active execution path unavailable
```

### Slice D — SMB Product Surface Simplification

Deliverables:

- daily navigation exposes role/product pages, not technical engines;
- technical pages move under Management/Admin;
- contextual action bars hide raw command keys and expose business actions;
- forms execute in work context;
- raw IDs/field keys are absent from P0 business surfaces.

Exit:

```text
sales/dispatcher/technician/supervisor can complete jobs without visiting Workflow, Automation, Catalog, Outbox, API Keys, or Extensions pages
```

### Slice E — Demo And Evidence

Deliverables:

- seed CRM + Sales Quote + FSM together;
- seed both command-only and workflow-backed scenarios;
- include readable personas, assignments, schedules, forms, and timeline events;
- add validator for dependency assumptions.

Exit:

```text
demo validator proves:
  command-only path exists
  workflow-backed path exists
  raw IDs are not primary labels
  assignments/schedules/forms are linked
  technical engines are not required for daily journey
```

## 9. Release Acceptance Checklist

Operational acceptance is tracked in
[`v0.5.1-local-commercial-acceptance-checklist.md`](./v0.5.1-local-commercial-acceptance-checklist.md).

v0.5 closure is accepted only when Product and Engineering can answer “yes” to all of these:

1. Can an SMB customer run the core FSM path without configuring Workflow?
2. Can the same customer run without configuring Automation?
3. Can field users submit required forms without opening the Forms management library?
4. Can sales/dispatch/technician/supervisor roles work from role-focused pages?
5. Are all governed state changes made through named commands?
6. Are Workflow V1 execution paths retired or blocked?
7. Are advanced engines hidden under Management/Admin unless enabled?
8. Does demo data tell a coherent business story with readable labels?
9. Can optional Workflow V2 still demonstrate approval/return/resubmit without owning business truth?
10. Do tests prevent future regressions into hidden dependencies?

If any answer is “no”, v0.5 should be considered technically advanced but not product-closed.

## 10. Product Principle For Future Versions

Every new capability should declare its dependency layer before implementation:

```text
Business kernel?
Product surface?
Optional enhancement?
Platform/enterprise layer?
```

If a feature belongs to an optional or platform layer, it must not become a hidden prerequisite for SMB default execution.

This is the constraint that keeps Runory simple enough for SMB customers and powerful enough for the AI/agent/extension era.
