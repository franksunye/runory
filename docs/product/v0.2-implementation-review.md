# Runory Cloud 0.2 Implementation Review

Status: Review
Date: 2026-06-23
Scope: committed `v0.2` productization implementation, especially the business workspace / workbench experience.

## Review Summary

The implementation is directionally correct and meaningfully improves the product shape of Runory Cloud:

- The workspace default experience now reads as a business workbench instead of a technical console.
- CRM Lite can be installed from the workspace and can be used to create an immediately understandable customer/contact/task experience.
- The global workspace entry, workspace shell, activity surface, and platform/admin split are moving in the right direction.
- Local verification passed:
  - `pnpm --filter @runory/cloud typecheck`
  - `pnpm --filter @runory/cloud build`
  - `pnpm --filter @runory/platform-core test -- --runInBand`
  - `pnpm --filter @runory/sdk-testing test`

That said, there are a few product-facing and architecture-facing issues that should be fixed before treating `v0.2` as a stable productization baseline.

## Findings

### P1: Global workspace cards now show incorrect record counts

The global dashboard still expects the old stats API shape:

- `apps/cloud/src/app/dashboard/page.tsx:78`
- `apps/cloud/src/app/dashboard/page.tsx:80`
- `apps/cloud/src/app/dashboard/page.tsx:81`
- `apps/cloud/src/app/dashboard/page.tsx:86`

It reads `json.data.records.total`, but the new workspace stats API returns `metrics` instead:

- `apps/cloud/src/app/api/workspaces/[id]/stats/route.ts:156`
- `apps/cloud/src/app/api/workspaces/[id]/stats/route.ts:159`
- `apps/cloud/src/app/api/workspaces/[id]/stats/route.ts:170`

Impact:

The `/dashboard` "My Workspaces" page can display `0 条记录` even when the workspace has customers, contacts, and tasks. This weakens the new navigation model because the global entry point is supposed to help users understand which workspace has real activity.

Recommendation:

Keep `/api/workspaces/[id]/stats` backward compatible for now by adding a simple `records: { total }` field derived from `metrics.customers.total + metrics.contacts.total + metrics.tasks.total`, and update `/dashboard` to prefer the new `metrics` shape when present. This is safer than only changing the frontend because other callers may already rely on the previous response contract.

### P1: Due-today and overdue task metrics are likely misclassified

The task date fields are stored as date-only strings, for example:

- `apps/cloud/src/lib/demo-data.ts:178`
- `apps/cloud/src/lib/demo-data.ts:187`
- `catalog/packs/crm-lite-pack/demo-data.json:56`
- `catalog/packs/crm-lite-pack/demo-data.json:70`

But the stats query compares them with ISO timestamps:

- `apps/cloud/src/app/api/workspaces/[id]/stats/route.ts:26`
- `apps/cloud/src/app/api/workspaces/[id]/stats/route.ts:27`
- `apps/cloud/src/app/api/workspaces/[id]/stats/route.ts:93`
- `apps/cloud/src/app/api/workspaces/[id]/stats/route.ts:94`

Impact:

SQLite string comparison can treat a date-only value like `2026-06-23` as earlier than `2026-06-23T00:00:00.000Z`. A task due today may be missed by `dueToday` and counted as overdue. This affects the highest-value business signal on the workbench: "what needs attention today?"

Recommendation:

Normalize comparisons to date granularity. Either compare against `todayDate = YYYY-MM-DD`, or use `DATE(due_date)` in SQL consistently:

```sql
DATE(due_date) = DATE(?)
DATE(due_date) < DATE(?)
```

Also add a focused test covering a task due today, yesterday, and tomorrow.

### P1: Demo data ownership is split between Pack/Core and Cloud UI

The platform core already supports pack-owned demo data:

- `packages/platform-core/src/installer.ts:46`
- `packages/platform-core/src/installer.ts:96`
- `packages/platform-core/src/installer.ts:297`
- `catalog/packs/crm-lite-pack/demo-data.json:1`

The modules page installs CRM Lite with pack demo data:

- `apps/cloud/src/app/w/[workspaceId]/modules/page.tsx:130`

But the workbench installs the pack without demo data and then calls a Cloud-local CRM seeding endpoint:

- `apps/cloud/src/app/w/[workspaceId]/dashboard/page.tsx:196`
- `apps/cloud/src/app/w/[workspaceId]/dashboard/page.tsx:199`
- `apps/cloud/src/app/w/[workspaceId]/dashboard/page.tsx:212`
- `apps/cloud/src/app/api/workspaces/[id]/seed-demo/route.ts:5`
- `apps/cloud/src/lib/demo-data.ts:1`

Impact:

There are now two sources of truth for CRM demo records. The module center and workbench can seed different sample datasets. More importantly, this blurs the architecture principle that business assets belong to packs/modules while SaaS Core provides common execution, installation, tenancy, audit, and permission capabilities.

Recommendation:

Consolidate demo seeding under the pack/catalog layer. The workbench can still offer a friendly "Load demo data" action, but it should call a platform-core-backed pack demo seeding capability instead of owning CRM-specific records in `apps/cloud`.

Suggested shape:

```text
POST /api/workspaces/[id]/packs/[packId]/demo-data
→ validates workspace permission
→ delegates to platform-core seedPackDemoData
→ writes audit event
→ returns created/skipped counts
```

### P2: Workbench readiness is currently inferred from any installed pack

The stats route treats any installation as enough to enable the CRM workbench:

- `apps/cloud/src/app/api/workspaces/[id]/stats/route.ts:52`
- `apps/cloud/src/app/api/workspaces/[id]/stats/route.ts:53`
- `apps/cloud/src/app/api/workspaces/[id]/stats/route.ts:54`

Impact:

This is acceptable while CRM Lite is the only meaningful business pack, but it will become wrong once multiple packs exist. A workspace could have a non-CRM pack installed and still see CRM-specific workbench empty states, metric labels, and demo-data prompts.

Recommendation:

For `v0.2`, document that the workbench is CRM Lite oriented. For `v0.2.1`, derive readiness from installed workbench contributions or required business objects, for example `customer`, `contact`, and `task`.

### P2: Stats helpers hide schema and query failures too aggressively

The stats route catches all query failures and turns them into zero or empty arrays:

- `apps/cloud/src/app/api/workspaces/[id]/stats/route.ts:29`
- `apps/cloud/src/app/api/workspaces/[id]/stats/route.ts:34`
- `apps/cloud/src/app/api/workspaces/[id]/stats/route.ts:39`
- `apps/cloud/src/app/api/workspaces/[id]/stats/route.ts:43`

Impact:

This makes the no-pack path robust, but it also hides real schema drift, migration failures, or broken SQL. In a production-grade product, silent zeroes can look like healthy empty data.

Recommendation:

Keep the safe behavior only for expected "object table not installed" cases. For other errors, log a structured warning with request ID, workspace ID, table/object, and query context. If possible, distinguish "table missing" from "query failed".

## Suggested Fix Order

1. Restore stats API compatibility and fix `/dashboard` record counts.
2. Fix date-only task comparisons and add a focused stats test.
3. Remove the Cloud-local CRM demo-data source or route it through a shared pack demo seeding service.
4. Tighten workbench readiness so it is explicitly CRM Lite for `v0.2`, then pack-composable for `v0.2.1`.
5. Add structured logging around stats query failures.

## Product Readiness Opinion

The implementation is good enough to continue toward `v0.2`, but I would not call this a polished `v0.2` baseline until the two P1 user-facing issues are fixed:

- global workspace record counts must be correct;
- due-today / overdue task signals must be correct.

The demo-data ownership issue is also important because it touches the long-term architecture boundary between SaaS Core and Runory business packs. It does not block local experience testing, but it should be corrected before the pack system grows.
