## 1. Version Thesis

`v0.2.x` expands and stabilizes the business pack foundation.

`v0.3` should shift focus from "more pack ideas" to **runtime maturity and product experience**.

The version should prove:

> Runory is not only composable; it is comfortable, reliable, governable, and useful in repeated daily work.

`v0.3` is where Runory should feel less like a promising platform and more like a product people can return to every day.

## 2. Product Boundary

`v0.3` should focus on:

```text
workflow runtime
automation
Agent operation quality
business UX/UI polish
dynamic object UI quality
permissions and roles
observability and audit readability
performance and reliability
mobile/responsive behavior
```

`v0.3` should avoid:

```text
adding many new packs
deep ERP/accounting scope
full CMS scope
full omnichannel support center
full marketing automation suite
```

New pack work can continue only when it directly validates runtime/automation/UX requirements.

## 3. Runtime Goals

### 3.1 Workflow Runtime

Goal:

Introduce a durable workflow model that can support business processes across packs.

Required capabilities:

```text
workflow definition
workflow instance
step / transition
status
assignee
due date
approval step
manual task step
system action step
audit trail
retry / failure state
```

Initial workflow examples:

```text
quote approval
work order lifecycle
ticket escalation
marketing content approval
after-sales warranty review
```

Acceptance:

- A workflow can pause for human approval and resume.
- A workflow can create a task or update a business object.
- Workflow state is visible in business language.
- Failures are recoverable and auditable.

### 3.2 Automation Runtime

Goal:

Allow users and Agents to configure simple, safe automations without writing code.

Recommended first automation model:

```text
When event happens
If conditions match
Then perform approved actions
```

Examples:

```text
When quote is approved, notify owner.
When form submission arrives, create contact and task.
When work order is overdue, create escalation task.
When ticket is resolved, request satisfaction follow-up.
When warranty expires soon, create renewal reminder.
```

Required capabilities:

```text
event trigger
condition builder
action catalog
dry run / preview
permission validation
audit log
disable / pause
failure notification
```

Acceptance:

- A workspace admin can create a simple automation through UI or Agent proposal.
- Automation can be previewed before activation.
- Automation runs are visible and debuggable.

### 3.3 Agent Operation Runtime

Goal:

Make Agent operations feel trustworthy and repeatable.

Required capabilities:

```text
operation catalog
plan / preview / apply / rollback consistency
risk labels
business-language diff
preflight checks
post-apply verification
operation history
re-run with changes
```

Acceptance:

- Agent can propose pack-aware changes.
- User can understand what will change without reading JSON.
- Failed operations produce actionable explanations.
- Successful operations deep-link to visible product changes.

## 4. UX/UI Goals

### 4.1 Business Object UI

Goal:

Dynamic object pages should feel like intentional product screens, not generic metadata tables.

Required improvements:

```text
standard list layout
standard detail layout
standard create/edit layout
related records panel
empty states
loading states
error states
field grouping
inline validation
saved views / filters
search and sort
bulk actions, later
```

Acceptance:

- Company, contact, deal, work order, quote, ticket, and submission pages share consistent interaction patterns.
- Relations are visible and navigable.
- Users can recover from validation errors without losing context.

### 4.2 Workbench UX

Goal:

Composable workbench should become a polished daily home.

Required improvements:

```text
widget skeletons
widget error boundaries
empty states per widget
personalization mode
pack-aware widget grouping
widget configuration UI
refresh and freshness indicators
mobile layout
```

Acceptance:

- Users can hide/reorder/configure widgets.
- Widget failures do not break the whole workbench.
- Multi-pack workbench remains understandable.

### 4.3 Navigation And IA

Goal:

Users should always know:

```text
global vs workspace vs manage vs platform
which pack they are using
which object they are editing
how to return to previous work
```

Required improvements:

```text
breadcrumbs
return-to-workspace
context switcher
pack grouping under Manage
recent items
command/search entry, later
```

Acceptance:

- Users can move across packs without feeling lost.
- Management surfaces are discoverable but not in the daily path.

### 4.4 Visual System

Goal:

Consolidate UI into a production-grade design system.

Required improvements:

```text
typography scale
spacing scale
button variants
form components
table components
card components
badge/status components
toast/notification patterns
modal/drawer patterns
responsive rules
accessibility checks
```

Acceptance:

- New pack pages can be built from consistent primitives.
- UI quality does not depend on per-page handcrafted styling.

## 5. Governance And Trust

### 5.1 Audit Readability

Goal:

Audit should be readable as product evidence.

Required improvements:

```text
business event labels
actor display names
before/after summary
entity links
filter by object/action/actor
export
```

### 5.2 Permissions

Goal:

Move beyond owner/admin/member basics.

Required capabilities:

```text
pack-aware permission groups
object-level permissions
action-level permissions
admin-only configuration
viewer/editor/admin roles per workspace
future team/group compatibility
```

### 5.3 Data Safety

Required improvements:

```text
soft delete policy
restore path for key objects
export consistency
import validation
public content unpublish
automation safety guardrails
```

## 6. Observability And Reliability

Required capabilities:

```text
structured logs
request IDs visible in errors
operation traces
workflow run traces
automation run traces
pack install diagnostics
demo data diagnostics
health checks
slow query visibility
```

Acceptance:

- A developer or operator can diagnose pack install, workflow, automation, and Agent operation failures quickly.
- Users see clear, non-technical error messages with supportable request IDs.

## 7. Performance Goals

Required improvements:

```text
workbench data loading strategy
pagination for object lists
server-side filtering/search
cache invalidation rules
bundle review
Next.js dev/prod stability
public page performance
```

Acceptance:

- Common workspace pages load predictably with demo-scale and early real-scale data.
- Multi-pack dashboards do not degrade dramatically as modules increase.

## 8. Internationalization And Content Quality

Required improvements:

```text
English default
Chinese complete
pack terminology localization
object field label localization
validation message localization
public landing page localization
Agent proposal language consistency
```

Acceptance:

- A workspace can use English as the default product language.
- Chinese UI is complete enough for local demos and early users.

- Post-apply verification.

- Standard object page layouts.
- Related record panels.
- Saved filters/search/sort.
- Shared UI primitives.

### 9.4 v0.3.3 — Workbench And Navigation Polish

- Personalization mode.
- Widget configuration UI.
- Breadcrumbs/recent items/context switcher.

### 9.5 v0.3.4 — Governance, Observability, Reliability

- Audit readability.
- Pack-aware permissions.
- Workflow/automation traces.
- Structured logs and diagnostics.

## 10. Exit Criteria For 0.3

Runory is ready to move beyond `v0.3` when:

- Workflows can run real quote approval, ticket escalation, and work order lifecycle scenarios.
- Automations can be configured safely by workspace admins or Agents.
- Agent operations are understandable, reversible where possible, and auditable.
- Business object screens feel consistent across packs.
- Workbench personalization works with multiple packs.
- Audit, permissions, diagnostics, and errors are production-grade enough for early customers.

The intended next step is a public free launch focused on English-first internationalization, website/docs readiness, and MCP / Skill / Agent operations 1.0. See [Runory Cloud 0.4 Public Free Launch Plan](./v0.4-public-free-launch-plan.md).
