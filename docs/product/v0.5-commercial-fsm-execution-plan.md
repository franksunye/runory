## 1. Executive Decision

The goal of `v0.5` should be:

> Make Runory capable of running and planning one real, auditable, multi-role field-service business loop from opportunity and quote through dispatch, scheduling, onsite execution, review, rework, and completion.

The version is not about adding more pack names. It is about using the existing `CRM Lite`, `FSM`, and `Sales Quote` packs to productize reusable execution, form, assignment, scheduling, spatial-view, and commercial-document capabilities until they can carry production work.

The recommended version name is:

```text
v0.5 — Commercial FSM Execution And Planning Core
```

The release promise is:

```text
A service business can use Runory to:

manage the customer and opportunity
→ prepare and revise a quote
→ approve or reject the quote
→ convert an accepted quote into service work
→ dispatch a technician and schedule visits
→ plan team work through calendar, resource timeline, and map views
→ complete required step forms in the field
→ submit service evidence for review
→ reject, return, resubmit, cancel, or reopen work under policy
→ complete the work with a trustworthy audit trail
```

This is a stronger and more useful goal than “CRM + FSM + Quote objects are installable.” The three packs already exist structurally. `v0.5` must prove that they operate as one system and leave behind reusable business runtimes rather than FSM-only hard coding.

## 2. Why This Is The Right v0.5 Challenge

`v0.4` makes Runory publicly discoverable, self-service, stable, and supportable. It intentionally does not deepen the business runtime.

The next product risk is no longer whether Runory can define and render objects. The risk is whether Runory can safely execute business change over time.

Current Runory strengths:

```text
metadata-driven objects and views
composable modules and packs
cross-pack relations
workspace extensions
basic workflow definitions and instances
basic conditions, approvals, and system actions
audit, automation, permissions metadata, and dynamic UI foundations
```

Current limitation:

```text
The system models business records well,
but still treats business execution mostly as CRUD plus a lightweight state transition.
```

A commercial FSM deployment is an effective forcing function because it combines:

```text
long-lived records
multiple business roles
scheduled and field work
required evidence
human handoffs
commercial approval
cross-object creation
exceptions and reverse operations
time limits and escalation
concurrent updates
audit and recovery requirements
```

## 3. Evidence Framework: XLink Case And Industry Benchmarks

Runory should use two different evidence sources:

```text
XLink
  = nearby real waterproof-repair case
  = proves concrete exceptions, forms, roles, and operating pain

Mature commercial FSM reference systems
  = cross-industry product evidence
  = reveals stable domain boundaries, execution patterns, and product surfaces
```

XLink should be used as a requirements mine and acceptance case, not as an architecture or vocabulary to copy. A behavior becomes a Runory core requirement only when it is supported by at least one of these:

```text
1. repeated across mature FSM products;
2. required to complete the canonical commercial FSM journey safely; or
3. proven by the XLink pilot to be unavoidable in the target service business.
```

### 3.1 XLink Workflow Evidence

XLink workflow definitions and runtime show these production concepts:

```text
workflow-level form binding
node-level form binding
read-only and fillable fields per node
node assignee rules
runtime todo nodes with actual handler
conditional routing
serial and parallel branches
automatic nodes
subflows
pre-action, action, rollback, and return-to behavior
node deadlines and multiple reminder deadlines
rollback to a prior node
take-back by the previous node
workflow cancellation
additional approvers / handlers
definition version snapshots on running instances
```

This proves that the target waterproof-repair operation needs more than “set status to next value”: a person receives work, supplies required business data, and submits it to progress. It does not prove that XLink's node model or scripting design is the correct general abstraction.

### 3.2 XLink Domain Evidence

XLink's core domain demonstrates several concrete needs in the waterproof-repair case:

```text
CRM:
  lead/contact/company ownership, lifecycle, assignment, follow-up, conversion

Commercial:
  product/service catalog, quote/order snapshot, discount, amount calculation,
  approval/withdrawal/review semantics

FSM:
  numbered work, customer/site details, service type, priority, business stage,
  resource/territory assignment, schedule, workflow tasks, SLA, evidence, completion
```

The lesson is not that Runory v0.5 needs every XLink entity. XLink suggests the following separation as a working hypothesis; the cross-vendor review below determines whether it becomes a Runory invariant:

```text
business record truth   — customer, quote, work order, visit, report
process truth           — current step, pending work, actor, deadline, transition history
commercial snapshot     — what was offered and accepted at that version and price
```

### 3.4 Cross-Vendor Execution Patterns

| Stable pattern | Salesforce | Dynamics 365 | ServiceNow | Oracle | Runory v0.5 decision |
|---|---|---|---|---|---|
| Requested work differs from schedulable execution | Work Order + Service Appointment | Work Order + Resource Booking | Work Order + Work Order Task | Activity is the schedulable unit | Keep Work Order as aggregate and Visit/Work Item as schedulable execution |
| Dispatch is a product workspace | Appointment list, Gantt, map | Requirement backlog, resource timeline/Gantt, map | Dispatch queue, task map, Dispatcher Workspace | Resource calendar and daily route | One Planning surface with backlog, calendar, resource timeline, and map |
| Field input is template-driven | Field Service records/reports are tied to work context | Published Inspection template bound to service task/work order/asset | Questionnaire can be required before task close | Activity types configure behavior and fields | One Forms product with versioned definitions and typed usage |
| Work lifecycle and booking lifecycle are related but distinct | Appointment status is separate from source work | Work Order status and Booking status synchronize under rules | Work Order and Work Order Task have related state flows | Activity has scheduled/ordered/route state | Keep authoritative lifecycle per aggregate; synchronize through commands, never duplicate editable mirrors |
| Resource availability is more than an assignee field | Service resource, operating hours, absences | Bookable resource, work hours, skills, territory | Agent/crew, schedule, skills, location | Resource calendar, work zones, route | Assignment history + availability/calendar contract; advanced matching deferred |
| Exceptions are first-class | schedule/reschedule/unschedule and multi-appointment work | rebook, cancel, follow-up booking without completing work order | reject, reassign, request more information, return to Draft | ordered, unordered, and unscheduled activities | Reject, return, reassign, reschedule, cancel, reopen, and retry are named commands |
| Commercial offers preserve versions | Opportunity can have multiple quotes | activated quote is read-only; revision creates a new editable version | — | — | Versioned quote, locked accepted snapshot, idempotent conversion to work |
| Role-specific surfaces matter | dispatcher console and mobile worker experience | schedule board for dispatcher, mobile work for technician | dispatcher workspace and agent work | field manager and mobile resource route | My Work for fulfiller, Planning for dispatcher, contextual configuration for admin |

### 3.5 Industry Invariants Adopted By v0.5

The cross-vendor patterns above strengthen the current plan in six places:

```text
1. Work Order is the business aggregate; Visit/Work Item is the schedulable execution unit.
2. Planning is one role-specific workspace, not a set of object pages.
3. Forms/inspections/checklists are versioned definitions bound into work, not fields hard-coded per FSM scenario.
4. Work, visit/booking, assignment, and workflow-step states each have an owner; related states synchronize through commands.
5. Availability, working time, duration, and location are scheduling inputs even before optimization exists.
6. Reverse and exception operations are normal execution paths, not administrative database repair.
```

Quote evidence adds a seventh invariant:

```text
7. Once a commercial offer is activated/accepted, its version and prices are immutable; changes create a revision.
```

### 3.6 What v0.5 Deliberately Does Not Copy

```text
XLink arbitrary Groovy scripts and implementation-specific object names
Salesforce enterprise territory, skills, optimization, and inventory breadth
Dynamics universal resource scheduling breadth and full offline mobile scope
ServiceNow enterprise task/ITSM hierarchy and plugin matrix
Oracle predictive routing, learned duration, and route optimization
full CPQ, order, invoice, tax, and contract suites
```

The rule is:

> Adopt the operational invariant, then design the smallest Runory-native product expression that passes the real FSM journey.

Every P0 capability should carry a short evidence card before implementation:

```text
target user job
XLink case evidence, if present
cross-vendor evidence from official documentation
Runory product decision
explicitly deferred enterprise complexity
acceptance scenario and success signal
```

This prevents both forms of overfitting: copying XLink because it is nearby, and copying an enterprise suite because it is mature.

### 4.2 The Most Serious Technical-Product Gap

Today, `transitionWorkflow` changes workflow state before synchronizing the business record and before running the system action. Failures in those later operations are logged but do not roll back the transition.

That is acceptable for a preview runtime. It is not an acceptable default for operations such as:

```text
approve quote and lock its version
accept quote and create work order
submit service report and move work to review
approve service report and complete work order
cancel work and release its scheduled visit
```

`v0.5` must introduce a first-class business command boundary. A user should execute an intention such as `quote.submit_for_approval` or `visit.complete`, not independently edit several records and then click a state button.

## 5. Product Boundary

### 5.1 Packs In Scope

The commercial FSM solution should be composed from:

```text
CRM Lite Pack
Sales Quote Pack
FSM Pack
```

All three are required, but the product should be presented as one canonical solution/template for v0.5:

```text
Commercial Field Service Template
```

This template is not a fourth monolithic pack. It is a tested composition and default journey over the three packs.

Shared capabilities such as Forms, guided work, planning, permissions, and business commands are dependencies of the solution template. They should install and upgrade through module/runtime boundaries without appearing as additional business Packs that an operator must understand.

### 5.2 Capabilities In Scope

```text
customer/account and contact context
opportunity/deal lifecycle
product/service catalog and real price-book entries
quote header + line editing
quote totals, versions, approval, rejection, withdrawal, and revision
accepted quote to work-order conversion
work-order triage, assignment, scheduling, blocking, cancellation, and reopening
one-to-many service visits
technician/member linkage and work inbox
required field-service step forms
reusable checklist templates as Forms blocks
configurable evidence policies and business deliverables
service evidence and configurable report review
generic assignment history and dispatch commands
datetime/timezone-aware schedule slots and conflict detection
calendar, resource timeline, and basic map views over one schedule source
basic SLA deadline and overdue escalation
business-command authorization
complete transition/submission/audit history
```

### 5.3 Explicit Non-goals

```text
payment collection
invoicing and accounting
tax compliance engine
call center, telephony, chat, or customer-service integrations
email/SMS/WeChat delivery integrations
route optimization, turn-by-turn navigation, live tracking, or automatic geocoding
inventory reservation and procurement
contract/e-signature vendor integration
full offline mobile application
full CPQ rules engine
arbitrary customer scripts
general BPMN designer
full XLink feature parity
```

Attachments and a captured customer acknowledgment are in scope as execution evidence. A basic provider-neutral map view using stored coordinates is in scope; map routing and location-provider integrations are not. Third-party e-signature and document-signing integrations are not.

### 5.4 User-Facing Product Model

The commercial FSM template should feel like one product, not three installed Packs plus a collection of technical runtimes.

Recommended role-aware product surfaces:

```text
Workbench / My Work   — what I need to do now
Customers             — companies, contacts, and their business timeline
Opportunities         — commercial work before an accepted offer
Quotes                — pricing, revision, approval, and acceptance
Planning              — backlog, calendar, resource timeline, and map
Work Orders           — service delivery and completion
Forms                 — one library for designs, checklist blocks, usage, and responses
Management            — roles, process configuration, audit, and diagnostics
```

Product rules:

```text
users do not switch between CRM/FSM/Quote Packs to complete the canonical journey
calendar, resource timeline, and map are views inside Planning, not separate products
workflow is presented through current stage, next action, My Work, and process history
checklist, evidence, and report are parts of Forms and Work Execution, not separate navigation products
commercial-document runtime is presented as Quotes in v0.5
technicians complete assigned work; they do not need access to the Forms configuration library
technical module/object names appear only in management/developer surfaces
```

The Packs and internal modules remain important installation, ownership, and upgrade boundaries. They must not dictate the operator-facing information architecture.

## 6. Enabling Capabilities Behind The Product

The architectural outcome of the version should be six reusable capabilities validated by one domain composition:

```text
1. Workflow Execution Runtime
2. Transactional Business Command Runtime
3. Forms Definition / Binding / Submission Runtime
4. Roles, Permission Enforcement, And Work Queue Runtime
5. Work Assignment, Scheduling, And Spatial Planning Runtime
6. Commercial Document And Pricing Runtime
7. CRM + Quote + FSM Domain Composition
```

The first six are reusable foundations. The seventh proves that they compose into a real product. FSM terminology and default templates belong to the Pack/template layer, not to the reusable runtimes.

### 6.1 Pillar A — Workflow Execution 2.0

Upgrade workflow from a record state helper into a durable human-work runtime.

Required definition concepts:

```text
workflow definition version
step definition
step type: human / approval / system
step form or view binding
field policy: visible / editable / required / hidden
assignee rule: user / permission group / record field / previous actor
deadline policy
transition command
transition guard
reverse-operation policy
```

Required runtime concepts:

```text
workflow instance pinned to a definition version
step instance / work item
assignee and actual handler
status: pending / active / completed / returned / cancelled / failed
due time and overdue state
form submission snapshot
transition event
revision / optimistic-lock version
idempotency key
error and retry state for durable actions
```

Required operations:

```text
start
claim / assign / reassign
submit step
approve / reject
return to a permitted prior step
withdraw / take back when the next step is still eligible
cancel workflow
reopen completed/cancelled work under explicit policy
retry failed system action
```

The release does not require a general visual BPM designer. Official workflow definitions may remain declarative and pack-owned, with a constrained editor for safe workspace adjustments.

### 6.2 Pillar B — Transactional Business Commands

Introduce a governed command layer above generic CRUD.

Examples:

```text
deal.create_quote
quote.add_line
quote.recalculate
quote.submit_for_approval
quote.reject
quote.create_revision
quote.accept
quote.convert_to_work_order
work.assign
work.accept_assignment
work.reject_assignment
work.reassign
work.schedule
work.reschedule
workflow_step.submit
work_order.triage
work_order.schedule_visit
visit.start
visit.submit_report
report.reject_for_rework
report.approve
work_order.complete
work_order.cancel
work_order.reopen
```

Domain commands may compose shared commands. For example, `work_order.schedule_visit` creates the FSM visit and invokes the generic assignment/schedule contracts; `visit.submit_report` invokes generic step submission with the configured checklist, evidence policy, and deliverable binding.

Every command must define:

```text
input schema
authorization rule
preconditions
records read and written
transaction boundary
idempotency behavior
audit event
resulting workflow transition
recoverable failure behavior
```

This action catalog should also become the safe surface used by the UI, Agent, MCP, and automation runtime. One business invariant should not be implemented four different ways.

### 6.3 Pillar C — One Forms Product With Multiple Bindings

Runory should expose one Forms product and one form designer. Workflow, FSM, Marketing, record actions, and future customer portals are consumers of the same versioned form definition; they are not separate form products.

User-facing model:

```text
Forms
├── Designs and versions
├── Used in
│   ├── Workflow step
│   ├── Published link/page
│   ├── Record action
│   └── Future: customer portal
└── Responses and history
```

The existing `runory.form` 1.x is not yet this shared product. Its `slug`, public publication state, campaign relation, target CRM object, submit-button label, and success message make it a Marketing Capture public form. In v0.5, `runory.form` should become the shared Forms module and those public-channel concerns should move into a typed public-endpoint binding. This is a module/model migration, not a second product called Public Forms.

Recommended internal model:

```text
form_definition
form_version
form_binding
  ├── workflow_step_binding
  ├── public_endpoint_binding
  └── record_action_binding
form_submission
```

Marketing may contribute campaign attribution, consent, abuse protection, conversion rules, and public-endpoint defaults. Workflow contributes step, assignee, field policy, draft/return/resubmit, and transition behavior. Both use the same designer, definition versions, validation, and submission engine.

The existing `runory.submission` is also Marketing-specific: it owns attribution, consent, spam/processing, and CRM conversion state. It must not compete with the canonical Forms response model. The v0.5 migration should make the immutable `form_submission` the source response and keep Marketing processing as a typed response/capture projection that references it. In the product, users see Responses in Forms and, where needed, a Marketing conversion queue—not two generic products both called Submissions.

Required form capabilities:

```text
datetime
long text
attachment/image evidence
acknowledgment/signature capture
step-specific field visibility/editability/required rules
server-side validation
conditional validation based on current command/step
read-only calculated and snapshot fields
repeatable groups and checklist blocks
draft and submit distinction for step forms
```

Checklist/evidence composition should be:

```text
form definition/version
  ├── ordinary field section
  ├── checklist section → references a versioned checklist template
  ├── evidence section → applies an evidence policy
  └── acknowledgment section

form submission
  ├── immutable submitted revision
  ├── checklist run and per-item result
  ├── evidence attachments
  └── generated/updated business deliverable
```

A checklist is not a second form engine. It is a reusable form block with execution semantics such as `pass / fail / not_applicable`, per-item notes, per-item evidence, completion progress, and failure-triggered transitions.

`service_report` may remain a stable FSM business object for search, reporting, and relations, but its visible name, content definition, required evidence, workflow step, and generation timing must be configurable. The same runtime must support names such as inspection report, installation confirmation, acceptance form, and maintenance report.

The user should not see separate navigation entries for workflow forms, FSM forms, and public forms. An administrator opens Forms, designs once, and binds the definition to the required usage. A technician sees only the bound form inside the assigned work step.

Required Forms experiences:

```text
design and version a form once
see where a form is used
configure usage-specific policy without cloning the design
complete a bound form inside assigned work or a record action
publish a governed public link when Marketing Capture is installed
review responses and revision history in context
```

Quote-line editing and Work Order aggregation are separate product compositions. They may reuse shared UI primitives, but they are not modeled as special cases of Forms.

### 6.4 Pillar D — Business Roles, Permission Enforcement, And Work Queues

Pack permission groups must become enforceable product behavior, not only stored metadata.

Minimum roles for the canonical scenario:

```text
sales operator
quote approver
dispatcher / service coordinator
technician
service reviewer
workspace admin
```

Required behavior:

```text
commands check object/action permission
workflow steps can target permission groups or explicit users
assigned work can be limited to the assignee where policy requires it
users see My Work, Unassigned, Overdue, and Team queues
reassignment is audited
technician business object can link to a workspace member
```

Fine-grained enterprise ABAC is not required. The v0.5 requirement is a coherent enforcement path for the canonical roles and commands.

### 6.5 Pillar E — Work Assignment, Scheduling, And Spatial Planning

Dispatch must be a reusable assignment and scheduling capability, not a status field hard-coded into `work_order`.

Required shared concepts:

```text
work item          — the schedulable work
resource           — person, crew, or other execution resource
assignment         — versioned relationship between work and resource
schedule slot      — planned start/end, timezone, and all-day semantics
location           — address and optional stored coordinates
dispatch policy    — eligible resources and action permissions
```

Required assignment lifecycle:

```text
unassigned
offered / assigned
accepted
rejected
reassigned
cancelled
completed
```

Required commands:

```text
work.assign
work.accept_assignment
work.reject_assignment
work.reassign
work.schedule
work.reschedule
work.unschedule
work.cancel_assignment
```

All schedule mutations must use these commands. Dragging a calendar event or resource bar must not directly edit date or assignee fields.

Required views over the same assignment and schedule source:

```text
backlog / unscheduled list
day, week, and month calendar
resource timeline / simplified Gantt
basic location map with status/resource filters
```

The resource timeline is the primary team-progress view for FSM: one row per technician/resource and one horizontal bar per scheduled work item. It must show duration, overlap, status, and unavailability, and support governed drag-to-assign/reschedule actions.

The basic map view uses stored service-site coordinates and does not include routing, navigation, live tracking, automatic dispatch, or route optimization.

### 6.6 Pillar F — Commercial Document And Pricing Runtime

Quote is a reusable commercial capability, not an FSM-only feature. It should remain a typed business object built on a shared commercial-document runtime rather than becoming one universal `document_type` table.

Required shared capabilities:

```text
document number
header and counterparty context
versioned line items
price-book lookup and price snapshot
quantity, unit price, discount, tax, and total calculation
currency consistency
revision lineage and comparison
approval and withdrawal integration
immutable approved/accepted snapshot
source and downstream object links
print/export data contract
```

Recommended ownership:

```text
shared business modules:
  runory.product-service
  runory.price-book
  runory.commercial-document runtime

typed commercial module:
  runory.quote

pack experience:
  Sales Quote Pack
```

Future estimates, budgets, sales orders, purchase documents, and invoices may reuse the document kernel while retaining separate objects, commands, and invariants.

### 6.7 Pillar G — CRM, Quote, And FSM Domain Composition

#### CRM

Keep the current nucleus:

```text
company
contact
deal / opportunity
task
```

Required uplift:

```text
generated business numbers where needed
owner/assignee references to members instead of free text
deal stage changes through governed commands
won/lost reason and timestamp
create quote from deal with customer/contact copied as references
customer timeline showing deal, quote, work order, visit, and report events
```

Decision on lead:

```text
XLink and mature CRM products both provide an explicit lead-qualification boundary.
Runory must not treat company/contact lifecycle as automatically equivalent.
```

## 7. Canonical Commercial FSM Workflow

The release must ship and test one opinionated workflow, not just expose generic primitives.

```text
Deal: qualified
  → create quote draft

Quote: draft
  → submit approval form
  → pending approval
  → reject back to draft/revision
  → or approve
  → send/accept as recorded business actions

Accepted quote
  → idempotently create linked work order

Work order: new
  → triage form
  → ready for dispatch
  → move from backlog onto technician resource timeline
  → assign technician and schedule visit through governed commands
  → surface time conflicts and location on calendar/map
  → scheduled

Service visit: scheduled
  → technician starts visit
  → in progress
  → complete checklist and service report with evidence
  → submitted for review

Service review
  → reject to technician for rework
  → resubmit
  → or approve

Approved service report
  → complete visit
  → if all required visits/tasks are complete, complete work order

Exception paths
  → block/unblock
  → reassign/reschedule
  → withdraw quote
  → cancel visit/work order
  → reopen completed work order under authorized policy
```

Every arrow above must correspond to a named command, permission check, audit event, and test.

## 12. Architecture Decisions To Lock Early

The team should explicitly decide these before broad implementation:

1. Each aggregate has one authoritative lifecycle: Work Order stage, Visit/Booking status, Assignment status, and Workflow Step status are distinct. Duplicate mirror fields are read-only projections, and cross-aggregate synchronization occurs only through governed commands.
2. A transition that changes business truth is executed through a command with an atomic or durable recoverable boundary.
3. Human steps always produce a work item, even if the first UI presents it inline on the record.
4. Step submissions are immutable revisions; returning work creates another revision.
5. Running instances pin a workflow definition version.
6. Official modules own command invariants; workspace extensions may add validated fields and safe guards but cannot bypass invariants.
7. Agent, UI, MCP, and automation call the same command catalog.
8. XLink's arbitrary scripting is not a v0.5 target; Runory keeps a governed declarative action model.
9. `runory.form` becomes the one shared Forms product; Workflow, Marketing, FSM, record actions, and future portals consume it through typed bindings.
10. Public endpoint is a form usage/channel policy, not a second product called Public Forms; existing Marketing-specific form data requires a governed 1.x → 2.0 migration.
11. Canonical `form_submission` stores the immutable response; Marketing-specific attribution, spam, consent, and conversion state is a typed projection/queue, not another generic Submission product.
12. Checklist is a reusable Forms block with execution semantics, not a separate form engine or a list of boolean fields.
13. Evidence requirements and business deliverable names/timing belong to binding/process configuration; `service_report` remains an optional stable FSM projection.
14. Dispatch is implemented by generic assignment and scheduling runtimes; FSM supplies technician, visit, and service-site defaults.
15. Calendar, resource timeline, map, and object detail are views over one schedule source and execute the same governed commands.
16. Quote is a typed shared business capability built on a commercial-document kernel, not an FSM-only object or a universal untyped document table.
17. Packs/modules/objects determine ownership and upgrade safety; role-based user jobs determine navigation and product information architecture.
18. Approval is a reusable Workflow work-item type with an immutable decision, not a separately editable business object per domain; Quote remains the sole owner of quote commercial state.

## 13. Product Design Review

This review applies one test to every v0.5 capability:

> Does this concept help an operator understand and complete a job, or is it an internal ownership/runtime boundary that should remain behind the product?

### 13.1 Findings And Corrections

| Risk found | Product correction | Internal boundary retained |
|---|---|---|

### 13.4 Review Verdict

The plan is product-coherent after the corrections above, with one continuing risk: the P0 scope is broad. The team should protect the core journey across My Work, Customers/Opportunities, Quotes, Planning, Work Orders, and Forms, and avoid spending v0.5 capacity on generic builders, extra top-level object pages, or advanced optimization before the canonical journey passes.

## 14. Final Scope Judgment

The initial assumption is correct: `CRM + FSM + Sales Quote` are the right three core packs for v0.5.

The important refinement is:

> The version should not be defined by those three packs being present. It should be defined by a production-grade execution and planning spine connecting them while exposing one reusable Forms product plus shared workflow, assignment, scheduling, planning-view, and commercial-document capabilities.

If the team only enriches their object fields, v0.5 will still be a stronger demo.
