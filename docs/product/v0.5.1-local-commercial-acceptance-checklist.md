# v0.5.1 Local Commercial Acceptance Checklist

Status: Active v0.5.1 acceptance-hardening gate
Owner: Product + Engineering
Last updated: 2026-07-11

## 1. Purpose

This checklist turns the v0.5/v0.5.1 commercial FSM direction into a concrete local acceptance gate.

The goal is not to prove that Runory already has every enterprise FSM capability. The goal is to prove that a non-technical SMB user can understand and operate the core commercial loop without relying on hidden technical modules:

CRM → Quote → Work Order → Dispatch/Schedule → Field Visit → Form/Checklist/Evidence → Service Report → Completion.

For v0.5.1, Workflow, Automation, Outbox, MCP, Entitlements, and Extensions must be optional enhancement layers. They may enrich the business process, but they must not be required for the default SMB execution path.

## 2. Acceptance principle

v0.5.1 is accepted only when these three statements are true:

1. Core FSM execution can run through governed commands without starting a workflow instance.
2. Demo data is readable, connected, and operational from the UI, not merely inserted into tables.
3. Product navigation presents the business journey clearly; technical modules do not become the user's mental model.

## 3. Local setup

```bash

node scripts/sync-resources.mjs

```

Avoid starting the app through a command that triggers an interactive package-manager prompt during closure verification.

## 5. Validator hard gates

The local validator must report these as pass:

- Schema and migration baseline is present.
- CRM Lite Pack, Sales Quote Pack, and FSM Pack are installed.
- Companies, quotes, work orders, visits, reports, resources, assignments, schedule entries, and form definitions are seeded.
- At least one technician resource is linked to a user.
- Work-order-to-company references resolve cleanly.
- Active Workflow instances are not required for the SMB core path.
- Field evidence exists in submitted form data.
- Service reports include evidence/photo references.
- Planning has geolocated customer-site entries for the map view.
- Planning includes at least one conflict example for dispatcher review.
- Open work orders and service visits are current-date-safe after demo reseeding.
- The retired Quote Approval module is not installed in the demo workspace.
- The Sales Quote template does not expose Quote Approvals as a top-level product surface.
- `.DS_Store` files are absent from the source app tree.

Current v0.5.1 optional-enhancement items:

- Workflow demo instances may be `0`.
- Work Items may be `0`.

These are not blockers for the SMB dependency-decoupling gate. The default My Work experience is allowed to be schedule-backed operational work. Workflow-backed work items are the richer approval/orchestration layer, not a prerequisite for ordinary SMB execution.

## 6. Manual product acceptance path

The reviewer should verify the product as an SMB operator, not as a developer.

Open:

```text
http://localhost:3000/w/<workspace-slug>
```

Then verify:

1. The workspace opens without requiring any setup knowledge.
2. Navigation makes the commercial journey visible: CRM, quotes, work orders, visits/dispatch, calendar or planning, forms, and reports.
3. Demo records are human-readable: names, customers, service context, statuses, dates, and assignments should look like real operations.
4. Opening a work order reveals enough related context to understand the customer, site/property, status, schedule, assigned technician, visit, form/checklist, evidence/report path, and next action.
5. A dispatcher can understand who is assigned to what.
6. A field-service visit can be understood as a job to execute, not only as a raw record.
7. Form/checklist/report concepts are presented as business execution artifacts, not as generic technical schema.
8. The user is not forced to open Workflow or Automation to understand the current job status.
9. The user is not forced to open internal IDs or system tables to understand relations.
10. The default demo should support a credible commercial walkthrough in 10-15 minutes.

## 7. Product boundary gates

These are the specific v0.5.1 boundary checks that prevent the product from drifting into technical-module thinking.

| Area | v0.5.1 accepted behavior | Not accepted |
| --- | --- | --- |
| Workflow | Optional orchestration layer | Required to move a normal SMB work order |
| Forms | Business execution artifact | Standalone generic form builder as the main story |
| Calendar/Planning | Operational view for dispatch and schedules | Isolated calendar module unrelated to jobs |
| Map | Dispatch/site context view | Independent GIS-style module |
| Approvals | Reusable policy/command guard | Per-module top-level approval pages such as quote-only approval |
| Reports | Output of completed service work | Separate top-level reporting object with no job context |
| Assignments | Core scheduling/dispatch capability | Hidden internal table only visible to developers |
| Status | Business state driven by commands | Duplicate workflow state competing with object status |

## 8. Current closure implementation evidence

The current closure implementation adds or verifies:

- `work_order.start` as a governed command from `planned`/`reopened` to `in_progress`.
- Command API permission coverage for `work_order.start`.
- FSM pack permissions and governed-field metadata for the new command.
- Demo resources linked to users.
- Demo assignments between resources and visits/work orders.
- A validator script for local v0.5 closure checks.
- A closure test proving the core FSM lifecycle can run without workflow instances.
- A tolerant migration for `quote.return_reason`.
- v0.5.1 hardening checks for form evidence, service-report photos, geolocated schedules, schedule conflicts, retired Quote Approval surfaces, and source-tree `.DS_Store` cleanup.
- Existing demo workspaces can be hardened in place by re-running FSM demo seeding; the installer patches missing demo presentation fields without deleting business records or changing their IDs.
- Relative-date schedule demo data is idempotent by operational subject/resource, not by the generated date string, so re-running the seed on a later day updates the same schedule rows instead of duplicating My Work and Planning entries.
- Open FSM demo work orders, visits, schedules, actual time, and service-report completion time are relative-date based, so the local walkthrough remains credible when the demo is refreshed later.
- The Timeline API resolves subjects through current pack business-table names, so visit/work-order history no longer depends on old pluralized table assumptions.

## 11. v0.5.1 Acceptance Hardening Items

This section records the closure work added after the initial v0.5.1 implementation, based on the product review question: can this be handed to a non-technical user for a smooth FSM walkthrough?

| Item | Hardening requirement | Evidence |
| --- | --- | --- |
| Demo business story | Demo records must show a coherent FSM story, not only table rows. | Validator requires CRM, quote, work order, visit, report, assignment, schedule, form definition, and form submission data. |
| UI/UX readiness | Product navigation must present the business journey and hide technical engines from the default mental model. | Quote Approval retired from active pack/template; Workflow/Automation remain optional management surfaces. |
| Mobile/PWA field work | Mobile shell must expose My Work, Schedule, Work Orders, Visit execution, and form submission against the same API/data. | Mobile routes under `/m`; service worker shell; form renderer supports checklist/evidence/signature. |
| Mobile shell boundary | Mobile routes must stay inside the mobile execution shell and must not link users into `/w/...` desktop record pages for core field-work objects. | Work order and visit drill-downs use `/m/w/...`; desktop navigation/left menu must not appear in mobile field-work walkthroughs. |
| Pack-aware mobile navigation | Bottom navigation must be contributed by installed packs and intended mobile roles, not hardcoded to a single industry pack or allowed to grow without bound. | CRM contributes Customers, Sales Quote contributes Quotes, FSM contributes Schedule and Work Orders; the mobile shell composes installed pack contributions via `mobileNavigation`, keeps the bottom bar bounded, and exposes overflow in Explore. |
| Forms / checklists / evidence / report | Evidence and sign-off must appear in the demo execution trail. | Demo seed includes `evi-photos` attachments in service-deliverable submissions and photo references in service reports; validator checks both. |
| Current-date demo credibility | Open demo jobs must not become stale just because the calendar moved. | FSM demo seed uses `$relativeDate`; installer patches existing open work-order/visit/report dates; validator checks stale open jobs. |
| Planning | Planning must go beyond Today and support dispatcher review across calendar/resource/map contexts. | Day/week/month, calendar/timeline/resource/map views; API supports `from`/`to`; validator checks geolocated entries and conflict examples. |
| Workflow optionality | Workflow must not be required for the default SMB execution path. | Validator allows zero Workflow instances/work items and requires schedule-backed My Work subjects to resolve. |
| Quote approval boundary | Quote approval must not reappear as a per-module top-level product page. | Active Sales Quote pack/template no longer include `runory.quote-approval` or `/quote-approvals`; legacy module remains `status: retired`. |
| Source hygiene | Local Mac metadata must not remain in source routes/components. | Validator checks no `.DS_Store` in `apps/cloud/src`. |

## 12. Release decision

v0.5.1 can proceed to implementation closure when:

- Automated gates pass.
- Validator hard gates pass.
- The manual product acceptance path can be completed without explaining internal architecture.
- Remaining warning-level items are either resolved or explicitly moved into the next milestone with owner and rationale.
