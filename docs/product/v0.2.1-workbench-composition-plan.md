## 1. Version Thesis

`v0.2` proved that a small business user can enter a Runory workspace and primarily experience a business operating space — workbench shows metrics, trends, open tasks, and recent activity.

However, the `v0.2` workbench is **hardcoded**:

- The dashboard page component writes business metric cards, trend chart, and lists directly in JSX.
- The stats API writes SQL queries against `customer`, `contact`, `task` tables directly.
- The module manifest schema has no way for a module to declare dashboard widgets.
- The pack manifest has no way to curate a default workbench layout.
- Users cannot customize which widgets appear, in what order, or with what parameters.

`v0.2.1` should prove that the workbench is **composable and customizable**:

> When a user installs a pack, the workbench is assembled from widgets contributed by the installed modules, curated by the pack's default layout, and personalized by the workspace.

The version should move Runory from "a hardcoded business dashboard" toward "a composable workbench that follows the same Module / Pack / Workspace model as the rest of the platform."

## 2. Problem Statement

### 2.1 Current Limitations

The `v0.2` workbench has three structural gaps:

```text
1. Module coupling
   The dashboard queries customer/contact/task tables by name.
   Uninstalling the task module leaves orphaned queries (silently caught, but semantically wrong).
   Installing a new module (e.g. runory.deal) cannot surface on the workbench without code changes.

2. Pack opacity
   The CRM Lite Pack installs three modules but does not declare which widgets should appear on the workbench.
   The "curated default experience" is implicit in the dashboard component, not in the pack manifest.

3. User rigidity
   A workspace admin cannot hide a metric card, reorder a list, or add a widget from another installed module.
   The workbench is the same for every workspace that installs the same pack.
```

### 2.2 Why v0.2.1

This is a patch on `v0.2`, not a new minor version, because:

- It does not introduce new platform capabilities — it extends an existing model (Module declares, Pack curates, Workspace personalizes) that already governs objects, views, and navigation.
- It does not break the existing trial journey — the current hardcoded workbench becomes the reference implementation that gets migrated to the new mechanism.
- It does not require schema migrations of business data — only a new runtime table for layout persistence.

## 3. Core Design Principle

### Three-Layer Composition

The workbench follows the same three-layer composition model as the rest of Runory:

```text
Layer 1 — Module declares available widgets
          (capability inventory, like objects/views/permissions)

Layer 2 — Pack curates default layout
          (best-practice curated experience, like pack composition)

Layer 3 — Workspace stores personalization
          (user customization, like managed workspace extensions)
```

This mirrors:

```text
Module  → objects, views, permissions, navigation, dashboard.widgets
Pack    → modules, defaultNavigation, dashboard.defaultLayout
Workspace → installed packs, navigation overrides, dashboard layout overrides
```

### Design Rules

- Modules declare **capabilities**, not layouts. A module says "I can contribute an open-tasks metric card," not "I should appear first in the metrics zone."
- Packs curate **layouts**, not data. A pack says "the open-tasks metric should appear in the metrics zone at position 3," not "query the task table."
- Workspaces store **overrides**, not redefinitions. A workspace says "hide the done-tasks metric, move open-tasks to position 1," not "replace the entire layout."
- Widget data is resolved by the **platform**, not by modules. Modules declare data intent (`count` / `group_count` / `recent` / `timeseries`); the platform translates intent into safe SQL against the module's declared objects.

## 4. Product Design Decisions

The following decisions are fixed for `v0.2.1`. They intentionally favor a stable business workbench over a fully general dashboard builder.

### 4.1 Zone Model Decision

Decision: **do not support custom zones in `v0.2.1`.**

`v0.2.1` keeps four platform-defined zones with fixed vertical order:

```text
metrics → trends → lists → activity
```

Workspace admins can personalize widgets **within** these zones: reorder, hide, show, add, remove, and configure. They cannot create new zones, rename zones, or move a widget to an arbitrary custom layout container.

Rationale:

- The workbench is the product home surface, not a generic BI dashboard.
- Fixed zones preserve visual consistency across packs and workspaces.
- Mobile layout, empty states, skeletons, and accessibility are much easier to make production-grade with known zones.
- Pack authors get a predictable curation target.
- Custom zones would force early decisions about responsive grid semantics, naming, permissions, and conflict resolution that are not needed for the 0.2 trial journey.

Future extension:

- `v0.3+` may introduce a platform-managed zone registry, but zones should still be declared by the platform or template layer, not ad hoc by individual modules.
- If custom zones are added later, they must be validated, named, ordered, responsive, and reversible through the same governed layout mechanism.

### 4.2 Multi-pack Merge Decision

Decision: **use deterministic install-order merge with explicit duplicate rules; defer intelligent conflict resolution UI.**

When multiple packs are installed:

1. Packs are processed in installation order.
2. Within each pack, zones are processed in canonical zone order.
3. Within each zone, widgets are appended by pack-declared order.
4. If a later pack references an already-present layout item, the earlier item wins.
5. If a pack intentionally wants the same module widget more than once with different config, it must provide an explicit `instance` key.
6. Workspace overrides apply after pack merge and always win over pack defaults.

Widget layout identity:

```text
layout item id = zone + module + widget + instance
```

If `instance` is omitted, it defaults to `default`.

Example:

```yaml
dashboard:
  defaultLayout:
    - zone: trends
      widgets:
        - { module: runory.task, widget: new_tasks_trend, instance: "14d", config: { data: { range: "14d" } } }
        - { module: runory.task, widget: new_tasks_trend, instance: "30d", config: { data: { range: "30d" } } }
```

Rationale:

- Deterministic behavior is more important than "smart" behavior at this stage.
- Silent reordering or automatic conflict choices would make pack installation hard to explain.
- Install order is easy to audit and easy to debug.
- Workspace admins can resolve visible layout preferences through Edit Mode.
- Explicit `instance` prevents a runtime-table mismatch when the same widget appears multiple times.

Deferred:

- No automatic priority scoring across packs.
- No semantic deduplication such as "these two metrics are similar."
- No conflict resolution wizard in `v0.2.1`.
- No per-pack layout priority beyond installation order.

### 4.3 Activity Feed Ownership Decision

Decision: **`activity_feed` is platform-owned in `v0.2.1`; modules cannot declare activity-feed widgets.**

Modules may not declare `type: activity_feed` in `dashboard.widgets`. The platform contributes the canonical `_platform/business_activity_feed` widget.

Modules can still affect the feed indirectly by emitting governed audit/domain events through platform APIs. The feed may render module-specific labels, icons, and entity links based on event metadata, but the feed query, visibility, ordering, retention, and tenant-safety rules belong to the platform.

Rationale:

- Activity is a trust surface. It should be complete, chronological, permission-aware, and tenant-safe.
- Multiple module-owned feeds would fragment the user's understanding of "what changed recently."
- Modules should not define their own audit/read models for the workbench.
- A canonical feed lets Runory improve filtering, grouping, and summarization centrally.

Future extension:

- `v0.3+` may allow modules to declare **activity descriptors** for better labels and grouping, for example "Task status changed" or "Customer field updated".
- These descriptors should enrich the platform feed; they should not create independent module-owned feeds by default.

## 5. Module Widget Declaration

### 5.1 Manifest Extension

A new `dashboard.widgets` section is added to the module manifest:

```yaml
dashboard:
  widgets:
    - key: open_tasks_metric
      type: metric_card
      label: Open Tasks
      icon: list-checks
      tone: amber
      data:
        kind: count
        object: task
        where: "status in ('todo','in_progress')"
      sub:
        kind: count
        object: task
        where: "status in ('todo','in_progress') and due_date < today"
        template: "{count} overdue"
      link: /tasks
      configurable:
        - path: sub.template
          label: Sub-label text
          type: text

    - key: task_status_breakdown
      type: breakdown
      label: Task Status Breakdown
      data:
        kind: group_count
        object: task
        groupBy: status
      configurable:
        - path: data.groupBy
          label: Grouping field
          type: select
          options: [status, priority, assignee]

    - key: open_tasks_list
      type: list
      label: Open Tasks
      data:
        kind: recent
        object: task
        where: "status in ('todo','in_progress')"
        orderBy: "priority desc, due_date asc"
        limit: 8
        columns: [title, status, priority, due_date, assignee]
      link: /tasks
      configurable:
        - path: data.limit
          label: Number of rows
          type: number
          min: 1
          max: 20
        - path: data.columns
          label: Display columns
          type: multiselect
          options: [title, status, priority, due_date, assignee]

    - key: new_tasks_trend
      type: trend_chart
      label: New Tasks Trend
      data:
        kind: timeseries
        object: task
        range: 14d
        groupBy: day
      configurable:
        - path: data.range
          label: Time range
          type: select
          options: ["7d", "14d", "30d"]
```

### 5.2 Widget Type Catalog

The initial widget types supported in `v0.2.1`:

| type | purpose | data.kind | rendering |
|---|---|---|---|
| `metric_card` | single value + sub label | `count` | icon, value, sub text, link |
| `trend_chart` | time series bar/line chart | `timeseries` | SVG chart with range selector |
| `breakdown` | distribution chart | `group_count` | donut or horizontal bars |
| `list` | recent records table | `recent` | columns, rows, link to full list |
| `activity_feed` | audit timeline | platform-built-in | does not use module `data` |

The `activity_feed` type is reserved for platform-contributed widgets that read from the audit log. Modules do not declare activity feeds.

### 5.3 Data Intent Grammar

Widget `data` is a **declarative intent**, not raw SQL. The platform resolves intent into safe SQL.

```text
kind: count | group_count | recent | timeseries
object: <must be a declared object of this module>
where: <restricted expression>
orderBy: <field> <asc|desc>, ...
limit: <int>
groupBy: <field>            (for group_count / timeseries)
range: 7d | 14d | 30d       (for timeseries)
columns: [<field>, ...]     (for recent)
```

### 5.4 Where Expression Safety

The `where` expression is parsed by the platform, not interpolated into SQL. Supported grammar:

```text
where := clause (and clause)*
clause := field operator value
operator := = | != | in | not in | < | <= | > | >= | like
value := string | number | "today" | "now"
```

Rules:

- `field` must be a declared field of the object (validated against manifest).
- `value` is parameterized — never concatenated into SQL.
- `today` / `now` are platform-resolved date/time constants.
- Unsupported operators or unknown fields cause the widget to render an error state, not a SQL failure.

### 5.5 Ownership and Validation

- `data.object` must be an object declared by the **same module**. A module cannot query another module's objects directly. Cross-module references (e.g. task.customer_id) are handled via relation fields in the object schema, not via widget data.
- Widget keys must be unique within a module.
- Modules cannot declare `activity_feed` widgets.
- The Catalog validation pipeline validates widget declarations at module build time, including:
  - `data.object` exists in the same manifest.
  - `where` / `orderBy` / `groupBy` / `columns` reference only declared fields.
  - `configurable` paths point to real schema locations.

## 6. Pack Default Layout

### 6.1 Pack Manifest Extension

A new `dashboard.defaultLayout` section is added to the pack manifest:

```yaml
dashboard:
  defaultLayout:
    - zone: metrics
      widgets:
        - { module: runory.customer, widget: customer_total_metric }
        - { module: runory.contact,  widget: contact_total_metric }
        - { module: runory.task,     widget: open_tasks_metric }
        - { module: runory.task,     widget: done_tasks_metric }

    - zone: trends
      widgets:
        - { module: runory.customer, widget: new_customers_trend }
        - { module: runory.task,     widget: new_tasks_trend }

    - zone: lists
      widgets:
        - { module: runory.task,     widget: open_tasks_list }
        - { module: runory.customer, widget: recent_customers_list }

    - zone: activity
      widgets:
        - { module: _platform,       widget: business_activity_feed }
```

### 6.2 Zone Model

The workbench is organized into zones. Initial zones for `v0.2.1`:

```text
metrics   — top row of metric cards (typically 4 cards)
trends    — trend chart section (typically 1-2 charts)
lists     — two-column list section (typically 2 lists)
activity  — full-width activity feed (typically 1 feed)
```

Zones are rendered in a fixed vertical order: `metrics → trends → lists → activity`. Horizontal layout within a zone is controlled by the zone's grid (e.g. metrics is a 4-column grid, lists is a 2-column grid).

Custom zones are not supported in `v0.2.1`. See [4.1 Zone Model Decision](#41-zone-model-decision).

### 6.3 Curation Rules

- A pack can only reference widgets declared by modules **included in the same pack** (or platform-built-in widgets).
- A pack can reference the same widget multiple times with different config overrides only when each layout item has a distinct `instance`.
- If a pack does not declare `dashboard.defaultLayout`, the workbench falls back to rendering all available widgets from installed modules in declaration order. This is a graceful default, not a recommended practice.
- Multi-pack layout merge follows deterministic installation-order append rules. See [4.2 Multi-pack Merge Decision](#42-multi-pack-merge-decision).

### 6.4 Platform-Built-in Widgets

The platform contributes a small set of built-in widgets under module `_platform`:

```text
_platform/business_activity_feed   — audit timeline
_platform/pack_status              — installed packs summary (informational)
```

These are available to any pack's default layout and cannot be overridden by modules.

## 7. Workspace Personalization

### 7.1 Runtime Table

A new runtime table stores workspace-level layout overrides:

```sql
CREATE TABLE runory_runtime_workspace_dashboard_layout (
  workspace_id     TEXT NOT NULL,
  zone             TEXT NOT NULL,
  widget_module    TEXT NOT NULL,
  widget_key       TEXT NOT NULL,
  widget_instance  TEXT NOT NULL DEFAULT 'default',
  position         INT  NOT NULL,
  hidden           BOOLEAN NOT NULL DEFAULT 0,
  config_override  TEXT,
  updated_at       TEXT NOT NULL,
  updated_by       TEXT NOT NULL,
  PRIMARY KEY (workspace_id, zone, widget_module, widget_key, widget_instance)
);

CREATE INDEX idx_wdl_workspace ON runory_runtime_workspace_dashboard_layout(workspace_id);
```

### 7.2 Personalization Capabilities

A workspace admin can:

```text
Reorder   — change position within a zone
Hide      — set hidden = true
Show      — set hidden = false (unhide a pack-hidden widget)
Add       — insert a widget from the available widget inventory
Remove    — remove a widget from the layout (sets hidden = true)
Configure — override configurable fields declared by the module
Reset     — clear all overrides, return to pack default layout
```

### 7.3 Override Resolution

The effective layout is resolved by merging three sources:

```text
1. Available widgets
   = union of dashboard.widgets across all installed modules
   + platform-built-in widgets

2. Default layout
   = first installed pack's dashboard.defaultLayout
   (if multiple packs are installed, they merge in installation order;
    later packs can append widgets to existing zones)

3. Workspace overrides
   = workspace_dashboard_layout rows for this workspace
   - hidden widgets are removed
   - position overrides reorder visible widgets
   - config_override merges into widget.config / widget.data
```

If a widget in the default layout is no longer available (module uninstalled), it is silently dropped. If a workspace override references an unavailable widget, the override is preserved but ignored (so reinstalling the module restores the personalization).

### 7.4 Permission Model

- **Workspace admin / owner**: full personalization capabilities.
- **Workspace member**: views the effective layout; cannot personalize.
- Personalization is workspace-scoped, not user-scoped. All members of a workspace see the same workbench. (User-scoped personalization is a future non-goal for `v0.2.1`.)

## 8. Rendering Layer

### 8.1 Dashboard Page Refactor

The current [dashboard/page.tsx](../../apps/cloud/src/app/w/[workspaceId]/dashboard/page.tsx) is refactored from a hardcoded component to a **composition shell**:

```text
1. Fetch installed modules → collect available widgets
2. Fetch installed pack(s) → resolve default layout
3. Fetch workspace_dashboard_layout → apply overrides
4. Render zones in fixed order
5. For each visible widget, render <WidgetRenderer> with resolved config
6. Each WidgetRenderer fetches its own data independently
```

### 8.2 Widget Data API

A new widget-scoped data endpoint replaces the monolithic stats API:

```text
GET /api/workspaces/[id]/widgets/[module]/[key]
  → resolves widget declaration
  → resolves workspace config_override
  → translates data intent into safe SQL
  → returns widget payload
```

Benefits:

- Widgets load independently — a slow trend chart does not block metric cards.
- Widgets fail independently — one broken widget shows an error state, not a broken dashboard.
- Widgets refresh independently — metric cards can poll every 30s while trend charts refresh on demand.

The existing `/api/workspaces/[id]/stats` endpoint is preserved for backward compatibility during `v0.2.1` and deprecated in `v0.2.2`.

### 8.3 Widget Renderer Contract

Each widget type has a renderer component that accepts a normalized props contract:

```typescript
interface WidgetProps<TData> {
  widget: WidgetDeclaration;
  config: ResolvedConfig;      // merged default + override
  data: TData | null;
  loading: boolean;
  error: Error | null;
  workspaceId: string;
  onRefresh: () => void;
}
```

The platform ships renderers for all five widget types in section 5.2. Modules do not ship custom renderers in `v0.2.1` — this keeps the rendering surface governed and consistent. (Custom renderers are an explicit non-goal for `v0.2.1`.)

### 8.4 Edit Mode

The workbench gains an **Edit Mode** toggle (visible to workspace admins only):

```text
View Mode
  → widgets render normally
  → no drag handles, no config buttons

Edit Mode
  → widgets show drag handles
  → widgets show hide/configure/remove buttons on hover
  → zones show "drop here" targets
  → a side panel lists available widgets not yet on the layout
  → "Save" persists overrides; "Cancel" discards
  → "Reset to default" clears all overrides
```

Edit Mode uses native HTML5 drag-and-drop (no heavy DnD library) for `v0.2.1`. Visual feedback is kept simple: dragged widget becomes semi-transparent, drop target shows a dashed outline.

## 9. Implementation Phases

### Phase 1 — Schema and Resolver (foundation)

Goal: Module and Pack manifests can declare widgets and layouts; the platform can resolve them.

Required outcomes:

- `dashboard.widgets` schema added to module manifest contract.
- `dashboard.defaultLayout` schema added to pack manifest contract.
- Catalog validation pipeline validates widget declarations.
- Widget data resolver implemented (translates `data` intent into safe SQL).
- `/api/workspaces/[id]/widgets/[module]/[key]` endpoint implemented.
- Unit tests cover: where expression parsing, data intent resolution, cross-module object validation.
- Unit tests cover: fixed zone validation, multi-pack deterministic merge, duplicate widget handling via `instance`, and rejection of module-declared `activity_feed`.

### Phase 2 — Layout Resolution and Rendering (migration)

Goal: The workbench renders from declared widgets instead of hardcoded components.

Required outcomes:

- Layout resolver implemented (available widgets + pack default + workspace overrides).
- `workspace_dashboard_layout` table created.
- Widget renderers implemented for all five widget types.
- Dashboard page refactored to composition shell.
- CRM Lite Pack manifest updated with `dashboard.defaultLayout`.
- `runory.customer`, `runory.contact`, `runory.task` manifests updated with `dashboard.widgets`.
- Existing `/api/workspaces/[id]/stats` endpoint preserved but marked deprecated.
- Acceptance: the refactored workbench visually matches the `v0.2` workbench for the same data.

### Phase 3 — Personalization UI (user-facing)

Goal: Workspace admins can customize the workbench without editing JSON.

Required outcomes:

- Edit Mode toggle implemented.
- Drag-and-drop reorder within zones.
- Hide / show / remove widget actions.
- Configure widget panel (renders `configurable` fields declared by module).
- Add widget side panel (lists available widgets not yet on layout).
- Save / Cancel / Reset actions.
- All personalization actions audited via the existing audit log.
- Acceptance: a workspace admin can hide the done-tasks metric, reorder the open-tasks list to position 1, and change the trend chart range to 30d, all through the UI.

### Phase 4 — Polish and Deprecation

Goal: The feature is production-ready for Early Access.

Required outcomes:

- Loading skeletons per widget.
- Error states per widget (with retry).
- Empty states per widget type.
- Mobile responsiveness for zones.
- Audit log entries use business language ("hid the Workbench widget Open Tasks", "changed the Customer Total Trend time range to 30 days").
- Documentation updated: module authoring guide includes dashboard.widgets section.
- Existing stats API marked deprecated in code and response headers.

## 10. Explicit Non-goals for 0.2.1

These are intentionally excluded from `v0.2.1` to keep the patch focused:

- **User-scoped personalization.** All members of a workspace see the same workbench. Per-user layouts are a future consideration.
- **Custom widget renderers.** Modules cannot ship their own React components. All rendering goes through the five built-in widget types.
- **Cross-workspace layout templates.** Layouts cannot be exported/imported across workspaces.
- **Widget marketplace.** Widgets are only contributed by installed modules — there is no standalone widget catalog.
- **Advanced chart types.** Only bar/line/donut/horizontal-bar are supported. Complex visualizations (scatter, heatmap, funnel) are deferred.
- **Real-time streaming.** Widgets poll on intervals; there is no WebSocket push for `v0.2.1`.
- **Widget permissions.** All workspace members see all widgets. Field-level / record-level ACL on widget data is a future concern.
- **Multi-pack layout conflicts.** When multiple packs are installed, their default layouts merge in installation order. A sophisticated conflict resolution UI is deferred.
- **Custom zones.** The platform owns the four canonical zones in `v0.2.1`.
- **Module-owned activity feeds.** Modules enrich the canonical platform feed through governed events; they do not contribute independent feed widgets.

## 11. Migration and Compatibility

### 11.1 Backward Compatibility

- Existing `v0.2` workbench behavior is preserved during `v0.2.1` rollout.
- Modules without `dashboard.widgets` continue to work — they simply do not contribute widgets.
- Packs without `dashboard.defaultLayout` fall back to rendering all available widgets in declaration order.
- The `/api/workspaces/[id]/stats` endpoint remains functional throughout `v0.2.1`.

### 11.2 Migration Path

```text
v0.2.1 Phase 2 complete
  → CRM Lite Pack migrates to declared widgets
  → hardcoded dashboard component is removed
  → stats API is deprecated but functional

v0.2.2
  → stats API is removed
  → all official modules migrate to dashboard.widgets

v0.3
  → user-scoped personalization considered
  → custom widget renderers considered
  → advanced chart types considered
```

### 11.3 Data Safety

- The new `workspace_dashboard_layout` table is a runtime table (prefix `runory_runtime_`), not business data.
- Uninstalling a module does not delete workspace layout overrides — the overrides are preserved and silently ignored, so reinstalling the module restores the personalization.
- Purging a workspace deletes all layout overrides along with other runtime data.

## 12. Trial Journey Extension

The `v0.2` trial journey is extended in `v0.2.1`:

```text
(Same as v0.2 up to "Review Audit")
→ Open Workbench
→ Enter Edit Mode
→ Hide the "Completed Tasks" metric card
→ Reorder "Open Tasks" list to the top of the lists zone
→ Change the trend chart range from 14d to 30d
→ Save
→ Verify the workbench reflects the personalized layout
→ Reset to default
→ Verify the workbench returns to the pack-curated layout
```

The user should not need to understand widget declarations, data intents, or layout resolution to complete this journey.

## 13. Definition of Done

`v0.2.1` is ready when:

- Module manifests can declare `dashboard.widgets` with all five widget types.
- Pack manifests can declare `dashboard.defaultLayout` with zone-based curation.
- The workbench renders from declared widgets, not hardcoded components.
- Workspace admins can reorder, hide, show, add, remove, and configure widgets through Edit Mode.
- Personalization persists across sessions and survives module reinstall.
- CRM Lite Pack and its three modules are migrated to the new mechanism.
- The refactored workbench visually matches the `v0.2` workbench for the same data.
- The trial journey extension passes end to end in a production-like environment.
- Widget data resolution is safe (no raw SQL injection, no cross-module object access).
- All personalization actions are audited in business language.
- Custom zones are rejected by validation.
- Module-declared `activity_feed` widgets are rejected by validation.
- Multi-pack merge behavior is deterministic and covered by tests.
