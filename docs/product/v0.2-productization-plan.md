## 1. Version Thesis

`v0.1` proves that Runory can safely host a Cloud workspace, install official business packs, apply governed Agent proposals, audit changes, and support a Catalog / Release control plane.

`v0.2` should prove that Runory is becoming a usable product:

> A small business user can enter a Runory workspace and primarily experience a business operating space, not a technical implementation console.

The version should move Runory from "architecture-complete MVP" toward "early product that a user can understand, navigate, and return to."

## 2. Core Product Principle

### Business Workspace First

The Workspace layer must become a **business workspace**.

The current implementation exposes several platform capabilities directly in the first-level workspace navigation: modules, workflows, audit, settings, Agent Proposal. These are real capabilities, but they are mostly implementation, administration, or platform-operator concerns.

For `v0.2`, the default workspace experience should answer:

```text
What is happening in my business?
What needs attention today?
Who are my customers and contacts?
What tasks are moving or blocked?
What changed recently?
```

It should not primarily answer:

```text
Which module installed this object?
What is the extension manifest?
Which audit event was emitted?
Which control-plane validation ran?
```

Those platform answers remain available, but they move behind management and operator surfaces.

## 3. Navigation Model

### 3.1 Global Layer

Global pages are outside a specific workspace.

```text
/dashboard
→ My Workspaces
→ Create Workspace
→ Account / Sessions
→ Organization-level entry points
→ Platform Admin entry point when allowed
```

This layer answers: "Which workspace or organization am I entering?"

### 3.2 Workspace Business Layer

This is the default daily-use surface.

Recommended first-level navigation:

```text
Workspace
├─ Workbench
├─ Customers
├─ Contacts
├─ Tasks
├─ Activity
└─ Manage
```

Chinese product labels can be:

```text
Workspace
├─ Workbench
├─ Customers
├─ Contacts
├─ Tasks
├─ Activity
└─ Manage
```

Rules:

- First-level navigation should be business-language first.
- Official module objects can appear as business entries when installed.
- Technical nouns should not dominate the ordinary user's path.
- Platform transparency remains available, but it is not the default daily frame.

### 3.3 Workspace Management Layer

Management is where workspace admins change how the workspace works.

```text
Manage
├─ Modules & Packs
├─ Customize Workspace
├─ Workflow Configuration
├─ Members & Access
├─ Audit Log
├─ Data Export
├─ API Keys
└─ Workspace Settings
```

Chinese product labels can be:

```text
Manage
├─ Modules & Packs
├─ Customize Workspace
├─ Workflow Configuration
├─ Members & Access
├─ Audit Log
├─ Data Export
├─ API Keys
└─ Workspace Settings
```

Rules:

- Agent Proposal should graduate from a demo inside Settings into `Customize Workspace`.
- Audit remains visible, but it is part of management and trust, not ordinary business navigation.
- Workflow has two surfaces:
  - business-visible workflow status and tasks;
  - admin-visible workflow configuration.
- Modules and Packs are management concerns after initial onboarding.

### 3.4 Platform Operator Layer

Platform operators need Catalog and Release visibility separate from workspace users.

```text
Platform Console
├─ Catalog Items
├─ Versions
├─ Validation Runs
├─ Releases
├─ Rollouts
├─ Compatibility Reports
└─ Platform Audit
```

This layer answers:

```text
What artifact exists?
What business schema will it install?
Is it validated?
Who approved release?
Which workspaces are affected?
Can rollout be paused or recovered?
```

## 4. 0.2 Landmark Work

### 4.1 Workspace Information Architecture

Goal: Users can always tell whether they are in global, workspace, management, or platform context.

Required outcomes:

- `/dashboard` is clearly "My Workspaces".
- `/w/:workspace/dashboard` becomes a business workbench, not a platform status page.
- Workspace shell has a clear return path to `My Workspaces`.
- First-level workspace navigation prioritizes business surfaces.
- Management capabilities move under `Manage`.
- Platform operator pages remain outside ordinary workspace navigation.

Acceptance:

- A user can enter a workspace, return to My Workspaces, and understand the difference between global dashboard and workspace workbench without explanation.
- A non-admin workspace member should not see first-level admin/platform controls as daily navigation.

### 4.2 CRM Lite Business Experience

Goal: CRM Lite feels like a small usable business workspace, not just metadata CRUD.

Required outcomes:

- Customers, contacts, and tasks have polished list/detail/edit flows.
- Demo data ships with official pack installation when installed from Cloud UI.
- Lists support basic search/filter/sort where useful.
- Relationships are understandable: contacts and tasks show their customer context.
- Business empty states explain what to do next.
- Workspace dashboard shows real business state: recent customers, open tasks, recent activity, and extension changes.

Acceptance:

- A first-time user can understand CRM Lite in less than 10 minutes.
- The workspace is not empty after installing the official pack from UI.
- Users can create, edit, and inspect customer-related records without seeing platform terminology.

### 4.3 Agent Customization Productization

Goal: Agent customization becomes a guided product flow, not a JSON editor.

Required outcomes:

- `Customize Workspace` page replaces the demo-only Agent Proposal placement in Settings.
- Add Field guided flow:
  - choose object;
  - choose field label/key/type;
  - choose list/form visibility;
  - generate governed plan;
  - validate;
  - preview diff;
  - approve and apply.
- Diff Preview explains changes in business language.
- Apply success leads to the exact business page where the change is visible.
- Rollback explains impact before execution.

Acceptance:

- A workspace admin can add a customer field without editing JSON.
- The underlying API contract remains `plan → preview → apply → audit → rollback`.
- Codex / MCP / SDK can still generate plans through the same governed API.

### 4.4 Management Center

Goal: Admin and technical controls are available without polluting daily business navigation.

Required outcomes:

- `Manage` entry point exists in workspace navigation.
- Modules & Packs, Customize Workspace, Workflow Configuration, Audit Log, API Keys, Export, and Settings live under Manage.
- Settings becomes smaller and focused on workspace metadata/preferences.
- Audit Log is readable as product evidence, not raw implementation logs.

Acceptance:

- Ordinary business navigation stays short and business-oriented.
- Workspace admin can still reach every governance and configuration function within two clicks.

### 4.5 Catalog Console Productization

Goal: Platform operators can inspect and release business modules with confidence.

Required outcomes:

- Catalog Item detail page shows type, publisher, visibility, status, and current versions.
- Version detail page shows manifest, objects, fields, views, permissions, extension points, checksum, and provenance.
- Validation run detail shows pass/fail evidence.
- Release and rollout pages show channel, approver, targets, progress, and failure state.
- Compatibility report is visible from both platform and workspace management contexts.

Acceptance:

- A platform operator can answer what a module installs, what changed, whether it passed validation, who approved it, and which workspaces are affected.

### 4.6 Trust and Account Surfaces

Goal: Early users can see that Runory is controlled, reversible, and not a black box.

Required outcomes:

- Account/session management is understandable.
- Export and data portability are easy to find.
- Members/invitations get a basic workspace management UI.
- Audit summaries are readable.
- Early Access limitations are visible but not alarming.

Acceptance:

- A user can find how to leave, export, review changes, and manage access without reading documentation.

## 5. Official 0.2 Trial Journey

The 0.2 trial should be a product journey, not a platform demo.

```text
Sign in with email
→ My Workspaces
→ Enter a workspace
→ See business workbench
→ Install CRM Lite with demo data
→ Review customers, contacts, tasks
→ Create or update a customer
→ Open Manage
→ Customize Workspace
→ Add Customer Tier through guided flow
→ Preview Diff
→ Approve & Apply
→ Return to Customers and fill the new field
→ Review Audit
→ Optional rollback
```

The user should not need to understand MCP, SDK, manifest files, migrations, or Catalog internals to complete this journey.

## 6. Explicit Non-goals for 0.2

These should not distract the version unless they become blockers:

- Full marketplace and third-party publisher onboarding.
- Complex custom roles / field ACL / record ACL.
- Full workflow automation builder.
- Billing plan complexity beyond Early Access visibility.
- Complete enterprise SSO.
- Fully generalized low-code app builder.
- Public ecosystem packaging.

## 7. Implementation Order

Recommended order:

1. Workspace navigation and management IA.
2. Business workbench and CRM Lite polish.
3. Customize Workspace guided flow.
4. Management Center pages.
5. Catalog Console detail pages.
6. Trust/account/member/export polish.
7. Browser-level 0.2 trial journey verification.

## 8. 0.2 Definition of Done

`v0.2` is ready when:

- Workspace first-level navigation feels like a business workspace.
- Technical/admin controls are grouped under management surfaces.
- CRM Lite is usable with demo data and basic relationships.
- Workspace admins can customize a field without editing JSON.
- Apply/rollback remains governed, audited, and reversible.
- Platform operators can inspect Catalog artifacts and release state.
- The official 0.2 trial journey passes end to end in a production-like environment.
