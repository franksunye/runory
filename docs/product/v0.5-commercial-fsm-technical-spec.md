## 1. Purpose And Normative Language

This specification turns the v0.5 product plan into implementation boundaries. It is the source of truth for v0.5 domain ownership, runtime contracts, API behavior, migration, and release acceptance.

The words **MUST**, **MUST NOT**, **SHOULD**, and **MAY** are normative. A change that contradicts a MUST requires an explicit architecture decision record and an update to this specification before implementation.

## 2. Release Contract

v0.5 MUST run this production-shaped journey without direct status editing:

```text
customer + opportunity
  -> versioned quote
  -> submit / approve / reject / return / withdraw
  -> accept quote
  -> idempotently create work order
  -> create visit
  -> assign and schedule resource
  -> execute required forms/checklists and capture evidence
  -> submit service deliverable
  -> review / return / resubmit
  -> complete or reopen under policy
```

The release is not accepted merely because the objects can be installed or edited. It is accepted when the journey is command-driven, auditable, recoverable, permission-enforced, and coherent across record detail, My Work, Planning, and APIs.

### 2.1 P0 Product Surfaces

```text
Workbench / My Work
Customers
Opportunities
Quotes
Planning: Calendar | Resource timeline | Map
Work Orders
Mobile PWA: Today / My Work | Schedule | Work Orders | Me
Forms (management)
Processes (management)
Members & Access (management)
Audit & Diagnostics (management)
```

Technical persistence does not imply navigation. `quote_line`, `service_visit`, `assignment`, `schedule_entry`, `work_item`, `form_submission`, `approval_decision`, checklist runs, and service deliverables MUST remain contextual unless a documented role has a recurring cross-record job that justifies a view.

### 2.2 Explicit Non-Goals

- payment, telephony, customer-service channel, ERP, or accounting integration;
- route optimization, travel-time optimization, or automatic dispatch optimization;
- arbitrary customer scripts in workflow nodes;
- a universal low-code business-object builder;
- a second Forms product for Workflow, FSM, Marketing, or public endpoints;
- a top-level approval product for every domain;
- offline-first mobile synchronization in v0.5.0.

## 3. Current-State Code Audit

The current repository is model-complete but not execution-safe.

| Finding | Current implementation | v0.5 consequence |
|---|---|---|
| Approval is duplicated | `runory.quote` owns `quote.status`; `runory.quote-approval` separately owns `quote_approval.status` | remove the second mutable lifecycle |
| Approval is exposed as CRUD | Quote Approval has create/update/delete permissions, list/form views, widgets, and `/quote-approvals` navigation | replace CRUD with governed decisions and My Work |
| Tests encode double writes | `sales-quote-pack.test.ts` updates Quote and Quote Approval independently | replace with one command and one transaction boundary |
| Workflow approval is a flag | `requiresApproval` means the caller must have workspace `admin` role | introduce assigned approval work and policy-based authorization |
| Pending approval is inferred | runtime scans instances for outgoing transitions with `requiresApproval` | persist work items; an available transition is not a pending request |
| Workflow history is JSON | instance has mutable `history_json` | add append-only events/decisions and version checks |
| State synchronization is best-effort | workflow state commits before record state and system actions | domain truth changes MUST be atomic or durably recoverable |
| Definitions are mutable | running instances reference `workflow_id`, not an immutable version | pin every instance to a definition version |
| FSM assignment is embedded | Work Order and Visit store technician identifiers directly | use a shared assignment lifecycle and project current assignment |
| Schedule is embedded | scheduled timestamps live on Work Order/Visit | establish one schedule source queried by all Planning views |
| Forms are Marketing-shaped | `runory.form` targets contact/company/deal and stores `fields_json` | migrate Forms to a channel-neutral definition/version/binding model |
| Dynamic objects receive product pages | persisted navigation and generic route conversion expose internal objects | add explicit presentation policy; installation alone MUST NOT create top-level IA |

## 4. Locked Architecture Decisions

### AD-01 — One Authoritative Lifecycle Per Aggregate

Each aggregate owns exactly one lifecycle:

| Aggregate/runtime | Authoritative state examples |
|---|---|
| Quote | `draft`, `in_review`, `approved`, `rejected`, `sent`, `accepted`, `declined`, `expired`, `withdrawn` |
| Work Order | `new`, `triaged`, `planned`, `in_progress`, `blocked`, `completed`, `cancelled`, `reopened` |
| Visit | `unplanned`, `scheduled`, `en_route`, `on_site`, `completed`, `cancelled` |
| Assignment | `proposed`, `assigned`, `accepted`, `rejected`, `released`, `cancelled` |
| Work Item | `ready`, `claimed`, `completed`, `returned`, `cancelled`, `expired` |
| Schedule Entry | `tentative`, `confirmed`, `completed`, `cancelled` |

Process state and business state are related but not interchangeable. A projection MAY mirror another aggregate's state for display; it MUST be read-only and MUST identify its source.

### AD-02 — Commands Own Business Change

Generic record CRUD MAY maintain drafts and non-governed descriptive fields. It MUST NOT directly change governed lifecycle, pricing totals, accepted snapshots, assignments, schedules, approvals, completion, cancellation, or reopening.

UI, API, automation, MCP, and future agents MUST call the same command catalog.

Every command MUST provide:

```text
command_id          globally unique idempotency key
workspace_id        tenant boundary
aggregate_type/id   target
expected_version    optimistic concurrency token
actor               authenticated principal
input                schema-validated command data
occurred_at          server timestamp
```

Every successful command MUST atomically write the authoritative state, aggregate version, domain event, audit record, and any immediately required work item. External or cross-boundary effects MUST use an outbox with retry status.

### AD-03 — Workflow Orchestrates; Domain Modules Enforce Invariants

Workflow chooses which governed command may occur next and creates human/system work. It MUST NOT bypass domain invariants with generic `set_field` or unrestricted `update_record` actions.

Official modules own rules such as quote total calculation, revision locking, one accepted quote version, schedule conflict policy, required evidence, and legal transitions.

### AD-04 — Approval Is A Work-Item Type, Not A Domain Object

Approval is a reusable human-decision step inside Workflow. It is not `quote_approval`, `work_order_approval`, or one approval table/module per business object.

An approval request is represented by:

```text
versioned approval step definition
  -> runtime work_item(kind = approval)
  -> one immutable approval_decision
  -> governed domain command/state change
```

The domain aggregate retains its meaningful business state. The approval runtime retains who was asked, who decided, what they decided, against which immutable input snapshot, and when.

### AD-05 — One Forms Product, Typed Uses

`runory.form` becomes the shared Forms capability. A form definition is channel-neutral. Workflow steps, public endpoints, record actions, Marketing capture, inspections/checklists, evidence capture, and service deliverables consume versioned forms through typed bindings.

Checklist is a repeatable/structured Forms block with execution rules, not another form engine. “Inspection”, “site evidence”, and “service report” are configurable business uses and labels, not three hard-coded form runtimes.

### AD-06 — Dispatch Uses Shared Assignment And Scheduling

Dispatch is a user action composed from reusable assignment and scheduling runtimes. FSM supplies Visit, Technician/Resource, Site, skill, territory, and policy context. It MUST NOT be implemented as a special `dispatch()` mutation that only understands Work Orders.

### AD-07 — One Planning Source, Multiple Views

Calendar, resource timeline, and map are views over the same `schedule_entry` query and invoke the same commands. None owns schedule truth. Map availability MUST NOT block calendar/timeline use when an address lacks coordinates.

### AD-08 — Typed Quote On A Reusable Commercial Kernel

Quote remains a typed product object. Reusable internals cover lines, price calculation, versions, immutable snapshots, and conversion provenance. v0.5 MUST NOT replace Quote with a universal untyped document.

### AD-09 — One Product, Desktop And Mobile Shells

The lightweight PWA is a focused shell inside `apps/cloud`, not a second business application. Desktop and mobile MUST share same-origin Session, command/API contracts, permissions, Forms definitions/submissions, records, and audit history. Mobile-specific code may own presentation, installation, browser-capability adapters, and service-worker behavior; it MUST NOT own independent business state or rules.

v0.5.1 provides installability, fast shell loading, safe weak-network recovery, and phone-first field execution. Full offline business-data synchronization remains deferred.

## 5. Target Runtime Model

All runtime tables are workspace-scoped and use Runory Runtime namespace prefixes. All mutable aggregate tables MUST include `version INTEGER NOT NULL DEFAULT 1`, `created_at`, and `updated_at`. IDs are server generated.

### 5.1 Workflow Definitions

#### `workflow_definitions`

Stable workflow identity and workspace ownership.

```text
id, workspace_id, workflow_key, name, target_object,
active_version_id, status(draft|active|retired), created_at, updated_at
UNIQUE(workspace_id, workflow_key)
```

#### `workflow_definition_versions`

Immutable after activation.

```text
id, workspace_id, workflow_definition_id, version_number,
definition_json, schema_version, published_by, published_at, created_at
UNIQUE(workflow_definition_id, version_number)
```

`definition_json` contains stable step IDs. Supported P0 step kinds:

```text
start
human_task
approval
system_command
wait
end
```

P0 control flow supports forward transitions, conditional branches, return to an explicitly allowed prior human step, cancellation, and reopening through an explicit command. Parallel branches, arbitrary subflows, and arbitrary scripting are not P0.

#### `workflow_instances`

```text
id, workspace_id, workflow_definition_id, definition_version_id,
object_type, record_id, status(running|completed|cancelled|failed),
current_step_id, version, started_by, started_at, completed_at,
created_at, updated_at
```

Constraint: one active instance per `(workspace_id, workflow_definition_id, object_type, record_id)` unless the definition explicitly declares multi-instance behavior in a future schema version.

#### `workflow_events`

Append-only process history.

```text
id, workspace_id, instance_id, sequence,
event_type, step_id, actor_type, actor_id,
payload_json, occurred_at
UNIQUE(instance_id, sequence)
```

Events MUST NOT be updated or deleted through product APIs.

### 5.2 Work Items And Approval

#### `work_items`

```text
id, workspace_id, instance_id, step_id,
kind(human_task|approval|form), status,
subject_type, subject_id,
assignee_type(user|permission_group), assignee_id,
candidate_rule_json,
due_at, claimed_by, claimed_at, completed_at,
form_binding_id, input_snapshot_json, input_snapshot_hash,
version, created_at, updated_at
```

Required indexes:

```text
(workspace_id, status, assignee_type, assignee_id, due_at)
(workspace_id, subject_type, subject_id)
(workspace_id, instance_id, step_id)
```

Candidate rules MUST resolve to concrete users when the item is created. Workspace roles (`admin/member/viewer`) are not approval policy. Business permission groups such as Sales Manager or Service Supervisor are.

#### `approval_decisions`

```text
id, workspace_id, work_item_id,
outcome(approved|rejected|returned|cancelled),
decided_by, comment, decision_payload_json,
input_snapshot_hash, decided_at, created_at
UNIQUE(work_item_id)
```

P0 supports exactly one terminal decision per approval work item. Reconsideration or resubmission creates a new work item; it never edits a prior decision. Multi-approver ANY/ALL/QUORUM strategies are deferred until a concrete v0.5 pilot requires them. Sequential review is modeled as multiple approval steps.

Approval invariants:

- requester and approver MUST be identifiable users;
- self-approval MUST default to denied and MAY be enabled only by explicit step policy;
- the actor MUST be the assignee or a currently resolved candidate with `workflow.approval.decide`;
- decision MUST carry `expected_version` and fail on stale work;
- decision input MUST be tied to an immutable hash/snapshot;
- deciding a cancelled, completed, returned, or expired item MUST return `409`;
- approving MUST invoke a named domain command, not set a field directly;
- rejection/return comment requirements are configurable and enforced server-side.

### 5.3 Command Runtime

#### `command_executions`

```text
id, workspace_id, command_id, command_type,
aggregate_type, aggregate_id, actor_type, actor_id,
input_hash, status(started|succeeded|failed),
result_json, error_code, created_at, completed_at
UNIQUE(workspace_id, command_id)
```

Reusing a `command_id` with the same input returns the stored result. Reusing it with a different input returns `409 IDEMPOTENCY_KEY_REUSED`.

#### `domain_events` and `outbox_messages`

Domain events are append-only. Outbox delivery is at least once; consumers MUST be idempotent. A command that commits business state and an outbox message is successful even if delivery is pending. Diagnostics MUST expose pending/failed outbox messages.

#### `workflow_timers`

```text
id, workspace_id, instance_id, work_item_id, timer_type,
due_at, status(scheduled|fired|cancelled),
payload_json, fired_at, created_at, updated_at
```

P0 supports a step due time, an overdue event, and one configured escalation target. Timer firing MUST be idempotent and MUST create a workflow event/outbox message rather than directly mutating a business aggregate. A reusable SLA-policy designer and multi-level escalation UI remain deferred.

### 5.4 Forms 2.0

#### `form_definitions`

```text
id, workspace_id, form_key, name, status(draft|active|retired),
active_version_id, created_at, updated_at
UNIQUE(workspace_id, form_key)
```

#### `form_definition_versions`

```text
id, workspace_id, form_definition_id, version_number,
schema_json, layout_json, published_by, published_at, created_at
```

Supported P0 blocks include field, section, checklist, evidence attachment, signature, and repeatable group. Validation is server authoritative.

#### `form_bindings`

```text
id, workspace_id, form_definition_id,
usage_type(workflow_step|record_action|public_endpoint|marketing_capture|service_deliverable),
usage_key, label_override, timing_json, requirement_policy_json,
target_mapping_json, active, created_at, updated_at
```

The binding determines name, trigger, required evidence, submitter, target mapping, and whether submission advances a process. The same form version MAY be reused by multiple bindings.

#### `form_submissions`

```text
id, workspace_id, form_definition_id, form_version_id, binding_id,
subject_type, subject_id, work_item_id,
revision_number, status(draft|submitted|accepted|returned|void),
answers_json, submitted_by, submitted_at,
supersedes_submission_id, created_at, updated_at
```

Submitted answers are immutable. A return/resubmit creates the next revision. Files are referenced through attachment records with content metadata and integrity hashes; raw binary data MUST NOT live in `answers_json`.

#### `attachments`

```text
id, workspace_id, owner_type, owner_id,
storage_key, file_name, content_type, byte_size, sha256,
uploaded_by, created_at, deleted_at
```

Download authorization is evaluated from the owning submission/record on every request. Deletion is soft and retention-policy controlled.

### 5.5 Assignment, Resources, And Scheduling

#### `resources`

Canonical schedulable identity.

```text
id, workspace_id, resource_type(user|crew|equipment),
user_id, display_name, timezone, active, metadata_json,
created_at, updated_at
```

`technician` becomes an FSM profile linked one-to-one to a user-backed resource for P0. Free-text assignees are prohibited for governed dispatch.

#### `assignments`

```text
id, workspace_id, subject_type, subject_id,
resource_id, role_key, status,
proposed_by, accepted_by, rejection_reason,
effective_from, effective_to, version, created_at, updated_at
```

Only one current primary assignment per subject/role is allowed. Reassignment closes/releases the old assignment and creates a new one in one command.

#### `schedule_entries`

```text
id, workspace_id, subject_type, subject_id,
resource_id, start_at, end_at, timezone,
status, location_type, location_id,
latitude, longitude, conflict_state,
version, created_at, updated_at
```

P0 conflict policy detects overlapping confirmed entries for the same resource. Workspace policy determines whether conflict is blocking or warning; overrides require `schedule.conflict.override` and a reason.

All time is stored as UTC instants plus an IANA timezone. `end_at` MUST be after `start_at`.

### 5.6 Quote And FSM Aggregates

Quote MUST add stable provenance and snapshot fields:

```text
quote_number, root_quote_id, previous_version_id, revision_number,
status, aggregate_version, price_book_id, currency,
subtotal, discount_total, tax_total, grand_total,
approved_at, accepted_at, rejected_reason, withdrawn_at,
snapshot_hash, locked_at
```

`price_book_item` is P0. Quote totals are calculated server-side from quote lines and price/tax/discount policy. Approved, rejected, sent, accepted, declined, expired, and withdrawn revisions are immutable; editing creates a revision. Only one accepted revision per `root_quote_id` is allowed.

Quote state meanings are fixed:

```text
draft       editable commercial revision
in_review   immutable snapshot waiting for internal approval
approved    internally approved, not yet sent
rejected    internally rejected by an approver; new work requires a revision
sent        delivered to the customer
accepted    accepted by the customer
declined    declined by the customer (not an internal approval rejection)
expired     validity elapsed
withdrawn   withdrawn by the seller
```

`return_for_changes` moves `in_review -> draft` and invalidates the submitted snapshot; `reject` moves `in_review -> rejected`. The two operations MUST NOT be aliases.

Work Order MUST add:

```text
work_order_number, aggregate_version,
source_type, source_id, source_snapshot_hash,
status, owner_resource_id, priority,
requested_at, sla_due_at,
completed_at, cancelled_at, reopened_at,
completion_reason, cancellation_reason, reopen_reason
```

Visit remains the schedulable FSM execution aggregate. Its assignment and time projections are read-only reflections of the current `assignment` and `schedule_entry`. Multiple visits per Work Order are supported.

Service report remains an optional typed FSM deliverable/projection. It MUST be created from an accepted `form_submission` binding and MUST NOT own a second editable review lifecycle.

## 6. Required Command Catalog

Command names are stable API identifiers. Implementations MAY be split internally, but callers MUST NOT coordinate multi-record writes themselves.

### 6.1 Quote

```text
quote.create_draft
quote.recalculate
quote.submit_for_approval
quote.approve
quote.reject
quote.return_for_changes
quote.withdraw
quote.create_revision
quote.mark_sent
quote.accept
quote.mark_declined
quote.expire
quote.convert_to_work_order
```

`quote.submit_for_approval` validates totals and completeness, pins the quote snapshot hash, moves Quote to `in_review`, starts/picks the configured workflow version, and creates an approval work item atomically.

`quote.approve` is callable only by the approval runtime with a valid undecided work item. It verifies the snapshot hash and moves Quote to `approved`. If the Quote changed, it fails with `409 SUBJECT_SNAPSHOT_CHANGED`.

`quote.convert_to_work_order` is idempotent by accepted quote revision. Repeated calls return the same Work Order.

```text
work_order.reopen
```

## 7. HTTP API Contract

Base path:

```text
/api/workspaces/{workspaceId}
```

### 7.1 Command Endpoint

```http
POST /commands/{commandType}
Idempotency-Key: cmd_...
If-Match: "{aggregateVersion}"
Content-Type: application/json
```

```json
{
  "aggregateType": "quote",
  "aggregateId": "quote_123",
  "input": { "comment": "Margin verified" }
}
```

Success returns the aggregate, emitted event IDs, created work-item IDs, and new version. The legacy dynamic-record update endpoint MUST reject governed field writes with `409 GOVERNED_FIELD_REQUIRES_COMMAND` and include the allowed command identifiers.

### 7.2 My Work


```json

  "commandId": "cmd_...",
  "expectedVersion": 3,
  "outcome": "approved",
  "comment": "Approved within delegated limit"
}
```

The decision endpoint returns the completed item, immutable decision, subject projection, and newly created next work item when applicable.

## 8. Authorization Contract

Workspace roles continue to control tenancy administration. Business permissions control work execution.

Minimum permission vocabulary:

```text
quote.read / quote.create / quote.edit_draft
quote.submit / quote.approve / quote.reject / quote.accept / quote.convert
work_order.read / work_order.triage / work_order.complete / work_order.reopen
visit.execute
assignment.manage / assignment.respond
schedule.manage / schedule.conflict.override
form.manage / form.submit / form.review
workflow.manage / workflow.approval.decide
```

P0 template groups:

```text
Sales Representative
Sales Manager
Dispatcher
Field Technician
Service Supervisor
Workspace Administrator
```

An actor MUST satisfy both tenant membership and the business permission. `admin` MUST NOT automatically mean “assigned approver” for a specific work item. Emergency administrative override, if enabled, requires a distinct permission, mandatory reason, and audit event.

## 9. Product And Route Rules

### 9.1 Approval Presentation

There is no default `/quote-approvals` page and no default top-level `/approvals` module.

Approval appears in exactly three contexts:

1. My Work filtered to approval tasks for the current actor;
2. the subject record's current action card;
3. the subject record's immutable process history.

Managers MAY receive a cross-team approval queue under My Work when their permission scope allows it. That queue is a work-item query, not a business object list.

### 9.2 Workflow Presentation

Operators see stage, current required input, available business action, assignee, due time, and history. They do not need to understand workflow instance IDs, node types, bindings, or runtime modules.

Process designers use Management > Processes. Publishing creates an immutable definition version and shows affected active bindings before activation.

### 9.3 Navigation Policy

Module manifests MUST declare presentation intent separately from object persistence:

```yaml
presentation:
  visibility: top_level | contextual | management | hidden
  surface: quotes | work_orders | planning | forms | processes | my_work
  audience: [sales, dispatcher]
```

Installer behavior MUST default undeclared technical child objects to `contextual`; it MUST NOT automatically create top-level navigation for every object.

## 10. Quote Approval Correction

The current `runory.quote-approval` module is deprecated in v0.5 and MUST NOT be installed into new workspaces.

Required catalog/UI changes:

- remove `runory.quote-approval` from `sales-quote-pack` dependencies;
- remove `quote_approval` terminology, navigation, widgets, role entry, demo data, and tests;
- remove `/quote-approvals` from navigation label mappings and canonical route expectations;
- replace duplicate widgets with a My Work approval query scoped to Quote;
- retain the `quote` object's meaningful lifecycle but make governed states command-only;
- update the Sales Quote plan to mark the earlier “simple approval object” choice as retired design debt.

Compatibility route for one release:

```text
/w/{workspaceId}/quote-approvals
  -> /w/{workspaceId}/my-work?kind=approval&subjectType=quote
```

The redirect MUST NOT render or permit CRUD against `quote_approval`.

## 11. v0.4 To v0.5 Migration

Migration is workspace-scoped, restartable, dry-runnable, and produces a report. It MUST preserve source rows until verification completes.

### Phase 1 — Inventory

For each installed workspace:

- count Quotes by status and Quote Approval rows by status;
- detect Quotes with zero, one, or multiple approval rows;
- compare Quote and Quote Approval states;
- detect missing requester/reviewer identities;
- record active workflow instances bound to Quote;
- classify every row as migratable, historical-only, or conflict.

### Phase 2 — Install Runtime V2

- create versioned workflow, work-item, decision, command, event, outbox, Forms 2.0, assignment, and schedule tables;
- publish the default Quote process version;
- migrate active v1 workflow definitions to immutable versions;
- pin running instances to their captured definition.

### Phase 3 — Transform Quote Approval

Rules:

| Legacy condition | Migration result |
|---|---|
| Quote `pending_approval` + approval `pending` | map Quote to v0.5 `in_review`; create active instance and ready approval work item |
| Quote `approved` + approval `approved` | create completed historical work item and immutable approved decision |
| Quote rejected + approval rejected | create historical rejected decision |
| Approval cancelled/withdrawn | create cancelled historical work item/event |
| statuses disagree | do not guess; emit `APPROVAL_STATE_CONFLICT` for manual resolution |
| multiple terminal decisions | preserve as imported audit events and flag conflict; do not synthesize one current decision |
| missing user identity | map to an explicit imported-legacy actor and flag identity remediation |

Migration MUST NOT silently overwrite Quote status from the Approval row or vice versa.

### Phase 4 — Cutover

- enable command-only guards for governed fields;
- replace navigation/widgets/routes;
- mark `runory.quote-approval` installation retired while retaining its table read-only;
- run count/hash/relation verification;
- after one compatibility release, archive the legacy table according to retention policy.

Rollback before cutover restores v0.4 reads. Rollback after new decisions have been written is forward-fix only; immutable v0.5 decisions MUST NOT be projected back into editable legacy approval rows.

## 12. Implementation Slices

Implementation order is dependency order, not parallel feature choice.

### 12.0 Repository Ownership Map

The implementation MUST keep these responsibilities stable:

| Repository area | v0.5 ownership |
|---|---|
| `packages/contracts` | public schemas, command envelopes/results, stable error codes, Workflow/Form/Planning DTOs |

| `apps/cloud` API routes | authentication, tenant context, contract parsing, command dispatch, HTTP mapping; no domain rules |
| `apps/cloud` UI | My Work, contextual actions, Planning and management surfaces; no direct governed-field coordination |
| `schema/migrations` | runtime tables and indexes; migrations are additive before cutover |

Domain rules MUST NOT be implemented only in React components or route handlers. Catalog YAML MAY declare policy and bindings, but executable invariants MUST live in tested server code. Shared runtime packages MUST NOT import FSM or Quote catalog modules.

## 13. Verification And Release Gates

### 13.1 Architecture Tests

- no active pack declares `quote_approval` or `/quote-approvals`;
- no aggregate has two independently writable lifecycle fields;
- every governed field maps to at least one command and rejects generic updates;
- every workflow instance references an immutable definition version;
- every approval decision references exactly one work item and is immutable;
- every top-level navigation item declares role/job/presentation intent;
- all Planning views use the same schedule query contract.

### 13.2 Contract And Concurrency Tests

- command retry with same key/input returns the same result;
- same key/different input returns `IDEMPOTENCY_KEY_REUSED`;
- two actors deciding the same item produce one success and one `VERSION_CONFLICT`/`WORK_ITEM_NOT_ACTIONABLE`;
- changing the Quote after submission invalidates approval through snapshot mismatch;
- unauthorized, unassigned, or self-approving actors are rejected;
- state, event, audit, and immediate next work item commit together;
- failed asynchronous side effects remain visible and retryable in outbox diagnostics.

### 13.3 Product Journey Tests

At minimum, automated browser/API coverage MUST prove:

1. create customer/opportunity and Quote;
2. calculate and submit Quote;
3. manager sees it in My Work, returns it, and salesperson resubmits;
4. manager approves, customer accepts, and conversion creates one Work Order;
5. dispatcher creates Visit, assigns resource, and schedules it from Planning;
6. technician completes configured checklist/evidence/signature;
7. supervisor returns the deliverable and technician resubmits;
8. accepted work completes Visit and Work Order;
9. authorized reopen preserves all prior submissions and decisions;
10. audit history identifies every actor, command, state change, and version.

### 13.4 Migration Tests

Fixtures MUST cover all mapping rows in Section 11, including multiple approvals, status disagreement, missing actors, interrupted migration/restart, and already migrated workspaces. Record counts and conflict reports are release artifacts.

### 13.5 Operational SLOs

For production-like data volumes agreed at scope lock:

```text
command API p95              <= 500 ms excluding file upload/external delivery
My Work query p95            <= 500 ms
Planning 7-day query p95     <= 1 s
duplicate command effects    = 0
lost workflow/audit events   = 0
silent migration conflicts   = 0
```

The exact load profile MUST be committed with the performance test; a latency number without record/resource/concurrency counts is not a release claim.

## 14. Drift-Control Checklist For Pull Requests

Every v0.5 PR must answer:

```text
Which aggregate owns the changed truth?
Which command changes it?
What is the expected_version/idempotency behavior?
Which domain event and audit event result?
Is any state duplicated as independently editable data?
Does this persistence object really require navigation?
Does the UI call the same contract as API/automation/agent callers?
What happens on return, cancellation, retry, stale input, and partial failure?
Which release-gate test proves the behavior?
```

PRs introducing a new top-level module/page, generic status mutation, per-domain approval object, second form runtime, view-owned schedule state, or best-effort business synchronization MUST be rejected unless this specification is deliberately amended first.

## 15. Final Boundary Summary

```text
CRM / Quote / FSM objects
  own business meaning and invariants

Workflow
  owns orchestration, versioned process, and pending work

Approval
  is an assigned Workflow work-item type with an immutable decision

Forms
  owns versioned input definitions and immutable submissions

Assignment + Scheduling
  own resource commitment and time placement

Planning
  is a product surface over Assignment + Scheduling

My Work
  is a role-scoped projection of actionable work items

Packs / Modules
  own installation, dependency, upgrade, and technical boundaries;
  they do not dictate product navigation
```

This boundary is the v0.5 implementation contract. In particular, `Quote Approvals` as a separately editable business module is retired: Quote owns the commercial state, Workflow creates the approval work, and the immutable decision proves how that state changed.
