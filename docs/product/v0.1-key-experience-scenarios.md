## 1. Purpose

This document defines the product experience that Runory Cloud 0.1 must make tangible.

The goal is not to list every implemented feature. The goal is to identify the smallest set of trial scenarios that prove Runory has moved beyond a POC:

```text
Cloud SaaS Core
+ composable Module / Pack / Template runtime
+ governed Agent extension
+ Catalog and release control plane
+ upgrade-safe business data
+ production-shaped UI and operations foundation
```

If these scenarios feel coherent in use, 0.1 can be understood as an Early Access MVP. If they fail, the system may still contain useful pieces, but it should not be presented as a complete 0.1 product baseline.

## 2. Experience Principles

### 2.1 Show a real product, not only infrastructure

SaaS Core, Catalog, SDK, migrations, audit, and RBAC are necessary, but an early user should experience them through a useful business workspace.

The reference business experience for 0.1 is CRM Lite:

```text
Small business user
→ starts from email
→ enters a Workspace
→ installs CRM Lite
→ manages Customers and Contacts
→ asks Runory to adapt the workspace
→ sees the change safely applied
```

### 2.2 Prove the platform loop, not only the CRM demo

Runory is not just a CRM. The platform team must also be able to demonstrate:

```text
Module source
→ SDK validation/build
→ immutable Catalog artifact
→ validation
→ release channel
→ workspace compatibility
→ safe upgrade
```

This is the key difference between "we built a sample app" and "we built the first version of a business capability platform."

### 2.3 Keep the 0.1 surface small

0.1 should avoid making every future capability visible. The trial should focus on:

- one polished public entry;
- one clear onboarding flow;
- one reference business pack;
- one governed extension scenario;
- one platform manufacturing and upgrade scenario;
- one recovery/security confidence scenario.

Team, advanced workflow design, public marketplace, third-party publishing, paid billing, SAML/OIDC, custom roles, and arbitrary code editing remain out of scope for the 0.1 trial.

## 3. Primary Trial Personas

### Evaluator

A potential user or partner who wants to understand what Runory is.

They should experience the website, open-source positioning, pricing, login, and first workspace without needing a sales call.

### SMB Workspace Admin

The first real 0.1 user.

They should be able to create a workspace, install CRM Lite, enter business data, and approve a safe customization.

### Platform Operator

An internal Runory operator or release manager.

They should be able to inspect Catalog items, versions, releases, validation evidence, compatibility, rollout status, and audit events.

### Module Developer

An internal developer building official modules.

They should use the SDK/toolchain to validate and build canonical module artifacts, then hand them to the Catalog control plane.

## 4. Canonical 0.1 Experience Map

### Scenario 0 — Public Product Evaluation

User story:

```text
Visitor opens Runory Cloud
→ understands Cloud-first + Open Source positioning
→ checks Pricing
→ sees that English is default and Chinese is available
→ starts login
```

What it proves:

- Runory has a product-grade entry, not only an internal app shell.
- Cloud paid / open-source positioning is clear enough for early users.
- Internationalization is present from day one.

Current product surface:

- Landing page
- Pricing page
- Open Source page
- Language switcher
- Login page

0.1 acceptance:

- Desktop and mobile render without blocking layout issues.
- English is the default locale.
- Chinese can be selected and persists.
- Open-source page does not overclaim licensing before a license is chosen.

### Scenario A — New Customer Onboarding

User story:

```text
User enters email
→ receives/verifies OTP
→ Organization and default Workspace are created
→ user enters Workspace
→ installs CRM Lite Pack
→ creates Customer and Contact
→ invites or prepares to invite a member
```

What it proves:

- Email-only authentication is enough for Early Access.
- SaaS Core can safely create user, organization, workspace, memberships, session, and default entitlement.
- The business product becomes useful quickly.

Current evidence path:

- Script: `apps/cloud/scripts/e2e-scenarios/scenario-a-onboarding.mjs`
- Release scenario: `Scenario A — New Customer Onboarding`

0.1 acceptance:

- The flow works in a production-like environment without dev identity headers.
- The first workspace has correct tenant and role boundaries.
- CRM Lite installation produces visible Customer and Contact objects.
- Manual invitation can be accepted as partial only if the invitation API and UI path are separately verified.

### Scenario B — Governed Workspace Extension

User story:

```text
Workspace Admin asks to add "Customer Tier"
→ Agent proposes an Extension Plan
→ user sees Diff Preview
→ user approves Apply
→ Customer UI shows the new field
→ Audit records the operation
→ rollback point can remove the extension
```

What it proves:

- Runory customization is not arbitrary code generation.
- Agent operations go through governed APIs.
- Workspace-specific changes do not mutate official module source.
- Extension ownership and rollback are visible.

Current evidence path:

- Script: `apps/cloud/scripts/e2e-scenarios/scenario-b-extension.mjs`
- APIs: `/agent/plan`, `/agent/preview`, `/agent/apply`, `/agent/rollback`

0.1 acceptance:

- Diff is understandable to a non-developer admin.
- Apply and rollback both produce audit events.
- Module-owned fields and workspace-extension fields remain distinguishable.
- The UI reflects schema changes without a code deploy.

Current v0.1 trial path:

```text
Workspace
→ Settings
→ Safe customization approval
→ 1. Validate Plan
→ 2. Preview Diff
→ 3. Approve & Apply
→ Success state links to Customers and Audit
→ Customers page shows the new workspace-extension field
→ Customer detail edit/save persists a value for the new field
→ Audit / extension history shows version, risk level, and change summary
→ Rollback removes the extension field
```

Post-apply verification checklist:

- Settings shows the extension as installed with a current version.
- Customers shows a visible workspace-extension field indicator.
- Opening a demo Customer record allows editing the new field through the schema-driven form.
- Audit shows `extension.apply` with version, risk level, and change summary.
- Rollback removes the field from Customer list/form and records an `extension.rollback` event.

Important boundary:

- 0.1 must not require the evaluator to install MCP, Codex plugin, or a local Skill just to experience Admin approval.
- Codex / MCP / Skill are advanced proposal generators. They should call the same governed APIs as the Cloud UI.
- The Cloud UI remains the canonical approval surface for Workspace Admin: Plan, Diff, Approve, Apply, Audit, Rollback.
- A future built-in Agent can replace the JSON proposal editor with a conversational proposal flow, but the approval contract should stay the same.

### Scenario C — Module Manufacturing and Upgrade

User story:

```text
Runory developer updates runory.customer
→ SDK validates/tests/builds the module
→ Catalog imports an immutable artifact
→ platform validation runs
→ Release Manager promotes through internal/beta/stable
→ Workspace compatibility report is generated
→ Workspace Admin approves upgrade
→ existing data and extensions remain usable
```

What it proves:

- Module / Pack / Template are platform-managed business assets.
- Catalog is a control plane, not a static file list.
- Upgrades are intentional and compatibility-aware.

Current evidence path:

- Script: `apps/cloud/scripts/e2e-scenarios/scenario-c-module-upgrade.mjs`
- SDK docs: `docs/sdk/module-sdk.md`
- Catalog docs: `docs/09-catalog-release-control-plane.md`

0.1 acceptance:

- SDK validation/build output is machine-readable.
- Catalog versions are immutable after ready/freeze.
- Stable promotion requires a human release manager.
- Workspace compatibility is visible before upgrade.
- Customer 1.0 to 1.1 upgrade preserves existing records and workspace extensions.

### Scenario D — Bad Release Containment

User story:

```text
Bad beta release fails on one target
→ target is marked failed
→ rollout pauses or stops expanding
→ unrelated Workspace remains available
→ operator can inspect evidence
→ fixed patch version is published instead of mutating the bad artifact
```

What it proves:

- Release failure is isolated.
- Immutable artifacts and rollout state protect existing customers.
- Operators have enough evidence to make a safe release decision.

Current evidence path:

- Script: `apps/cloud/scripts/e2e-scenarios/scenario-d-bad-release.mjs`
- Rollout APIs: `/api/platform/rollouts/:id/pause`, resume, cancel

0.1 acceptance:

- A failed target does not corrupt other workspaces.
- Error responses do not leak stack traces or SQL internals.
- Rollout pause/resume/cancel behavior is auditable.
- Fixes are new versions, not overwritten artifacts.

### Scenario E — Recovery and Trust

User story:

```text
Operator restores database backup
→ migrations replay or verify cleanly
→ tenant isolation tests still pass
→ CRM journey still works
→ Catalog installation state remains consistent
```

What it proves:

- Runory can recover from operational failure.
- Multi-tenant boundaries survive backup/restore.
- Catalog and runtime state remain linked after recovery.

Current evidence path:

- Script: `apps/cloud/scripts/e2e-scenarios/scenario-e-recovery.mjs`
- Backup runbook: `docs/operations/backup-restore-runbook.md`

0.1 acceptance:

- A real backup restore drill is completed.
- Post-restore health, tenant isolation, CRM CRUD, and installation integrity pass.
- Export/backup artifacts do not include auth/session/API key secrets.

## 5. Supporting Experience Scenarios

### Workspace Daily Use

This is the "does the product feel alive" layer:

- Dashboard explains current workspace state.
- Navigation exposes Customers, Contacts, Tasks, Modules, Workflows, Audit, Settings, and Billing without overwhelming the user.
- Empty states guide the next useful action.
- Loading and error states are calm and recoverable.

0.1 does not need deep functionality in every tab, but every visible tab should either be useful or clearly marked as early/coming soon.

### Module Center

Workspace admins should understand:

- what is installed;
- what can be installed;
- which version is active;
- whether an upgrade is available;
- whether compatibility is safe.

The Module Center is the user-facing half of Catalog. The platform Catalog Console is the operator-facing half.

### Platform Catalog Console

Platform operators should see enough detail to make release decisions. A production-shaped operator view should include:

- Catalog Item identity: module / pack / template, stable name, publisher, visibility, status.
- Version lifecycle: draft, validating, ready, deprecated, withdrawn, frozen timestamp.
- Artifact evidence: checksum, source repository, source commit, build ID, provenance.
- Module business shape: objects, fields, field types, required flags, views, permissions, extension points.
- Pack composition: module dependency refs and frozen lock.
- Template entry: default packs, navigation, terminology, homepage/layout intent.
- Validation evidence: manifest schema, SemVer, dependency resolution, permissions, migration files, key collisions, extension point checks.
- Release state: internal / beta / stable, approver, release notes, status.
- Rollout state: targets, success/failure progress, pause/resume/cancel history.
- Workspace impact: installations, compatibility reports, extension conflicts.
- Audit trail: import, validate, freeze, promote, deprecate, withdraw, rollout actions.

For 0.1, the Console can be compact, but it must let an operator answer:

```text
What is this artifact?
What business schema will it install?
What changed in this version?
Is it validated?
Who approved release?
Which workspaces are affected?
Can we stop or rollback rollout safely?
```

### Audit and Accountability

The trial should make audit visible for:

- login/session-sensitive operations;
- pack install;
- extension apply/rollback;
- membership or invitation changes;
- release/control-plane operations.

Audit is not decorative in Runory. It is how users learn to trust Agent-assisted operations.

### Internationalized First Impression

For 0.1, English should be the default and Chinese should be available for public pages and key onboarding surfaces.

Authenticated product pages can complete localization incrementally, but any visible mixed-language state in the canonical 0.1 journey should be treated as a polish blocker.

## 6. Experience Readiness Matrix

| Scenario | Trial audience | Current automation | 0.1 readiness meaning |
| --- | --- | --- | --- |
| Public Product Evaluation | Evaluator | Browser/manual | Product positioning and first impression are credible |
| A. New Customer Onboarding | SMB Admin | E2E script + browser | Email to useful workspace works |
| B. Governed Extension | SMB Admin | E2E script + browser | Agent customization is governed and reversible |
| C. Manufacturing and Upgrade | Developer / Operator | Partial E2E + manual approval | Runory can ship business capability versions safely |
| D. Bad Release Containment | Operator | Partial E2E | Platform failure does not spread across workspaces |
| E. Recovery and Trust | Operator | Post-restore E2E + manual drill | Operational recovery preserves data and boundaries |

## 7. Recommended Demo Flow

### 30-minute product demo

```text
Landing page
→ Pricing / Open Source
→ Login with email OTP
→ Dashboard / default Workspace
→ Install CRM Lite
→ Create Customer and Contact
→ Ask Agent to add Customer Tier
→ Preview Diff
→ Apply
→ See updated UI and Audit
→ Rollback
```

This is the best demo for early SMB users and design partners.

### 45-minute platform demo

```text
Show Catalog item/version/release model
→ Run SDK validate/test/build for runory.customer
→ Import artifact
→ Validate
→ Generate compatibility report
→ Discuss beta/stable approval
→ Show bad release containment
→ Show backup/restore evidence
```

This is the best demo for technical partners, internal release review, and investor/product architecture review.

## 8. What 0.1 Should Not Try To Demonstrate

Avoid expanding the 0.1 trial into:

- full marketplace;
- third-party publisher onboarding;
- drag-and-drop app builder;
- arbitrary SQL or code editor;
- custom role/field ACL matrix;
- enterprise SSO;
- paid self-service checkout;
- broad ERP module coverage;
- production SLA claims;
- local/private deployment as the default path.

Those can be future proof in the architecture, but making them part of the 0.1 experience will blur the product and slow the release.

## 9. Documentation and Test Alignment

The release definition already contains the required end-to-end scenarios. This document turns them into product experience language.

Recommended alignment:

- Keep `docs/releases/v0.1.0-cloud-early-access.md` as the release gate and evidence index.
- Keep this document as the product trial map.
- Keep `apps/cloud/scripts/e2e-scenarios/*` mapped one-to-one to Scenarios A-E.
- Add browser-level evidence for Scenario 0 and the 30-minute product demo.
- Any new feature proposed for 0.1 should explicitly attach to one of these scenarios, or be deferred.
