## 1. Purpose And Release Decision

v0.5 delivered most of the required runtime primitives, but the current product does not yet expose them as one executable commercial FSM experience. v0.5.1 is the productization, command-kernel hardening, and Workflow V1 retirement release.

The release goal is:

> A non-technical team can enter one prepared workspace and execute Quote → optional approval → accepted commercial revision → Work Order → assignment → schedule → Visit → configured field form/evidence → optional review/return/resubmit → completion, using role-focused product surfaces and named commands. Workflow V2 is available as an optional orchestration overlay for approvals, returns, and cross-role obligations, but a business pack's core lifecycle MUST run without requiring workflow.

v0.5.1 is not accepted because tables, APIs, or isolated pages exist. It is accepted only when the canonical journey works through the supported UI and APIs, all governed changes use named commands, the visible data is understandable, and the same outcome is proven by automated acceptance tests.

Normative terms in this document use `MUST`, `MUST NOT`, `SHOULD`, and `MAY` in their usual specification sense. Any deviation from a `MUST` requires an explicit written waiver in the release evidence.

## 3. Scope Lock

### 3.1 P0 Deliverables

v0.5.1 MUST deliver all of the following:

1. Workflow V1 retirement, V2 migration for workflow-backed scenarios, and proof that business packs can execute their core lifecycle without workflow.
2. A deterministic Golden Demo Workspace containing CRM + Sales Quote + FSM and connected role identities.
3. One command-only lifecycle path for Quote, Work Order, Assignment, Schedule, Visit, and deliverable review.
4. Role-focused desktop navigation, My Work, contextual actions, and readable business data.
5. One Forms product with a visual designer, typed bindings, execution, evidence, review, return, and revision history.
6. Planning with backlog, calendar, resource timeline, basic map, conflict visibility, assign/reassign, schedule/reschedule, and unschedule.
7. Customer, service-site, asset, Quote, Work Order, and Visit operational history projection.
8. The lightweight mobile PWA and field execution requirements in the required mobile sub-specification.
9. Automated release gates for migration, canonical journeys, permissions, concurrency, data consistency, and presentation quality.

### 3.2 Explicit Non-Goals

The following remain out of scope unless separately approved:

```text
offline-first business-data synchronization
route optimization and automatic dispatch
live GPS tracking
inventory, purchasing, truck stock, and parts costing
invoice, payment, and accounting integrations
telephone, SMS, email, and customer-support integrations
customer portal and regulated remote e-signature
parallel workflow branches and arbitrary scripts
forcing every business pack or SMB workflow to use Workflow V2
general ABAC, territory, skills, and capacity optimization
```

Weak-network recovery, evidence upload retry, and an installable PWA remain P0; “not offline-first” MUST NOT be used to omit safe failure and retry behavior.

## 4. Locked Architecture Invariants

### 4.1 One Authoritative Owner Per State

| Concern | Authoritative owner | Projections that MUST NOT be independently editable |
|---|---|---|
| Quote lifecycle | Quote aggregate through Quote commands | workflow step, dashboard status, approval display |
| Work Order lifecycle | Work Order aggregate through FSM commands | workflow step and list/dashboard status |
| Visit lifecycle | Visit aggregate through Visit commands | form state and schedule display |
| Current assignee | active `assignment` row | `assigned_to`, technician display, My Work label |
| Scheduled time/resource | `schedule_entry` | Work Order/Visit scheduled fields and calendar blocks |
| Human obligation | `work_item`, generated by command or Workflow V2 | approval queues and role badges |
| Form response | immutable/versioned `form_submission` | service-report rendering and workflow completion display |
| Workflow execution, when enabled | V2 instance pinned to an immutable definition version | record status mirrors |

Synchronizing projections MUST occur inside the same command transaction or through a durable, idempotent projection event. UI code, generic CRUD, automation actions, MCP, and agents MUST NOT coordinate duplicate writes.

### 4.2 One Command Boundary

UI, API clients, automation, MCP, Skills, and future agents MUST call the same command catalog. Every governed command MUST provide:

```text
commandId / idempotency key
commandType
workspace and actor context
aggregate type and ID
expected aggregate/work-item version
schema-validated input
permission and assignment requirement
atomic state/event/audit/work-item result
stable error code and user-safe message
```

Generic record create/update APIs MUST reject governed field writes with `409 GOVERNED_FIELD_REQUIRES_COMMAND` and return the allowed command identifiers. The same restriction MUST apply during create when a supplied initial state would bypass an initialization command.

### 4.3 Commands Are The Business Execution Kernel

Business packs MUST declare and run their core lifecycle through named commands and governed state machines. Workflow MUST NOT be required for the basic execution of Quote, Work Order, Assignment, Schedule, Visit, form submission, deliverable acceptance, or completion.

The default SMB mode is command-only execution:

```text
object model + lifecycle state + named commands + permissions + audit/events + contextual UI
```

The workflow-backed mode is an optional overlay:

```text
command kernel + Workflow V2 definitions + Work Items + cross-role obligations + return/resubmit paths
```

### 4.4 Workflow Is Orchestration, Not A Second Business State

Workflow V2 may determine the next Work Item, candidate actor, due date, and allowed command. It MUST NOT use unrestricted `set_field`, generic `update_record`, or direct SQL to advance a governed lifecycle.

No production or demo test MAY use direct SQL to cross a business state transition. A missing command is a release-blocking product gap, not a test workaround.

### 4.5 Technical Identifiers Are Not Product Labels

`rec_*`, `wfi_*`, `wi_*`, raw UUIDs, field keys, binding IDs, permission-group keys, command keys, and raw enum values MUST NOT be the primary visible label on business surfaces. They MAY appear in an administrator diagnostics drawer.

## 5. Workflow V1 Retirement And Optional Workflow V2 Overlay

Workflow is not the business execution kernel. Section 5 applies to legacy workflow data and to workflow-backed scenarios. It MUST NOT be interpreted as requiring every pack, record, or SMB customer journey to start a Workflow V2 instance.

### 5.1 Required Inventory

Before mutation, the migration MUST inventory and persist counts for:

```text
V1 definitions and instances by target object/current state
V2 definitions, versions, instances, events, and work items
legacy Quote Approval rows and state combinations
records whose governed lifecycle disagrees with workflow state
missing actors, assignees, forms, targets, or records
duplicate active instances and terminal decisions
```

The migration MUST expose dry-run inventory through an admin API and UI. Dry run MUST perform no business-data writes.

### 5.2 Definition Migration

For each active V1 definition:

1. Convert it to the V2 definition schema with stable step IDs.
2. Validate every human step, command, return path, and assignee rule.
3. Reject definitions that depend on generic field mutation or an unknown command.
4. Publish one immutable V2 version.
5. Record a `legacy_definition_id → v2_definition_id/version_id` mapping.

Workflow files shipped by modules MUST use the `.workflow.json` suffix, pass the V2 contract validator, and be covered by an architecture test. A workflow file that exists but is not discovered is a release failure.

### 5.3 Instance Migration

For each V1 instance:

- terminal instances MUST become terminal V2 instances with imported historical events and no actionable work item;
- running instances MUST pin the migrated definition version, preserve the current logical step, append an `workflow.imported` event, and create exactly one valid current Work Item when the step is human;
- V1 history MUST be converted to ordered immutable events without inventing actors or decisions;
- missing actors MUST map to an explicit `imported_legacy` actor and create an identity-remediation warning;
- workflow/record state disagreement MUST produce a conflict and MUST NOT be silently resolved;
- duplicate active instances or terminal decisions MUST be flagged for manual resolution.

The migration MUST be restartable. Re-running the same migration MUST not duplicate definitions, versions, events, work items, decisions, or audit records.

### 5.4 Quote Approval Migration

The existing Quote Approval transformation remains required, but MUST follow the same conflict, identity, history, and idempotency rules as general Workflow migration. Cancelled/withdrawn approvals MUST become historical cancelled work items/events rather than active pending approvals.

### 5.5 Cutover

After a workspace passes verification:

```text
all new workflow-backed starts use V2
all workflow-backed reads in business UI use V2
V1 write/start/transition APIs return 410 WORKFLOW_V1_RETIRED
V1 tables remain read-only for one compatibility release
legacy Quote Approval navigation and CRUD remain retired
the migration status records cutover time, counts, hashes, conflicts, and actor
```

Rollback after new V2 decisions exist is forward-fix only. V2 decisions and events MUST NOT be written back into editable V1 rows.

### 5.6 Migration Verification

Cutover MUST fail unless:

```text
active V1 instances remaining            = 0
unmapped V1 definitions                  = 0
running V2 human steps without work item = 0
V2 instances without definition version = 0
silent state disagreements               = 0
duplicate imported decisions             = 0
orphan subject/actor relations            = 0, except explicitly accepted identity warnings
command-only lifecycle fixture passes without creating a workflow instance
```

The release artifact MUST include before/after counts, relation checks, conflict report, and an interrupted-run/restart result.

## 6. Canonical Command Catalog

The following named intentions MUST exist, enforce permission/assignment, write audit/events, and be reachable from the appropriate UI. Exact internal function names MAY differ; public `commandType` values are locked.

### 6.1 Quote

```text
quote.create_draft
quote.recalculate
quote.submit_for_approval
quote.return_for_changes
quote.approve
quote.reject
quote.withdraw
quote.create_revision
quote.mark_sent
quote.accept
quote.mark_declined
quote.expire
quote.convert_to_work_order
```

Approved, rejected, sent, accepted, declined, expired, and withdrawn revisions are immutable. Editing an immutable Quote MUST offer Create Revision instead of generic Edit. Conversion MUST create exactly one correctly linked Work Order for the accepted revision.

### 6.2 Work Order And Visit

```text
work_order.triage
work_order.plan
work_order.start
work_order.create_visit
work_order.block
work_order.unblock
work_order.complete
work_order.cancel
work_order.reopen

visit.start_travel
visit.arrive
visit.submit_work
visit.complete
visit.cancel
visit.reopen
```

Every documented lifecycle edge MUST have a named command. Tests MUST NOT update status through SQL or generic CRUD.

### 6.3 Assignment And Schedule

```text
assignment.propose
assignment.assign
assignment.accept
assignment.reject
assignment.reassign
assignment.release

schedule.plan
schedule.confirm
schedule.reschedule
schedule.unschedule
schedule.cancel
```

Reassignment MUST close the prior active assignment and create the new assignment atomically. Schedule conflict policy MUST be server-authoritative; an override requires a dedicated permission and mandatory reason.

### 6.4 Work Item, Form, And Review

```text
work_item.claim
work_item.complete
form_submission.save_draft
form_submission.submit
form_submission.return
form_submission.resubmit
form_submission.accept
```

A required form Work Item cannot complete without a valid submission of the pinned form version. Return preserves the previous submission and creates a new revision/work item.

Work Items MAY be command-generated or workflow-generated. A workflow-generated Work Item MUST still complete through a named command or a command-backed form submission path.

## 7. Golden Demo Workspace Specification

### 7.1 Deterministic Installation

`bootstrap:demo` MUST create or reset one canonical workspace through a documented, idempotent path and install:

```text
crm-lite-pack
sales-quote-pack
fsm-pack
```

The script MUST support `--validate-only` and MUST exit non-zero if any invariant in this section fails. It MUST not depend on pre-existing V1 data.

### 7.2 Personas

The demo MUST contain member-backed identities for:

| Persona | Required capabilities |
|---|---|
| Sales representative | create/revise/submit Quote |
| Sales manager | return/approve/reject Quote |
| Dispatcher | triage, assign, schedule, reassign, reschedule |
| Technician | see personal work, accept assignment, execute Visit/forms |
| Field supervisor | review, return, accept deliverable, complete/reopen work |

At least one local demo account MAY switch personas without creating a second authentication system. Persona switching MUST change the effective permission/assignment context and MUST be disabled outside explicit demo/development mode.

Every Technician MUST reference exactly one user-backed Resource. Every scheduled subject MUST reference an active Resource and current Assignment where policy requires one.

### 7.3 Data Scenarios

The demo MUST include at least:

1. one Quote awaiting manager review;
2. one returned Quote ready for revision;
3. one accepted Quote already converted to a Work Order;
4. one unscheduled Work Order in Planning backlog;
5. one assigned Visit scheduled today;
6. one schedule conflict example;
7. one Visit awaiting technician execution;
8. one submitted deliverable awaiting supervisor review;
9. one returned deliverable ready for resubmission;
10. one completed service history containing Quote, Work Order, Visit, form, evidence, and audit events;
11. one command-only Work Order/Visit path that has no active workflow instance but still completes through commands, form submission, audit, and timeline;
12. one failed Outbox message with a safe retry example.

Dates MUST be generated relative to the seed time and workspace timezone. The demo MUST not become empty, overdue, or contradictory merely because the calendar date changes.

### 7.4 Demo Data Integrity

The validator MUST prove:

```text
all relation aliases resolve
no displayed relation falls back to an ID
all price books contain price_book_item rows
Quote totals equal server calculations
accepted Quote revision is immutable and linked to one Work Order
all running V2 human steps own one Work Item
all personal Work Items resolve to a real persona
all Visit assignments and schedule entries are bidirectionally resolvable
record schedule projections match schedule_entry
all form submissions reference a pinned form version, subject, and revision
all dashboard and timeline subjects resolve to display labels
at least one command-only lifecycle path has no workflow instance and still passes all command/state/audit checks
```

The demo SHOULD use one coherent locale, currency, timezone, address region, terminology, and industry story per workspace. Cross-locale examples belong in separate fixtures.

## 8. API And Read-Model Contracts

### 8.1 My Work

```http
GET /api/workspaces/{workspaceId}/my-work?queue=mine|unassigned|team|overdue&kind=...&cursor=...
```

Each item MUST return presentation-ready context without client joins:

```json
{
  "id": "wi_...",
  "kind": "approval",
  "status": "ready",
  "title": "Approve Quote Q-2026-0042",
  "subject": { "type": "quote", "id": "rec_...", "label": "Q-2026-0042 · Acme HVAC Renewal" },
  "customer": { "id": "rec_...", "label": "Acme Facilities" },
  "site": { "id": "rec_...", "label": "Shanghai Plant A" },
  "assignee": { "type": "user", "id": "usr_...", "label": "Sales Manager" },
  "currentStageLabel": "Manager review",
  "nextActionLabel": "Approve or return",
  "dueAt": "...",
  "availableCommands": ["quote.approve", "quote.return_for_changes"],
  "version": 3
}
```

Raw binding IDs and permission-group keys MUST not be required to understand the item.

### 8.2 Planning

```http
GET  /api/workspaces/{workspaceId}/planning/entries?from=...&to=...&resourceIds=...&status=...
GET  /api/workspaces/{workspaceId}/planning/backlog?cursor=...
POST /api/workspaces/{workspaceId}/commands/assignment.assign
POST /api/workspaces/{workspaceId}/commands/assignment.reassign
POST /api/workspaces/{workspaceId}/commands/schedule.plan
POST /api/workspaces/{workspaceId}/commands/schedule.reschedule
POST /api/workspaces/{workspaceId}/commands/schedule.unschedule
```

Planning entries MUST return subject label, resource label, customer/site, address, coordinates when present, timezone, conflict state, version, and available commands. Calendar, timeline, map, record detail, and mobile schedule MUST consume the same query contract.

### 8.3 Presentation Metadata

The metadata API MUST provide:

```text
localized object/field/section labels
localized enum labels and semantic colors
relation display-field and route metadata
date/time/currency/unit formatting policy
governed/read-only field reason
available contextual commands
```

Dashboard widgets, list pages, detail pages, forms, timelines, and mobile MUST use the same presentation layer. A widget MUST not render a relation by calling `String(record[field])`.

### 8.4 Operational Timeline

```http
GET /api/workspaces/{workspaceId}/timeline?subjectType=...&subjectId=...&cursor=...
```

The timeline is a permission-filtered projection over domain events, commands, optional V2 workflow events, assignments, schedules, form submissions, and relevant audit records. It is not a new editable business object.

## 9. Product Surface Specification

### 9.1 Role-Focused Navigation

Default business navigation MUST be based on user jobs:

```text
Workbench / My Work
Customers
Sales / Quotes
Planning
Work Execution
```

Forms, Processes, Automations, Members & Access, Migration, Audit, Outbox, Packs, and customization MUST be grouped under Management and shown only to appropriate roles. Workflow, Outbox, and Migration MUST NOT appear as normal daily-work destinations for technicians and sales users.

### 9.2 Contextual Action Bar

Quote, Work Order, Visit, Assignment, and form-submission detail MUST show commands allowed for the current actor and state. Buttons use business language such as Submit for Approval, Return for Changes, Assign Technician, Reschedule, Start Travel, Arrive, Submit Service Report, Reopen.

The UI MUST NOT:

- show generic Edit for immutable revisions;
- render a governed lifecycle field as an editable select/input;
- claim success before the command response commits;
- hide a rejected command behind a generic “save failed” message;
- expose actions the actor cannot perform and only rely on server rejection.

The server remains authoritative even when the UI hides unavailable actions.

### 9.3 Forms

The primary Forms authoring experience MUST provide a visual block designer for:

```text
section/header
short and long text
number, currency, date, datetime, select, boolean
checklist with pass/fail/not-applicable and per-item note/evidence policy
evidence/file/photo
acknowledgment/signature
repeatable group
```

JSON MAY exist in an Advanced/Diagnostics view for authorized administrators; it MUST NOT be the default create/edit path. Publishing creates an immutable form version. Bindings are created through labeled selectors for usage type, target, trigger/step, and requirement policy rather than free-text internal keys.

Technicians see the bound form inside assigned work, not the Forms management library.

### 9.4 Planning

Planning MUST provide four synchronized views:

1. backlog of unscheduled/unassigned work;
2. day/week calendar;
3. resource timeline / simplified Gantt;
4. basic marker map with a list fallback when coordinates are absent.

All mutations invoke Assignment/Schedule commands. Dragging or dropping MUST submit expected version and idempotency key, display conflicts, and restore/reload authoritative state on failure.

### 9.5 Readability And Localization

All P0 business surfaces MUST:

- show related record labels as links instead of raw IDs;
- use workspace locale/timezone/currency consistently;
- translate enum values using metadata rather than hard-coded page maps;
- show business field labels instead of field keys;
- provide useful empty states tied to the user’s role and next setup action;
- preserve technical detail in an optional diagnostics view.

### 9.6 Mobile

The `/m` shell, PWA manifest/service worker policy, mobile execution journey, evidence handling, security, interaction, and performance budgets MUST conform to the required mobile sub-specification. The mobile shell consumes the same commands, Work Items when present, forms, assignments, schedule, and timeline contracts defined here.

## 10. Security, Permissions, And Audit

- Workspace membership and the business permission are both required.
- Work-item actions also require assignment/candidate eligibility; workspace admin alone does not imply assigned approver. Command-only actions use the command's permission and assignment policy directly.
- Self-approval is rejected unless an explicit process policy and permission allow it.
- Administrative override requires a distinct permission, mandatory reason, and audit event.
- Every successful command writes actor, command type/ID, aggregate/version, before/after state, domain events, and resulting Work Item IDs when a Work Item is created or completed.
- Every rejected governed command returns a stable code, request ID, and safe recovery details.
- Evidence and customer/site history follow tenant and relation-level authorization.
- Persona switching and demo shortcuts are impossible in production mode.

## 11. Verification And Release Gates

### 11.1 Required Automated Suites

```text
architecture tests       ownership, module discovery, no active V1 path, workflow optionality, no duplicate lifecycle owners
migration fixtures       every V1 state, terminal/running, conflict, missing actor, restart/idempotency
command contract tests   permission, assignment, idempotency, optimistic locking, atomic event/audit/work item
demo validator           all Section 7 relations and visible labels
browser/API journeys     role-based canonical scenarios on a reset local mirror
responsive/PWA tests     desktop and phone viewports plus weak-network/offline safety
accessibility tests      keyboard, labels, focus, errors, contrast for P0 paths
performance tests        My Work, Planning, commands, and mobile budgets with documented load profile
```

Platform unit tests passing is necessary but not sufficient. A release with no browser journey coverage fails acceptance.

### 11.2 Canonical Browser Journeys

Automated tests MUST prove, through supported UI actions:

1. sales representative revises and submits a Quote;
2. sales manager sees it in My Work, returns it, and the representative resubmits;
3. manager approves, customer acceptance is recorded, and conversion creates exactly one Work Order;
4. dispatcher sees the Work Order in backlog, assigns a technician, and schedules a Visit;
5. conflict is displayed and an unauthorized override is rejected;
6. technician sees the Visit in personal My Work/mobile Today and executes travel/arrival;
7. required checklist, evidence, and acknowledgment block incomplete submission;
8. supervisor returns the deliverable with a reason and prior revision remains immutable;
9. technician resubmits; supervisor accepts; Visit and Work Order complete;
10. an authorized reopen preserves prior events, decisions, submissions, and schedule history;
11. customer/site/asset timeline identifies every material step with readable labels;
12. another tenant cannot resolve any identifier or cached response from the scenario.
13. a command-only Work Order/Visit journey completes without creating or requiring a Workflow V2 instance.

No scenario setup MAY directly update governed business state in SQL.

### 11.3 Presentation Gates

For the Golden Demo P0 routes:

```text
raw relation IDs visible as primary values       = 0
raw enum values visible as business labels       = 0
editable governed lifecycle fields               = 0
daily-role pages requiring schema/command keys    = 0
broken or unresolved relation links               = 0
mixed locale caused by seed data                  = 0
```

### 11.4 Operational SLOs

```text
command API p95                 <= 500 ms excluding upload/external delivery
My Work query p95               <= 500 ms
Planning 7-day query p95        <= 1 s
duplicate command effects       = 0
lost domain/audit event         = 0
lost workflow event, when used  = 0
silent migration conflict       = 0
```

The release report MUST state record count, resources, concurrent actors, device, network, build, and measurement method.

## 14. Pull Request Drift Control

Every v0.5.1 pull request MUST state:

```text
spec sections implemented
persona and product outcome changed
authoritative aggregate/state owner
named commands used
permission and assignment checks
event/audit/work-item outcome, if any
demo data or migration impact
automated acceptance evidence
explicit deferral or waiver, if any
```

Review MUST reject a change that:

- adds a direct lifecycle write;
- introduces a second schedule/assignment/form truth;
- exposes an internal module/object as a new product page without role justification;
- adds a new P0 API without a typed contract and test;
- treats raw JSON or IDs as the primary business experience;
- changes the Golden Demo without validator coverage;
- declares completion using only unit tests.

## 15. Definition Of Done

v0.5.1 is complete only when:

1. all P0 deliverables in Section 3.1 are implemented;
2. every `MUST` has an automated test or documented inspection artifact;
3. Workflow V1 is read-only historical data and no supported path executes it;
4. the Golden Demo is deterministic, connected, current-date-safe, and understandable;
5. five personas complete the canonical journey without database or JSON manipulation;
6. desktop and mobile use identical commands, permissions, forms, and business data; workflow-backed and command-only journeys both work through the same command kernel;
7. migration, product journey, presentation, security, accessibility, and SLO gates pass;
8. unresolved P0 defects and silent migration conflicts are zero;
9. the release evidence is reviewed jointly by Product and Engineering.

Anything less is an internal preview, not the v0.5.1 commercial FSM release.

## 16. Requirement Traceability Matrix

Engineering tickets, pull requests, tests, and release evidence MUST reference one or more of these stable requirement IDs:

| ID | Required outcome | Minimum acceptance evidence |
|---|---|---|
| MIG-01 | all active V1 definitions/instances are inventoried and migrated or explicitly conflicted | migration fixtures plus before/after report |
| MIG-02 | migration is restartable and idempotent; V1 execution is retired | interrupted-run test, rerun test, V1 API `410` test |
| CMD-01 | every canonical lifecycle edge uses a named command | command registry architecture test |
| CMD-02 | governed create/update and immutable revisions cannot bypass commands | API contract and browser tests |
| CMD-03 | state, event, audit, projection, and next Work Item commit coherently | transaction/failure-injection tests |
| DEMO-01 | Golden Demo installs CRM + Sales Quote + FSM with all five personas | reset/seed/validate CI job |
| DEMO-02 | demo identity, resource, assignment, schedule, optional workflow, form, and subject relations are complete | demo relation validator |
| DEMO-03 | relative dates and one schedule truth keep Today/My Work/Planning useful | timezone/date-rollover fixture |
| PRES-01 | relations, enums, fields, dates, currency, and units use shared presentation metadata | presentation gates on all P0 routes |
| UX-01 | business detail pages expose contextual commands and never ordinary governed-state editing | role-based browser tests |
| NAV-01 | daily navigation is role/job based; technical management surfaces are restricted | persona navigation snapshots/assertions |
| WORK-01 | Mine, Unassigned, Team, and Overdue queues return readable, actionable items | My Work API and browser tests |
| FORM-01 | visual designer publishes immutable versions and typed bindings without required JSON editing | administrator authoring journey |
| FORM-02 | required data/evidence, return, resubmit, and immutable submission revisions work end to end | technician/supervisor journey |
| PLAN-01 | backlog, calendar, resource timeline, and map share one schedule query | cross-view consistency test |
| PLAN-02 | assign/reassign/schedule/reschedule/unschedule use commands and expose conflicts | dispatcher browser/concurrency tests |
| HIST-01 | customer/site/asset operational history is readable and permission-filtered | timeline API, role, and tenant tests |
| PWA-01 | mobile implementation satisfies the required mobile sub-specification | device/viewport, cache, weak-network, and performance report |
| SEC-01 | membership, permission, assignment, self-approval, override, and tenant isolation are enforced | negative authorization suite |
| E2E-01 | all canonical browser journeys in Section 11.2 pass without direct SQL | production-like browser/API report |
| OPS-01 | SLO, duplicate-effect, event-loss, and migration-conflict gates pass | load/recovery report with declared profile |

A requirement MAY be split across tickets, but it MUST have one accountable owner and one final evidence link. A ticket title or manual statement is not evidence.
