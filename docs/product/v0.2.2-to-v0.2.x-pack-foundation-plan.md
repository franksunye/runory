## 1. Version Thesis

`v0.2` proves that Runory is becoming usable as a business workspace.

`v0.2.1` makes the workbench composable so installed modules and packs can contribute business widgets without hardcoding dashboard logic.

`v0.2.2` through `v0.2.x` should prove that Runory can support a **credible official business pack portfolio** without collapsing into a monolithic suite.

The goal is not to fully implement 30+ modules during `v0.2.x`.

The goal is to make the architecture, catalog, object ownership, demo data, and reference implementations strong enough that new packs can be added predictably.

## 2. Product Boundary

`v0.2.x` is the pack foundation series.

It should focus on:

```text
shared business modules
pack-owned object models
pack-owned demo data
cross-pack references
pack-aware workbench composition
business routes/views for MVP packs
Agent-safe customization on shared and pack-owned objects
```

It should not focus on:

```text
full workflow engine
advanced automation builder
enterprise UI polish across all surfaces
omnichannel messaging
accounting / tax / inventory
full marketing automation
full GEO platform
```

Those belong to `v0.3+` or later pack-specific versions.

## 3. Proposed 0.2.x Sequence

### 3.1 v0.2.2 — CRM Core Object Model Correction

Goal:

Replace the early `customer`-centric CRM model with a more mature shared business foundation.

Required outcomes:

```text
runory.company
runory.contact
runory.deal
runory.task
crm-lite-pack uses company/contact/deal/task
customer becomes lifecycle_stage, not the primary object
demo data uses company/contact/deal/task
Workbench widgets use company/contact/deal/task declarations
Agent customization examples target company/contact/deal
```

Acceptance:

- A new workspace can install CRM Lite and see companies, contacts, deals, and tasks.
- Existing `customer` references are either migrated, aliased, or clearly deprecated.
- The old "Customer schema contains extension field" demo is replaced with a company/contact/deal-oriented customization.
- Pack-owned demo data is the only CRM demo data source.

### 3.2 v0.2.3 — Shared Business Module Contract

Goal:

Make shared business modules safe to reuse across multiple packs.

Required outcomes:

```text
one object key, one owning module
pack dependency dedupe
cross-pack relation declaration
shared module extension point rules
pack-specific terminology overlays
pack install compatibility checks
cross-pack demo data references
```

Acceptance:

- Two packs can depend on `runory.company`, `runory.contact`, or `runory.task` without duplicate install or duplicate navigation chaos.
- A pack can label a shared object differently through template terminology without forking the object.
- Catalog validation rejects duplicate object ownership across installed packs unless it is an explicit compatible upgrade.

### 3.3 v0.2.4 — FSM Pack MVP

Goal:

Introduce the second credible official business pack and use it to validate shared object reuse.

MVP modules:

```text
runory.service-site
runory.asset
runory.work-order
runory.service-visit
runory.service-report
runory.technician
```

Shared dependencies:

```text
runory.company
runory.contact
runory.task
```

Required outcomes:

- FSM pack manifest and template.
- Pack-owned demo data.
- Work orders, sites, assets, technicians, visits, and service reports have basic list/detail/create/edit surfaces.
- Workbench shows operational metrics: open work orders, scheduled visits today, overdue work orders, active assets.
- FSM demo journey works end to end.

Acceptance:

```text
Install FSM Pack
Load demo data
Open today's schedule
Assign a technician
Complete a service visit
Submit a service report
See workbench and activity feed update
```

### 3.4 v0.2.5 — Sales Quote Pack MVP

Goal:

Introduce the third official pack to validate commercial proposal flows and cross-pack references.

MVP modules:

```text
runory.product-service
runory.price-book
runory.quote
runory.quote-approval
```

Required outcomes:

- Quote can reference company, contact, deal, work order, service site, and asset.
- Quote lines support products/services, quantity, unit price, and totals as snapshots.
- Quote approval is lightweight and auditable.
- Quote demo data includes deal-origin and work-order-origin quotes.
- Workbench shows open quotes, pending approvals, quote value, and expiring quotes.

Acceptance:

```text
Create quote from CRM deal
Create quote from FSM work order
Add quote lines
Submit for approval
Approve
Mark accepted
See workbench and activity feed update
```

### 3.5 v0.2.6 — Marketing Capture MVP

Goal:

Add public-facing demand capture without becoming a full CMS or marketing automation platform.

MVP modules:

```text
runory.campaign
runory.form
runory.landing-page
runory.submission
runory.consent
```

Required outcomes:

- Public landing page/minisite route.
- Public form submission route with spam/rate-limit foundation.
- Submissions can create or link contacts/companies/deals.
- Source attribution and consent fields are captured.
- Agent can propose form fields and landing page content through review/approval.

Acceptance:

```text
Create landing page
Add form
Publish safely
Submit anonymously
Create contact/deal from submission
See submission source in CRM
```

### 3.6 v0.2.7 — AI Visibility / GEO Seed Capability

Goal:

Add the first GEO/AEO capability layer on top of Marketing Capture without building a standalone GEO platform.

MVP modules or capabilities:

```text
entity_profile
question_map
answer_block
citation_source
ai_visibility_check
```

Required outcomes:

- Generate question map for product/service or landing page.
- Generate approved answer blocks with evidence sources.
- Connect answer blocks to landing pages, minisites, or knowledge articles.
- Store simple AI visibility checks by query/engine/locale.
- Agent suggests improvements but does not auto-publish public content.

Acceptance:

```text
Create question map
Generate answer blocks
Approve content
Publish on landing page
Run visibility check
See improvement suggestions
```

### 3.7 v0.2.8 — Customer Service MVP

Goal:

Add support communication and issue resolution as a separate pack from FSM and After-sales.

MVP modules:

```text
runory.ticket
runory.conversation
runory.knowledge
runory.support-sla
```

Required outcomes:

- Tickets can link company/contact/asset/work_order/quote/task.
- Ticket conversations and comments are available.
- Knowledge articles can resolve tickets.
- SLA light status is visible.
- Tickets can create FSM work orders when onsite service is needed.

Acceptance:

```text
Create ticket
Add conversation/comment
Link company/contact/asset
Escalate to FSM work order
Resolve with knowledge article
See support workbench update
```

### 3.8 v0.2.9+ — After-sales Service MVP

Goal:

Add post-sale obligations after CRM, FSM, Quote, and Customer Service have the required foundation.

MVP modules:

```text
runory.warranty
runory.entitlement
runory.return-request
runory.repair-request
runory.maintenance-plan
runory.customer-success
```

Required outcomes:

- Warranty and entitlement can reference company/contact/product_service/asset/quote.
- Return and repair requests can link tickets and work orders.
- Maintenance plans can generate follow-up tasks or work orders.
- Customer success follow-ups appear in workbench.

Acceptance:

```text
Open support ticket
Check warranty/entitlement
Create repair request
Create FSM work order if needed
Create quote if paid repair
Close loop with service report and follow-up
```

## 4. Cross-cutting Engineering Requirements

Each `v0.2.x` pack or module slice must include:

```text
catalog manifest
migrations
demo data
basic list/detail/create/edit views
workbench widget declarations
pack default layout
navigation/template terminology
permissions
extension points
Agent customization examples
install tests
demo data tests
cross-pack reference tests when applicable
upgrade policy
```

## 5. Risk Controls

### 5.1 Do Not Build Everything At Once

The portfolio can reach 30 MVP modules and 50+ later modules. `v0.2.x` must still be delivered in thin, complete vertical slices.

### 5.2 Keep SaaS Core Clean

Shared business modules are not SaaS Core.

SaaS Core owns:

```text
auth
tenant / organization / workspace
membership
catalog
installations
audit
extension runtime
usage
API keys
sessions
```

Business modules own:

```text
company
contact
task
asset
quote
ticket
submission
```

### 5.3 Avoid Public Surface Risk

Marketing Capture and GEO introduce public routes and anonymous submissions. They must include:

```text
rate limiting
spam mitigation foundation
safe publishing states
content approval
audit
unpublish path
```

### 5.4 Avoid Fake Workflow

`v0.2.x` can use lightweight status objects and approval records. Full workflow and automation should wait for `v0.3`.

## 6. Exit Criteria For 0.2.x

Runory is ready to move into `v0.3` when:

- CRM object model is corrected.
- At least one second pack, preferably FSM, works end to end.
- Sales Quote pack has a working MVP or well-tested implementation slice.
- Pack install, demo data, workbench composition, and shared object reuse are reliable.
- Marketing Capture and GEO have either shipped as MVP slices or are explicitly deferred.
- Product and engineering agree no more major pack model expansion is needed before runtime/UX work.

## 7. Handoff To 0.3

`v0.3` should not continue broadening the pack portfolio by default.

`v0.3` should polish:

```text
workflow runtime
automation
Agent operations
UX/UI system
business object experience
observability
performance
permissions
mobile/responsive quality
```

See [Runory Cloud 0.3 Runtime And Experience Plan](./v0.3-runtime-experience-plan.md).
