# Runory v0.8 External Agent Roadmap

Status: Proposed  
Date: 2026-07-15  
Scope: v0.8 Agent-native FSM implementation

## 1. Purpose

v0.8 proves Runory's central product differentiation:

> **Runory does not build a proprietary Agent inside the FSM product. It enables capable external Super Agents to configure, operate, implement, and deploy Runory through governed interfaces.**

Runory's strategy is to work with Agent ecosystems such as ChatGPT, Codex, Claude, Cursor, Trae, WorkBuddy, and future enterprise Agents.

```text
External Super Agent
        ↓
MCP / Skill / SDK
        ↓
Runory Runtime
        ↓
Governed FSM Execution
```

v0.8 is not a full Marketplace release, universal autonomous implementation platform, or proprietary Agent product.

Architecture handoff:
[v0.6 Deferred Work Handoff](../architecture/v0.6-deferred-work-handoff.md).
The public SDK template and Agent discovery Runtime/API are admitted through
`V06-D03` and `V06-D04` only when an approved external Agent consumes them in
an end-to-end scenario.

## 2. Product Boundary

External Agents may:

* discover supported capabilities and schemas;
* query authorized FSM data;
* execute named business Commands;
* configure a Workspace through governed metadata changes;
* install and adapt supported FSM Packs;
* assist supported Cloud-to-Local deployment;
* produce plans, Diffs, reports, diagnostics, and verification evidence.

External Agents may not:

* directly mutate the database;
* bypass permissions or Command contracts;
* modify official Module source;
* deploy arbitrary generated production code;
* silently apply high-risk changes;
* assume unsupported capabilities exist.

## 3. Core Execution Contract

All Agent operations use the same governed path:

```text
Discover
→ Plan
→ Preview / Diff
→ Permission Check
→ User Confirmation when required
→ Apply through named Command or configuration API
→ Verify
→ Audit
→ Rollback when supported
```

MCP, Skills, and SDKs are first-class product interfaces, not secondary developer conveniences.

## 4. v0.8.0 — External Agent Operates Runory FSM

An approved external Agent can safely:

```text
inspect available objects, fields, relations, views, and Commands
query authorized Contacts and Companies across Lead/customer lifecycle stages, Quotes, Work Orders, Visits, schedules, invoices, and payments
create or update records through named Commands
execute supported FSM lifecycle actions
retrieve result, error, audit, and verification information
```

### Required interfaces

```text
capability discovery
object and schema discovery
authorized query tools
versioned Command tools
stable result and error contracts
request, actor, Workspace, and Agent identity propagation
```

### Release gate

A real FSM workflow can be completed through an external Agent without direct database access or a special Agent-only business path.

## 5. v0.8.1 — External Agent Configures a Cloud Workspace

An approved Agent can:

```text
inspect the effective Workspace model
add or change governed custom fields
modify forms, views, statuses, notifications, roles, and permissions
configure supported workflows and automations
create reports, dashboards, and document templates
produce a change plan and machine-readable Diff
preview, confirm, apply, verify, audit, and rollback changes
```

Workspace customization must remain separate from Runory Core and official Module source.

### Release gate

A customer-specific FSM configuration can be implemented and later upgraded without a Core fork.

## 6. v0.8.2 — External Agent Installs and Adapts an FSM Pack

An approved Agent can:

```text
discover supported FSM Packs and compatibility ranges
select a Pack based on declared customer requirements
install the locked Pack and Module dependency set
apply Template, seed, terminology, role, and navigation configuration
adapt fields, forms, workflows, reports, and templates through Workspace Extensions
run compatibility, migration, and acceptance checks
produce an implementation report
```

Initial validation stays within FSM industries, for example:

* HVAC;
* plumbing;
* waterproofing;
* landscaping;
* pool service;
* cleaning;
* installation and project service.

### Release gate

An external Agent can turn a clean Workspace into an accepted, industry-shaped FSM implementation using supported artifacts and governed extensions.

## 7. v0.8.3 — External Agent Deploys a Supported Local Edition

An approved Agent can:

```text
preflight the target environment
export supported Cloud Workspace configuration, extensions, and data
initialize Local database, storage, identity, and MCP configuration
collect required secrets and provider settings through supported boundaries
install using a supported containerized or packaged path
run health, acceptance, backup, restore, and upgrade checks
produce a deployment and Cloud-to-Local difference report
```

This milestone does not promise universal infrastructure automation or arbitrary platform support.

### Release gate

An external Agent can produce a verifiable supported Local deployment from a new or existing Workspace without manual database construction.

## 8. v0.8.4 — Agent Access Governance and Evaluation

Required governance:

```text
Agent identity and delegated authorization
API key / OAuth-ready authentication boundary
Tool-level and object-level scopes
Workspace and tenant isolation
risk-classified confirmation policy
execution, time, token, and cost limits
secret and sensitive-data handling
idempotency and concurrency protection
Agent Run log and diagnostics
Diff, verification, and rollback evidence
task-level correctness and safety evaluation
```

The same authorization rules apply to Web, API, MCP, background jobs, and external Agents.

## 9. SDK and Interface Deliverables

### MCP

* stable versioned tool names;
* capability and schema discovery;
* query and Command execution;
* Workspace configuration tools;
* Pack installation and compatibility tools;
* deployment and diagnostics tools;
* structured errors and retriable failure semantics.

### Skills

* official FSM operation Skills;
* configuration and implementation Skills;
* Pack selection and installation Skills;
* migration and acceptance Skills;
* Local deployment Skills;
* clear preconditions, limits, and escalation boundaries.

### SDK

* typed contracts for supported languages;
* authentication and RequestContext handling;
* discovery, query, Command, configuration, and installation clients;
* idempotency and retry helpers;
* test fixtures and sandbox support;
* compatibility and version negotiation.

## 10. Explicit Non-goals

v0.8 does not require:

* a Runory proprietary Built-in Agent;
* training or hosting a foundation model;
* a general chat product;
* a full public Module Marketplace;
* unrestricted third-party code execution;
* fully autonomous implementation without human confirmation;
* support for every Super Agent or infrastructure provider;
* direct Agent database access;
* a second business execution path built only for Agents.

## 11. Success Criteria

v0.8 succeeds when all three primary scenarios work through approved external Agents:

1. **Operate** — query and execute real FSM business work;
2. **Implement** — configure a Workspace and install/adapt an FSM Pack;
3. **Deploy** — create and verify a supported Local environment.

Across all scenarios:

* execution uses the same governed Runtime as the UI;
* permissions and Workspace isolation are enforced;
* changes are observable and auditable;
* failures are visible and diagnosable;
* risky changes require confirmation;
* official Modules remain upgrade-safe;
* no direct database repair is required for normal operation.

## 12. Release Outcome

> **An authorized user can use a capable external Super Agent to operate, configure, implement, and deploy Runory FSM through MCP, Skills, and SDKs, while Runory remains the deterministic and governed business execution layer.**
