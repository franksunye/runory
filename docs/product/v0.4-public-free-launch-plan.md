## 1. Version Thesis

If `v0.3` passes internal validation, `v0.4` should become the first **publicly usable free launch** of Runory Cloud.

`v0.4` is not primarily about adding more business packs or deeper runtime features.

It is about making Runory usable by the expected first audience:

```text
English-first users
open-source curious developers
SMB operators trying Cloud for free
builders using MCP / Skill / Agent workflows
early adopters evaluating Runory's pack model
```

Version promise:

> A user can discover Runory from the website, understand what it is, create a free Cloud workspace, install official packs, use Agent-guided operations safely, and follow public documentation without private explanation.

## 2. Product Boundary

`v0.4` should focus on:

```text
English-first internationalization
public Cloud launch readiness
website and documentation refresh
free plan packaging
MCP / Skill / Agent operations 1.0
onboarding and demo journeys
release quality and production checks
```

`v0.4` should explicitly not include:

```text
Stripe integration
billing workflows
paid subscriptions
complex pricing tiers
enterprise procurement
full marketplace monetization
```

The public offer is:

```text
Free version only
No Stripe / Billing integration
Clear future-commercialization note
No surprise paywall
```

## 3. Iteration Plan

`v0.4` should be executed as five focused iterations.

The order matters. Do not start by adding more runtime depth or more business packs. The product already has enough internal capability to support a first public free launch. The immediate work is to make that capability understandable, stable, self-service, and safe for external users.

### 3.1 v0.4.0 - Foundation Cleanup And Architecture Lock

Goal:

Make the `v0.3` runtime clean enough that `v0.4` launch work does not keep paying for old route, migration, and development-environment ambiguity.

Required outcomes:

```text
dynamic object route shell is the only default object page path
no fixed route wrappers for official object list/detail/create pages
object route segment conversion is canonical and tested
navigation API contract is current-only, no legacy array fallback
local dev database migration state is reliable
demo workspace bootstrap is reproducible
root-level typecheck is green
production build is green
critical object routes are browser-verified
```

Object runtime boundary:

```text
metadata-driven:
  /w/[workspaceId]/[objectKey]
  /w/[workspaceId]/[objectKey]/new
  /w/[workspaceId]/[objectKey]/[id]

explicit product pages:
  dashboard
  modules / pack onboarding
  customize / Agent proposal review
  workflows
  automations
  audit / governance
  export / trash / settings / members
  landing-pages, where the page is a composite publishing experience
```

Engineering tasks:

- Delete remaining fixed object route wrappers when they only pass `objectKey`, `viewKey`, labels, or relation props into shared components.
- Move object title, list labels, create labels, relation display, and route generation into metadata/runtime helpers.
- Ensure `service-sites`, `work-orders`, `companies`, and other plural route segments map deterministically to object keys.
- Remove backward compatibility branches for pre-v0.3 navigation API responses.
- Add or update focused tests for dynamic route conversion and current navigation response shape.
- Keep local dev repair scripts or documented commands for migration drift, but do not normalize drift in product code.

Acceptance:

- CRM and FSM object URLs all resolve through the dynamic shell.
- New, detail, and list routes work for at least company, contact, deal, task, work order, service site, asset, service visit, service report, and technician.
- No official object page requires a per-object Next.js route file.
- `pnpm typecheck`, core tests, and production build pass.

### 3.2 v0.4.1 - English-First Product UI

Goal:

Make the Cloud product usable by an English-speaking user without private explanation or mixed-language blocking surfaces.

Required outcomes:

```text
English is the default product language
workspace shell is English-complete
global dashboard is English-complete
pack install and demo data flows are English-complete
dynamic object pages are English-complete
customize / extension / Agent proposal surfaces are English-complete
workflow and automation pages are English-complete enough for launch
errors, request IDs, empty states, loading states, and toasts are English-complete
Chinese remains available for local demos
```

Engineering tasks:

- Migrate remaining workspace page-level hardcoded Chinese strings to message keys.
- Move object/page nouns that still live in static page wrappers into metadata or locale-aware helpers.
- Normalize pack terminology display so English-first launch does not depend on Chinese manifest labels.
- Add locale-aware formatting for dates, numbers, currency-like fields, and relative time where user-visible.
- Add a mixed-language scan to the release checklist.

Acceptance:

- A fresh English user can complete the canonical workspace journey without Chinese-only UI.
- Locale fallback always returns English.
- Chinese still works for existing local demos.
- No launch-critical Cloud page has hardcoded Chinese copy outside pack-provided sample data.

### 3.3 v0.4.2 - Website And Documentation Launch Surface

Goal:

Make Runory understandable from public pages and docs without a founder walkthrough.

Required outcomes:

```text
home page explains the product category and promise
product / how-it-works page explains workspace, pack, module, and Agent operations
packs page distinguishes available, exploratory, and planned packs
pricing page states Free Preview clearly

docs index exists and is navigable
getting started guide covers the canonical journey
concept docs explain SaaS Core, Module, Pack, Workspace Extension, Agent Operation, Catalog, Workflow, and Automation
release notes and changelog are public-ready
```

Engineering and content tasks:

- Refresh website navigation and page copy around the free public preview offer.
- Add a Packs page that does not overclaim future packs.
- Update Pricing to Free-only, no payment method, no Stripe, fair-use posture.
- Write docs for Getting Started, Concepts, Workspace Guide, Packs and Modules, Agent Operations, MCP / Skill Usage, SDK / Module Development, Admin / Governance, Troubleshooting, and Release Notes.
- Refresh README with Cloud, local development, and public preview positioning.

Acceptance:

- A new visitor can understand what Runory is, what is free, what is available now, and what is not yet available.
- Docs can guide a user through account creation, workspace creation, pack install, demo data loading, workbench use, safe customization, and audit inspection.
- Public pages do not make paid, compliance, marketplace, or enterprise claims beyond the current product.

### 3.4 v0.4.3 - Free Onboarding And Canonical Demo Journey

Goal:

Make the first workspace experience feel intentional, alive, and supportable.

Required outcomes:

```text
new user onboarding path is clear
free plan boundaries are visible
pack recommendations are understandable
install and demo data are separate and reliable
demo data tells a coherent story
dashboard gives next-step guidance
object pages show useful demo records and relations
audit and governance are discoverable
known limitations are visible without scaring users away
```

Engineering tasks:

- Define the first recommended pack journey for public preview. Prefer one canonical path over many shallow ones.
- Ensure pack install, demo data loading, dashboard widgets, object pages, and audit all work from a fresh workspace.
- Add guided next steps after pack install and after demo data load.
- Verify demo data does not expose raw IDs where relation labels should be available, except where explicitly acceptable for current limitations.
- Define free plan boundaries in UI and docs: workspace/member limits, pack availability, operation limits, MCP/API access, storage posture, and support expectations.

Acceptance:

- A new user can create a workspace, install an official pack, load demo data, understand the dashboard, inspect object records, perform a safe customization, and inspect audit.
- The same journey works after clearing local state and creating a fresh workspace.
- Failed install/demo states show request IDs and actionable diagnostics.

### 3.5 v0.4.4 - MCP / Skill / Agent Operations 1.0 And Release Gate

Goal:

Ship the first stable external operations surface and pass public launch quality gates.

Required outcomes:

```text
operation schemas are documented
MCP or equivalent external operation interface is available
Runory Codex Skill is available or documented
workspace.inspect works
pack.list works
pack.install works
object.inspect works
object.field.add works through plan / preview / apply
view.inspect and view.modify work for safe view changes
agent_operation.history works
audit.search works
operation safety model is documented
release quality gates pass
```

Engineering tasks:

- Define the external operation contract: discover, plan, validate, preview, apply, verify, audit, rollback where possible.
- Implement or document the minimum MCP interface for local and Cloud use.
- Create examples for safe field addition and view modification.
- Ensure operations route through the same governed APIs used by the Cloud UI.
- Add critical E2E coverage for the canonical journey and Agent customization flow.
- Run mobile/responsive, accessibility, public-page performance, build, typecheck, tests, auth, pack install, demo data, and error-state gates.

Acceptance:

- A developer can connect an Agent/MCP client to a Runory workspace.
- The agent can inspect installed packs and objects.
- The agent can propose a safe field or view customization.
- The user can preview and approve the change in Runory.
- The final state is visible in UI and audit.
- v0.4 release notes, changelog, known limitations, and feedback path are ready.

## 4. Internationalization Scope

### 4.0 v0.3 Foundation And Deferred Work

v0.3 delivered the i18n infrastructure and shared component migration:

- `LocaleProvider` with `{param}` interpolation, 306+ message keys (`en` + `zh`)
- All 13 shared components fully migrated (SchemaForm, SchemaTable, SchemaField, NavigationShell, EarlyAccessBanner, ObjectListPage, ObjectDetailPage, ObjectCreatePage, WidgetRenderer, DashboardEditMode, ExtensionPanel, AddFieldWizard, AuditTimeline, etc.)
- `formatRelativeTime` unified into SchemaTable

v0.3 deferred to v0.4 (recorded 2026-06-24):

- **Workspace page-level i18n migration**: non-object workspace pages still contain hardcoded Chinese strings such as page titles, descriptions, table headers, form labels, and action copy. Shared components and dynamic object pages are already i18n-ready; remaining work is concentrated in composite workspace pages and product-specific flows.
- Mobile/responsive pass, accessibility pass, public page performance pass — not explicitly verified in v0.3.

These deferred items are mandatory inputs to v0.4 launch readiness.

### 4.1 English First

English must be the default, complete, and launch-ready language.

## 12. Exit Criteria For v0.4

Runory `v0.4` is launch-ready when:

- English-first public website and docs are complete enough for self-service evaluation.
- A new user can create and use a free workspace without private help.
- At least one official pack journey works end to end with demo data.
- MCP / Skill / Agent operations 1.0 can safely inspect and customize a workspace.
- Free plan boundaries are clear.
- No billing or Stripe dependency blocks use.
- Release notes and known limitations are public.
