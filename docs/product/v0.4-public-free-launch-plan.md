## 1. Version Thesis

If `v0.3` passes internal validation, `v0.4` should become the first **publicly usable free launch** of Runory Cloud.

`v0.4` is not primarily about adding more business packs or deeper runtime features.

It is about making Runory usable by the expected first audience:

```text
English-first users
open-source curious developers
SMB operators trying Cloud for free
builders using MCP / Skill / Agent workflows
early adopters evaluating Runory's pack model
```

Version promise:

> A user can discover Runory from the website, understand what it is, create a free Cloud workspace, install official packs, use Agent-guided operations safely, and follow public documentation without private explanation.

## 2. Product Boundary

`v0.4` should focus on:

```text
English-first internationalization
public Cloud launch readiness
website and documentation refresh
free plan packaging
MCP / Skill / Agent operations 1.0
onboarding and demo journeys
release quality and production checks
```

`v0.4` should explicitly not include:

```text
Stripe integration
billing workflows
paid subscriptions
complex pricing tiers
enterprise procurement
full marketplace monetization
```

The public offer is:

```text
Free version only
No Stripe / Billing integration
Clear future-commercialization note
No surprise paywall
```

## 3. Internationalization Scope

### 3.0 v0.3 Foundation And Deferred Work

v0.3 delivered the i18n infrastructure and shared component migration:

- `LocaleProvider` with `{param}` interpolation, 306+ message keys (`en` + `zh`)
- All 13 shared components fully migrated (SchemaForm, SchemaTable, SchemaField, NavigationShell, EarlyAccessBanner, ObjectListPage, ObjectDetailPage, ObjectCreatePage, WidgetRenderer, DashboardEditMode, ExtensionPanel, AddFieldWizard, AuditTimeline, etc.)
- `formatRelativeTime` unified into SchemaTable

v0.3 deferred to v0.4 (recorded 2026-06-24):

- **Workspace page-level i18n migration**: 39 page files under `apps/cloud/src/app/w/[workspaceId]/` contain ~4,027 lines with hardcoded Chinese (page titles, descriptions, table headers, form labels). Shared components are already i18n-complete; remaining work is page-specific strings.
- Mobile/responsive pass, accessibility pass, public page performance pass — not explicitly verified in v0.3.

These deferred items are mandatory inputs to v0.4 launch readiness.

### 3.1 English First

English must be the default, complete, and launch-ready language.

Required outcomes:

```text
all public website pages in English
login / onboarding in English
global dashboard in English
workspace shell in English
pack install flows in English
core object pages in English
workbench in English
Agent operation flows in English
errors and empty states in English
docs navigation in English
```

Chinese remains supported, but `v0.4` launch readiness is judged primarily against English.

### 3.2 Internationalization Acceptance

Acceptance:

- A new English-speaking user can complete the canonical journey without mixed-language blocking UI.
- Locale fallback always returns English.
- Object labels, field labels, pack names, navigation, and empty states support localized display.
- Dates, numbers, currency, relative time, and validation messages use locale-aware presentation.
- Public pages have correct document language and metadata.

### 3.3 Deferred

Deferred until after launch:

```text
many additional locales
professional translation workflow
localized SEO content clusters
locale-specific support operations
paid localization packages
```

## 6. MCP / Skill / Agent Operations 1.0

`v0.4` should define and ship the first stable external operations surface.

### 6.1 Operation Contract

All Agent-facing operations should follow:

```text
discover
plan
validate
preview
apply
verify
audit
rollback where possible
```

### 6.2 Required Operation Families

MVP operation families:

```text

pack.list
pack.install
object.inspect
object.field.add
view.inspect
view.modify
workflow.inspect
automation.inspect
agent_operation.history

```

Optional if ready:

```text
demo_data.load
quote.draft
form.generate
answer_block.generate
```

### 6.3 MCP Server / Skill Deliverables

Required deliverables:

```text
Runory MCP server or documented MCP interface
Runory Codex Skill
operation schemas
examples
local dev instructions
Cloud auth instructions
safety model
known limitations
```

### 6.4 Acceptance

Acceptance:

- A developer can connect an Agent/MCP client to a Runory workspace.
- The agent can inspect installed packs and objects.
- The agent can propose a safe field/view customization.
- The user can preview and approve the change in Runory.
- The final state is visible in UI and audit.

## 7. Free Plan Definition

`v0.4` ships only a free plan.

Free plan should define:

```text
workspace limit
member limit
pack availability
operation limits
demo data availability
MCP/API access level
storage limits if any
support expectations
```

If limits are not technically enforced yet, the website and docs must say so honestly.

Recommended initial posture:

```text
Free during public preview.
Fair-use limits may apply.
Paid plans are not active.
No payment method required.
```

## 11. Exit Criteria For v0.4

Runory `v0.4` is launch-ready when:

- English-first public website and docs are complete enough for self-service evaluation.
- A new user can create and use a free workspace without private help.
- At least one official pack journey works end to end with demo data.
- MCP / Skill / Agent operations 1.0 can safely inspect and customize a workspace.
- Free plan boundaries are clear.
- No billing or Stripe dependency blocks use.
- Release notes and known limitations are public.
