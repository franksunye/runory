# Architecture Review: SaaS Core / Runtime / Business Decoupling

> **Date**: 2026-07-06 (initial), 2026-07-07 (revised)  
> **Version reviewed**: v0.5  
> **Scope**: Three-layer decoupling analysis across `saas_`, `runory_runtime_`, and `runory_business_` table namespaces and their owning source modules. This review is also a complexity-control document: it separates what v0.5/v0.5.1 must harden now from what can remain nice-to-have or future physical modularization.

---

## 0. Review Intent And Decision Frame

This document was created to answer two related questions:

1. **Are we making Runory too complex for the current stage?**
2. **Are the major capabilities decoupled clearly enough that future product work will not drift into accidental coupling?**

The answer is nuanced:

- The **conceptual architecture is not too complex**. SaaS Core, Runtime, and Business are the right three product/technical layers for Runory's long-term ambition as an AI-era business application platform.
- The **current implementation has too little enforcement**. Because everything lives in one physical package and the public barrel exports internals, developers can accidentally bypass the intended layers.
- The **current release should not turn into a package-split project**. v0.5.1 should prioritize interface discipline, command boundaries, Workflow V1 retirement, optional Workflow V2 overlay readiness, demo quality, and product-facing UX. Physical package extraction is a later hardening step.

For v0.5.1, the practical decision rule is:

| Category | Decision rule | Examples |
|----------|---------------|----------|
| Must harden now | Directly affects commercial FSM execution, demo credibility, governed state, or product UX | command catalog, governed fields, Workflow V1 retirement, optional Workflow V2 overlay, forms/work items, assignment/schedule, golden demo, role-focused navigation |
| Guard now, split later | Creates architecture drift if left uncontrolled, but does not require immediate physical package extraction | public barrel filtering, import-boundary tests, raw SQL access rules |
| Nice to have | Useful platform capability, but not needed to prove v0.5.1 commercial FSM execution | catalog rollout control plane, entitlements activation, full lifecycle export/delete UX, external outbox consumer |
| Future modularization | Valuable once the module contracts stabilize | `@runory/saas-core`, `@runory/runtime`, `@runory/business-fsm`, `@runory/business-quote` package split |

Related release specification: [v0.5.1 Commercial FSM Productization, Command Kernel And Optional Workflow Technical Specification](../product/v0.5.1-commercial-fsm-productization-technical-spec.md).

Related SMB closure plan: [v0.5 SMB Dependency Decoupling And Closure Plan](../product/v0.5-smb-dependency-decoupling-and-closure-plan.md).

## 1. Architecture Overview

### 1.1 Physical vs Logical Layering

Runory uses a **single physical package** (`@runory/platform-core`) containing all three logical layers. The Business layer is declarative (catalog YAML + SQL) for module definitions, but also contains business-domain code (FSM commands, quote commands) that has no proper home and currently lives inside `platform-core`. Layer isolation relies on table-prefix conventions and file naming, not compiler-enforced boundaries.

### 1.4 Architectural Layer Classification (Industry-Validated)

Based on industry practice (Salesforce Force.com, Microsoft Power Platform, ServiceNow Now Platform, Zoho Creator), the three-layer model maps to:

| Runory Layer | Industry Concept | Contains |
|--------------|-------------------|----------|
| SaaS Core | Multi-tenant SaaS infrastructure | Identity, auth, tenancy, RBAC, audit, billing, quotas |
| Runtime | PaaS / aPaaS platform layer | Metadata engine, workflow engine, automation engine, form engine, extension engine, dashboard engine, installer |
| Business | SaaS application layer | Domain commands (FSM state machine, quote lifecycle), domain calculations, domain-specific scheduling, declarative module/pack definitions |

Key principle (validated against Salesforce architecture): **Workflow and Automation are platform-level engines (PaaS), not SaaS application features.** Salesforce places Flow (workflow) and Automation in Force.com (PaaS), not in Sales Cloud (SaaS). Runory's placement of `workflow-v2.ts` and `automation.ts` in Runtime is correct. However, this does not mean every Business pack must depend on Workflow. Business execution should be command-first; Workflow should be an optional orchestration overlay.

Conversely, **domain commands** (`fsm-commands.ts`, `quote-commands.ts`, `quote-calculation.ts`) are **Business layer logic** that should not live in the platform package. They encode FSM and Quote domain rules — this is SaaS application code, not PaaS engine code.

## 2. Design Strengths

### 2.1 Business Layer Is Declarative

Business modules are defined as `manifest.yaml` + `install.sql` + optional `demo-data.json`. Adding a new business module requires zero platform-core code changes. The runtime dynamically discovers objects, fields, and views from metadata tables populated at install time.

### 2.2 Governed Change Framework

The `command-runtime.ts` + `governed-fields.ts` combination provides a clean separation:
- **Generic CRUD** (`createRecord`/`updateRecord`) handles draft and descriptive fields (title, notes, description)
- **Named commands** (`completeWorkOrder`, `approveQuote`, etc.) handle lifecycle/financial/snapshot fields with idempotency, optimistic locking, atomic batch (business state + audit + events in one transaction), and before/after snapshots
- `governed-fields.ts` acts as a guard: `updateRecord` calls `assertNotGovernedUpdate()` which rejects attempts to write `status`, `grand_total`, `accepted_at` etc. via generic CRUD, and directs the caller to use the appropriate command

### 2.3 Bridge Layer Is Clean

`apps/cloud/src/lib/auth.ts` (`requireWorkspaceContext`, `requirePlatformAdmin`) and `lib/http.ts` (`successResponse`, `handleError`) form a focused HTTP adaptation layer. All routes go through these for authentication and response wrapping.

### 2.4 DB Client Fully Encapsulated

Zero direct `@libsql/client` imports in `apps/cloud/src/`. All database access flows through `platform-core/db.ts`'s Proxy singleton with `ensureSchema()` lazy migration (cached via `globalThis.__platformSchemaReady`, zero overhead after first call).

### 2.5 Table Namespace Convention

Three-tier prefix system (`saas_` / `runory_runtime_` / `runory_business_`) with environment-variable customization (`SAAS_TABLE_PREFIX`, `BUSINESS_TABLE_PREFIX`, etc.). Module migrations use `{{BUSINESS_TABLE_PREFIX}}` placeholders rendered by `renderSqlWithPrefix()`.

### 2.6 Recursion Guard

`AsyncLocalStorage`-based `_automationSuppress` prevents infinite recursion when automation `create_task` / `update_record` actions trigger `fireAutomationTriggers()` → `createRecord()` → `fireAutomationTriggers()` chains.

### 2.7 Topological Install Ordering

`installer.ts` uses Kahn's algorithm (`topologicalSort()`) to resolve module dependencies before installation, ensuring shared modules (e.g., `runory.company`) are installed before dependents.

## 3. Critical Issues

### 3.1 P0 — `index.ts` Exposes Internal Implementation

**Problem**: `platform-core/src/index.ts` uses `export *` for all 40 modules with zero barrel filtering. The following internal implementation details leak to all consumers:

| Exported Internals | Source | Impact |
|---------------------|--------|--------|
| `TABLES` (resolved physical table names) | `contracts.ts` | Any consumer can construct `saas_users` and write raw SQL |
| `queryAll`, `queryOne`, `execute`, `batch`, `genId`, `now` | `db.ts` | Raw SQL execution available to all layers, bypassing service APIs |
| `runMigrations` | `migrations.ts` | Operational internals exposed |
| `PLATFORM_CONFIG` | `platform-config.ts` | All env names, globalThis keys, header names |
| `renderSchemaSql`, `schemaStatements` | `contracts.ts` | Schema rendering internals |
| `BUSINESS_TABLE_PREFIX`, `businessTable()`, `tablePrefix()` | `contracts.ts` | Physical naming mechanism exposed |

**Root cause**: This is the structural enabler of all cross-layer SQL violations documented below. Because `TABLES` + DB primitives are freely available, developers naturally bypass service functions.

**File**: `/Users/yesun/Code/runory/packages/platform-core/src/index.ts` (lines 1-40)

### 3.2 P1 — Business Logic Misplaced in Platform-Core

**Problem**: 8 business-domain modules (`fsm-commands.ts`, `quote-commands.ts`, `quote-calculation.ts`, `assignment.ts`, `schedule.ts`, `governed-fields.ts`, `command-runtime.ts`, `outbox.ts`) currently live inside `@runory/platform-core` alongside Runtime engine code. These encode FSM and Quote domain rules — they are SaaS application logic, not PaaS platform engine code.

**Impact**: 
- Business logic can directly import Runtime and SaaS Core internals (no barrier)
- Runtime engine code and Business domain code are in the same package, making it impossible to enforce "Business depends on Runtime, never vice versa"
- `governed-fields.ts` registers FSM/Quote governed fields at module load time, creating an implicit Business → Runtime coupling at the package level

**Ideal structure** (deferred to Phase 5):
```
@runory/saas-core       — auth, tenancy, audit, billing
@runory/runtime          — metadata, installer, dashboard, workflow engine, automation engine
@runory/business-core    — governed-fields, command-runtime, outbox (shared governance base)
@runory/business-fsm     — fsm-commands, assignment, schedule
@runory/business-quote   — quote-commands, quote-calculation
```

### 3.3 P1 — Runtime Directly Accesses SaaS Core Tables

Six code paths in the Runtime layer directly read/write `saas_`-prefixed tables, bypassing SaaS Core service APIs:

| File | Lines | Tables Accessed | Bypassed Service API |
|------|-------|----------------|---------------------|
| `installer.ts` | 383, 394, 407, 413, 423, 429 | `saas_users`, `saas_auth_identities`, `saas_workspace_memberships` | `tenancy.ts` / `memberships.ts` |
| `metadata.ts` | 149, 161, 167, 193, 200 | `saas_workspaces` | `tenancy.ts` (`getWorkspace`, `createWorkspace`) |
| `dashboard.ts` | 491 | `saas_audit_logs` | `audit-service.ts` |
| `command-runtime.ts` | 164 | `saas_audit_logs` | `audit-service.writeAuditEvent` (duplicated implementation) |
| `cloud/lib/auth.ts` | 199-203 | `saas_organization_memberships`, `saas_users` | `tenancy.authorizeOrganization` (re-implemented) |

**Most severe**: `installer.ts` `seedPackDemoData()` directly INSERTs users, auth identities, and workspace memberships. This should call `tenancy.ts` / `memberships.ts` / `auth.ts` service functions.

### 3.4 P1 — SaaS Core Reverse-Depends on Runtime

**Problem**: `lifecycle.ts` (SaaS Core) directly imports `exportWorkspace` from `audit.ts` (Runtime), and uses `BUSINESS_TABLE_PREFIX` to pattern-match business tables for deletion.

This is the only SaaS Core → Runtime code-level reverse dependency. It breaks the expected dependency direction (Runtime → SaaS Core, never SaaS Core → Runtime).

**File**: `/Users/yesun/Code/runory/packages/platform-core/src/lifecycle.ts` (line 12)

### 3.6 P2 — Runtime Internal Coupling

| Problem | Location | Impact |
|---------|----------|--------|
| `fireAutomationTriggers()` defined in `metadata.ts` instead of `automation.ts` | `metadata.ts:11-54` | metadata.ts carries orchestration responsibility that belongs to the automation module |
| `forms-v2.ts` `completeWorkItem()` is a simplified private impl that doesn't advance workflow | `forms-v2.ts:1035-1054` | Work items completed via forms don't create next-step items or write workflow events, unlike `workflow-v2.ts` `completeWorkItem()` (line 1099) |
| `forms-v2.ts` `createServiceReportProjection()` directly INSERTs business table | `forms-v2.ts:1169` | Bypasses `metadata.ts` `createRecord()` abstraction |

### 3.7 P0 — Product-Facing Module Boundary Risk

The code-level layers can be technically correct while the product still feels fragmented. v0.5.1 must avoid exposing internal engine concepts as daily-work destinations.

| Risk | Bad product outcome | Required correction |
|------|---------------------|---------------------|
| Forms becomes a standalone technical module | Technicians and dispatchers must understand form libraries before doing work | Forms are managed under configuration, but executed contextually inside Quote/FSM work |
| Workflow becomes a top-level business module | Users see workflow instances instead of obligations and actions | Workflow V2 owns orchestration only when enabled; UI shows My Work, approvals, returns, and next actions |
| Calendar/map becomes an isolated module | Dispatching splits from work orders, assignments, and visits | Planning is the product surface; calendar/timeline/map are views over assignment and schedule commands |
| Quote Approvals becomes a special page pattern | Every module grows its own approval page | Approval is a Work Item pattern, workflow-backed only when configured; modules expose contextual actions and role queues |
| Business status duplicates workflow status | Records, workflows, dashboards, and lists disagree | One authoritative state owner per concern; projections are read-only and command-updated |

This product-facing boundary is part of architecture, not only UX. If the layer boundaries are hidden inside technical modules but the user experience exposes them as unrelated products, Runory will still feel over-complex.

### 3.8 P0 — Workflow Must Be Decoupled From Pack Execution

The review's complexity-control conclusion is that Workflow must not be the hidden dependency that every SMB business pack needs before it can run. SMB customers often need simple, opinionated operations more than a process designer.

The required architecture is:

```text
Business Pack
  ├─ object model
  ├─ lifecycle state
  ├─ named commands
  ├─ permissions / assignment policy
  ├─ forms and evidence policy
  └─ audit / domain events

Optional Workflow Overlay
  ├─ cross-role obligations
  ├─ approvals
  ├─ returns / resubmissions
  ├─ SLA / due dates
  └─ complex orchestration
```

Therefore:

- a Pack MUST be executable without starting a workflow instance;
- every governed lifecycle transition MUST have a named command;
- Workflow MAY create Work Items and determine the next obligation;
- Workflow MUST call or authorize named commands, not mutate business state directly;
- My Work MAY contain command-generated or workflow-generated Work Items;
- v0.5.1 MUST retire active Workflow V1 execution, but MUST NOT make Workflow V2 mandatory for all Pack execution.

## 5. Repair Roadmap

### Phase 0 — Lock v0.5.1 Product Architecture Before Deep Refactor (P0)

**Goal**: prevent this architecture work from becoming a large package-split project before v0.5.1 is commercially demonstrable.

1. Treat [v0.5.1 commercial FSM specification](../product/v0.5.1-commercial-fsm-productization-technical-spec.md) as the release contract.
2. Keep physical package splitting out of v0.5.1 unless a small extraction is necessary to enforce a release-blocking boundary.
3. Require every governed lifecycle transition to go through a named command, including tests and demo seed/migration scripts.
4. Require business packs to run a command-only core lifecycle without workflow.
5. Require Workflow V2 only when a scenario is workflow-backed, such as approval/return/resubmit or cross-role human obligations.
6. Hide technical engines from daily-work navigation: Workflow, Forms, Outbox, Migration, and raw Administration belong under management/configuration unless the actor is an admin.

### Phase 1 — Close the Export Surface (P0)

**Goal**: Prevent structural cross-layer SQL by restricting what `index.ts` exposes.

1. Move `db.ts`, `contracts.ts`, `migrations.ts`, `platform-config.ts` exports to `./internal` sub-path
2. Public barrel (`index.ts`) exports only service functions and types:
   - SaaS Core: `resolveSession`, `requestOtp`, `verifyOtp`, `provisionWorkspaceTenant`, `authorizeWorkspace`, ...
   - Runtime: `getRecords`, `createRecord`, `installPack`, `startWorkflowV2`, ...
   - Types: `Principal`, `RequestContext`, `ObjectDefinition`, `FieldDefinition`, ...
3. Add architecture guard tests: enforce that non-`internal` files cannot import `TABLES` or `query`/`execute`

**Impact**: High. This alone will cause compile-time errors for all P1 violations, forcing them to be fixed. This is recommended for v0.5.1 because it is a boundary-enforcement change, not a broad product expansion.

### Phase 2 — Eliminate Cross-Layer Table Access (P1)

4. `installer.ts` `seedPackDemoData()`: replace direct `saas_users`/`saas_auth_identities`/`saas_workspace_memberships` INSERTs with calls to `tenancy.ts` / `memberships.ts` / `auth.ts` service functions
5. `metadata.ts` workspace operations: delegate to `tenancy.ts` (`getWorkspace`, `createWorkspace`)
6. `command-runtime.ts` and `dashboard.ts`: replace direct `saas_audit_logs` access with `audit-service.writeAuditEvent` / query functions
7. `lifecycle.ts`: refactor `exportWorkspace` to an interface injected by Runtime, or move the export logic up to Runtime layer entirely
8. `cloud/lib/auth.ts`: replace raw SQL `authorizeOrganization` with platform-core exported function

### Phase 3 — Clean API Route Layer (P2)

9. Extract `timeline/route.ts` query logic into `platform-core` `getTimelineEntries()` domain function; fix the `businessTableForSubject` prefix bug
10. Add `platform-core` domain functions for remaining 5 routes with raw SQL (`getAdminStats`, `updateUserPreferences`, `getAttachment`, `getWorkflowDefinitionsWithVersions`, `updateOutboxMessages`)

### Phase 4 — Runtime Internal Cleanup (P2)

11. Move `fireAutomationTriggers()` from `metadata.ts` to `automation.ts`
12. Make `forms-v2.ts` `completeWorkItem()` call `workflow-v2.ts` `completeWorkItem()` via dynamic import (avoiding static circular dependency)
13. Replace `forms-v2.ts` `createServiceReportProjection()` direct INSERT with `createRecord()` call

### Phase 5 — Separate Business Logic from Platform-Core (Future, not v0.5.1)

14. Extract business-domain modules from `platform-core` into separate packages:
    - `@runory/business-core` — `governed-fields.ts`, `command-runtime.ts`, `outbox.ts` (shared governance base)
    - `@runory/business-fsm` — `fsm-commands.ts`, `assignment.ts`, `schedule.ts`
    - `@runory/business-quote` — `quote-commands.ts`, `quote-calculation.ts`

## 7. Assessment Summary

| Dimension | Score | Notes |
|-----------|-------|-------|
| **Layer intent clarity** | Good | Three-tier prefix + file naming + documentation (`07-saas-core-boundaries.md`, `database-namespaces.md`) |
| **Physical isolation** | Poor | Single package, `export *` with no barrel filtering |
| **Business logic placement** | Fair | Declarative catalog is excellent; domain command code misplaced in platform-core |
| **Table namespace discipline** | Fair | Convention exists but 6+ cross-layer violations; no guard tests |
| **Dependency direction** | Fair | 1 reverse dependency (lifecycle → audit), multiple Runtime → SaaS Core SQL bypasses |
| **Business layer design** | Excellent | Fully declarative, zero-code module addition, `businessTable()` helper |
| **Governed change framework** | Good | command-runtime + governed-fields provides clean separation of CRUD vs domain commands |
| **Bridge layer (cloud)** | Good | `lib/auth.ts` + `lib/http.ts` are clean and focused |
| **DB encapsulation** | Excellent | 100% through `db.ts` Proxy, zero direct `@libsql/client` in runtime |
| **Guard tests** | Poor | `architecture-tests.test.ts` only checks field/error invariants; no import-boundary or table-access enforcement |

**Overall**: The architecture is well-conceived with clear intent, but enforcement is lacking. For v0.5.1, the highest-leverage path is not to split every package immediately; it is to harden the boundaries that protect commercial FSM execution:

1. **Close the `index.ts` export surface** (P0) — makes cross-layer SQL violations visible and enforceable.
2. **Add import/table-access guard tests** — prevents the same coupling from returning quietly.
3. **Make named commands the only business state-change path** — prevents workflow, automation, UI, SQL, or agents from becoming duplicate lifecycle owners.
4. **Keep technical engines out of daily-work navigation** — Forms, Workflow, Outbox, and Migration are management/configuration surfaces, not separate products for technicians or sales users.
5. **Keep Workflow V2 optional but production-safe** — retire active V1, support workflow-backed obligations, and prove at least one command-only Pack journey.
6. **Defer physical package extraction** until v0.5.1 command contracts, workflow contracts, and product surfaces are stable.
