## 1. Purpose

The v0.5 Command migration corrected a real consistency problem: governed
business changes previously depended on feature code remembering every related
Assignment, Schedule, Workflow, Form, event, and audit update.

The architectural direction is accepted. This document records the remaining
implementation debt so the team can absorb the current model before changing
the kernel again. It is a v0.6 stabilization backlog, not authorization for an
immediate rewrite.

## 2. Frozen v0.5 Baseline

The following baseline is considered working and must remain compatible:

```text
37 executeCommand entry points
26 business Module-owned Command Contracts
11 Platform Service-owned Command Contracts (Workflow 6 + Forms 5)
workspace-scoped, versioned Contract snapshots
atomic semantic capability providers
idempotency, optimistic concurrency, audit and Domain Events
663 Platform Core regression tests passing at migration completion
```

Until v0.6 stabilization begins, changes to the Command kernel are limited to:

- security or tenant-isolation defects;
- proven data corruption or incomplete transaction defects;
- production-blocking compatibility or migration defects;
- the minimum change required to add a necessary v0.5 business Command.

Code organization, naming, generalization, and speculative extensibility alone
are not reasons to reopen the kernel during the digestion period.

## 3. Decisions That Must Be Preserved

These are platform invariants, not TODO items:

1. Named Commands remain the only entry point for governed business change.
2. UI, API, mobile, Workflow, Automation, MCP, Skills, and agents call the same
   Command rather than coordinating generic updates.
3. Business Modules and Platform Services are distinct Contract sources.
4. A Workspace resolves the exact Contract snapshot provisioned for it.
5. Cross-capability authoritative writes use declared consistency and a
   compatible Provider.
6. Workflow and Automation orchestrate Commands; they do not replace Command
   invariants or transactions.
7. Projections such as My Work, Planning, and Activity remain read models, not
   competing owners of lifecycle state.

## 4. Why Stabilization Is Still Required

The model is sound, but the first migration concentrated too many
responsibilities in a small number of implementation files. In addition, some
Contract declarations are validated structurally but are not yet fully
executable at Runtime.

The v0.6 objective is therefore:

> Preserve the semantics and compatibility of v0.5 while reducing accidental
> implementation complexity and closing the remaining enforcement gaps.

This is a refactor of boundaries and guarantees, not a redesign of the business
model.

## 5. v0.6 TODO Backlog

### A. Inventory and safety net — do first

- [ ] Generate a repository check proving every `executeCommand` key maps to
      exactly one provisionable Contract.
- [ ] Detect duplicate Command ownership across Module and Platform Service
      manifests.
- [ ] Detect declared Commands without a callable implementation.
- [ ] Produce a generated inventory of Command, source, aggregate, transition,
      permission, effects, emitted events, and Provider version.
- [ ] Add compatibility fixtures for existing workspace snapshots and upgrades.
- [ ] Record Contract source/version and resolved Provider versions in Command
      execution diagnostics.

Exit gate: the current 37-Command baseline is machine-auditable before files are
reorganized.

### B. Split implementation responsibilities — behavior-preserving

- [ ] Split the current Command Contract implementation by responsibility:

```text
command-contracts/
  schema-and-types
  registry
  workspace-resolution
  validation
  runtime-plan
  diagnostics
  providers/
    assignment
    scheduling
    workflow
    forms
    quote
    fsm
```

- [ ] Keep Provider implementations with their bounded context or a clearly
      named integration adapter.
- [ ] Remove circular or implicit provider-registration dependencies.
- [ ] Define a single public API for Contract loading, validation, snapshotting,
      resolution, and execution planning.
- [ ] Keep this slice behavior-neutral; no lifecycle or permission changes.

Exit gate: the full regression suite and snapshot compatibility fixtures pass
without changing observable Command results.

### C. Close Runtime enforcement gaps

- [ ] Validate the actual source state against `transition.from` before writes.
- [ ] Validate the resulting state against `transition.to` after handler and
      Provider preparation.
- [ ] Replace critical free-text postconditions with typed, executable
      assertions; retain prose only as documentation.
- [ ] Enforce Contract permission and actor policy through one Runtime
      authorization boundary.
- [ ] Define a consistent concurrency policy for Form Submission revision,
      draft promotion, return, and acceptance.
- [ ] Verify Provider cardinality against prepared and affected records rather
      than only the presence of SQL statements.
- [ ] Add negative tests proving Workflow, Automation, and generic CRUD cannot
      bypass governed transitions.

Exit gate: a valid Contract is not merely descriptive; Runtime can reject an
implementation that violates its transition, authorization, effect, event,
audit, or critical postcondition semantics.

### D. Productize provisioning and upgrades

- [ ] Sync Platform Service snapshots only during workspace provisioning,
      Platform Service upgrade, or explicit repair—not every ordinary Module
      installation.
- [ ] Add an idempotent backfill/repair operation with dry-run diagnostics.
- [ ] Persist and expose workspace Contract inventory for support and
      administration.
- [ ] Add upgrade impact analysis for changed Commands, removed Providers, and
      incompatible Module requirements.
- [ ] Complete workspace backfill, then remove the two legacy process-level
      static Contract bridges.
- [ ] Define uninstall behavior when another installed source consumes a
      capability or Command owned by the source being removed.

Exit gate: provisioning, upgrade, repair, and uninstall behavior is explicit,
observable, and safe for existing Workspaces.

### E. Developer experience and governance

- [ ] Publish a short “How to add a governed Command” guide.
- [ ] Provide a manifest template and generated Contract tests.
- [ ] Add Catalog/CI gates for capability closure and breaking Contract changes.
- [ ] Add architecture tests preventing cross-domain physical SQL from leaking
      into unrelated handlers.
- [ ] Document when an effect belongs inside one aggregate boundary versus a
      semantic Provider.
- [ ] Add concise operational errors that identify Command, workspace source,
      missing capability, and remediation.

Exit gate: a Module developer can add a correct Command without understanding
the entire kernel or copying an existing handler blindly.

### F. Agent-first Workspace customization — parallel design track

This track protects Runory's original customization goal. It runs alongside
Command stabilization but must not be absorbed into the Command kernel
refactor.

- [ ] Define three explicit customization tiers:
      workspace metadata and presentation, configurable process composition,
      and governed extension development.
- [ ] Keep ordinary field, view, form, relation, navigation, and small custom
      object changes in Workspace Metadata rather than official manifests.
- [ ] Let configurable Workflow change steps, assignees, conditions, required
      forms, SLA, and notifications while invoking existing Commands at
      governed business boundaries.
- [ ] Require an Extension Module, Command Contract, and compatible Provider
      only when customization changes a core invariant or cross-aggregate
      authoritative transaction.
- [ ] Design Agent-facing customization operations for draft, validation,
      impact preview, publish, audit, rollback, and version history.
- [ ] Ensure end users and Agents do not need to edit YAML manifests or know
      physical table names.
- [ ] Define the safe default capabilities of a small custom object: fields,
      relations, CRUD, permissions, views, search, audit, forms, and Workflow
      association.
- [ ] Evaluate a versioned Custom Object State Machine Runtime for user-defined
      lifecycles without introducing an unrestricted generic status update.
- [ ] Add architecture tests proving ordinary Workspace extensions survive
      Module upgrades and cannot override protected official invariants.
- [ ] Provide Agent-readable discovery APIs for available objects, fields,
      Commands, current allowed actions, effects, and customization limits.

Exit gate: a user or authorized Agent can implement common SME customization
without code or manifest editing, while changes to authoritative business
semantics still pass the governed extension boundary.

## 6. Explicit Non-Goals

The stabilization effort must not:

- replace Commands with Workflow or Automation;
- introduce distributed transactions for local atomic facts;
- turn every ordinary field update into a Command;
- create one generic status-transition Command;
- move physical SQL into manifests;
- redesign all v0.5 FSM, Quote, Workflow, or Form lifecycles;
- add abstraction solely to support hypothetical future Packs;
- require ordinary Workspace customization to publish a business Module;
- expose physical manifests or SQL as the primary Agent customization API;
- block v0.6 product work on non-critical naming or formatting cleanup.

## 7. Proposed Execution Order

```text
Slice 1  Inventory, generated coverage checks, compatibility fixtures
Slice 2  Behavior-preserving file and Provider boundary split
Slice 3  Runtime transition, authorization, and postcondition enforcement
Slice 4  Provisioning, upgrade, repair, uninstall, legacy bridge removal
Slice 5  Agent-first Workspace customization contract and management APIs
Slice 6  SDK guidance, Catalog gates, diagnostics, final architecture review
```

Each slice must be independently reviewable and keep the product runnable. Do
not combine this stabilization with a large FSM feature or UI redesign.

## 8. Start Criteria

Begin the v0.6 stabilization only when:

- the team can explain Command, Contract, Runtime, Provider, snapshot, Workflow,
  and projection ownership in consistent terms;
- at least one v0.5 acceptance cycle has exercised the architecture without
  direct database repair;
- the generated inventory work is scoped and the existing snapshot upgrade
  fixtures are ready;
- a named owner and time box are assigned.

Start earlier only for a security, consistency, migration, or production
reliability issue. Otherwise, preserve the v0.5 baseline and collect evidence
in this backlog.

## 9. Completion Criteria

The v0.6 stabilization is complete when:

- all governed Commands have exactly one source and one Runtime plan;
- critical transition and postcondition semantics are executable;
- permissions are enforced consistently at the Runtime boundary;
- Provider selection and versions are diagnosable historically;
- the legacy static bridge is removed;
- workspace provisioning and upgrades do not perform surprising global repair;
- core files have clear bounded responsibilities;
- existing v0.5 journeys remain compatible and pass without data repair;
- the architecture can be explained to a new developer using one concise
  lifecycle example;
- common field, form, Workflow, and small custom-object changes remain
  available through versioned Workspace customization APIs without editing
  official manifests.
