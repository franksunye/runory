## Purpose

This register is the single handoff point for work deliberately excluded from
v0.6. It prevents accepted deferrals from becoming an unowned backlog and
prevents rejected abstractions from silently returning in a later release.

A target version is an admission point, not authorization to implement the
item automatically. Every item must still satisfy its trigger and the target
release's product question.

## Closed Non-goals

The following ideas were considered during v0.6 and are not carried into a
future-version backlog:

- a universal callback framework over billing, payments, voice, and email;
- a distributed queue or workflow engine replacing the bounded Outbox;
- runtime Provider auto-discovery;
- a permanent load-testing platform before a hosted topology exists;
- a standalone upgrade analyzer without a real upgrade mutation consumer.

Reopening one of these decisions requires new product evidence and an ADR. A
later version number alone is not sufficient.

## v0.7 Discussion Boundary

The v0.7 discussion receives `V06-D01`, `V06-D02`, and product candidates
`V06-P01–V06-P05`:

- `V06-D01` belongs to implementation readiness and safe upgrade operations;
- `V06-D02` is conditional architecture support for the capabilities actually
  admitted to v0.7.

Agent discovery and public SDK generation stay in v0.8. Hosted performance
evidence stays with deployment operations. `V06-P06` moves to the v0.9
repeatability gate. None of these items should be counted as a v0.7 product
feature until the v0.7 scope decision explicitly admits it.
