## 1. Decision

v0.5 is frozen as the implemented feature baseline. Remaining broad commercial
benchmark, pilot, and release-closure work no longer blocks that baseline and
moves into the v0.6 real-customer validation stream.

v0.6.0 is an architecture stabilization milestone. It precedes new Lead,
recurring-service, invoicing, operational-inbox, and customer-access product
work.

The objective is:

> Preserve observable v0.5 business behavior while making the shared runtime
> easier to reason about, extend, test, upgrade, diagnose, and operate.

This is not a large rewrite. Work proceeds in independently reviewable slices,
each with compatibility evidence and a runnable product.

## 2. Why Stabilization Is Required

v0.5 introduced or materially expanded:

- governed Commands, Contract snapshots, semantic Providers, Workflow, and
  Forms;
- Quote, planning, field execution, row-level visibility, and composable
  business roles;
- voice intake and external conversation callbacks;
- merchant payment, Stripe callbacks, reconciliation, and refund behavior;
- Runory SaaS subscription billing and Organization entitlement updates;
- Agent-facing operations, scheduled-agent direction, and live product proof;
- additional projections, background work, audit, Outbox, and performance
  requirements.

Each capability has a bounded design, but their shared infrastructure has not
yet received one end-to-end architecture review. Without consolidation, new
v0.6 features would amplify duplicated provider adapters, authorization paths,
webhook handling, event semantics, provisioning behavior, and diagnostics.

## 3. Architecture Outcomes

v0.6.0 must leave the following properties demonstrably true.

### 3.1 Small and explicit Core

- Platform Core owns generic runtime guarantees, not FSM-, payment-, voice-, or
  billing-specific policy.
- Platform Services, Business Modules, Provider Adapters, and projections have
  explicit ownership and dependency directions.
- Cross-domain integration occurs through named contracts rather than physical
  table knowledge or circular imports.

### 3.2 One governed mutation model

- UI, mobile, Workflow, Automation, MCP, Agents, webhooks, and background jobs
  use the same governed Command boundary for authoritative business change.
- Command transition, permission, concurrency, effects, events, audit, and
  critical postconditions are executable guarantees.
- Generic CRUD cannot bypass protected lifecycle or commercial invariants.

### 3.3 One external-integration model

- Voice, merchant payment, subscription billing, and future providers share a
  small integration kernel for verified ingress, idempotency, replay, Outbox,
  delivery attempts, reconciliation, diagnostics, and secret references.
- Provider payloads remain outside canonical business models.
- Provider adapters translate external events into governed application
  intents; they do not become business truth owners.

### 3.4 Consistent identity and authorization

- Interactive users, portal users, Agents, API keys, webhooks, and background
  jobs resolve an explicit server-derived Principal and Workspace or
  Organization scope.
- Actor policy, tenant isolation, row visibility, and audit attribution are
  enforced at shared boundaries.
- Merchant payment and business records remain Workspace-scoped; SaaS
  subscription and entitlement state remains Organization-scoped.

### 3.5 Safe lifecycle management

- Workspace provisioning, Module and Platform Service installation, snapshot
  resolution, upgrade, repair, and uninstall are explicit and observable.
- Existing workspace snapshots and released schemas remain compatible or have
  tested migrations.
- Managed Workspace Extensions survive supported upgrades and cannot replace
  protected official invariants.

### 3.6 Operable runtime

- Logs, metrics, traces, audit, Domain Events, Outbox delivery, webhook receipt,
  reconciliation, and projection lag use stable correlation identifiers.
- Failure is visible and retryable without duplicating authoritative change.
- Performance budgets cover interactive Commands, webhook ingestion,
  projection freshness, and operational queries.

## 4. Workstreams

### A. Inventory and dependency map

- Generate the current inventory of Commands, Contracts, Providers, events,
  projections, webhook endpoints, background jobs, Platform Services, Business
  Modules, and external adapters.
- Map source ownership, tenant scope, transaction boundary, dependencies,
  emitted events, and operational diagnostics.
- Detect duplicate ownership, undeclared implementations, orphan declarations,
  cross-domain SQL, and dependency cycles.
- Record the current schema and workspace snapshot compatibility matrix.

Exit gate: the implemented v0.5 baseline is machine-auditable before structural
changes begin.

### B. Package and bounded-context boundaries

- Confirm the public responsibilities of Platform Core, SaaS Core, Platform
  Services, Business Modules, Provider Adapters, and Cloud delivery code.
- Split concentrated registries and handlers by responsibility without
  changing behavior.
- Define dependency rules and architecture tests.
- Remove feature-specific policy from generic runtime code where ownership is
  clear and migration is behavior-preserving.

Exit gate: dependency direction is documented and mechanically enforced; the
full compatibility suite still passes.

### C. Command and transaction kernel

Execute the detailed
[v0.6 Command Architecture Stabilization TODO](./v0.6-command-architecture-stabilization-todo.md):

- generated coverage and snapshot compatibility;
- responsibility split and Provider boundaries;
- Runtime transition, authorization, cardinality, and postcondition
  enforcement;
- provisioning, repair, upgrade, uninstall, and legacy bridge removal;
- SDK guidance, Catalog gates, and diagnostics.

Exit gate: a Contract is executable rather than descriptive, and every
governed mutation channel shares the same enforcement path.

### D. Integration, event, and asynchronous-effects kernel

- Define common verified-ingress and delivery contracts without forcing all
  providers into one business abstraction.
- Standardize webhook receipt identity, signature result, idempotency key,
  provider event identity, processing state, replay policy, and retention.
- Standardize Outbox messages, delivery attempts, backoff, dead-letter or
  intervention state, and reconciliation jobs.
- Separate Domain Events, integration events, audit records, and rebuildable
  projections.
- Prove duplicate, reordered, delayed, and replayed provider events cannot
  corrupt business state.

Exit gate: voice, merchant payment, and subscription billing use compatible
operational primitives while retaining separate domain authority.

### E. Identity, authorization, and tenancy

- Inventory every Principal source and eliminate implicit trusted identity
  paths.
- Centralize RequestContext and Principal derivation for HTTP, MCP, Agent,
  webhook, and job execution.
- Verify Organization-versus-Workspace ownership for billing, entitlement,
  provider configuration, business records, audit, and secrets.
- Extend cross-tenant and row-visibility negative tests to new callbacks,
  projections, and support operations.

Exit gate: no execution channel can select its own tenant or actor authority
without server-side validation.

### F. Provisioning, versioning, and extensibility

- Define version ownership for Core, Platform Services, Modules, Packs,
  Templates, Contracts, Providers, Extensions, and schemas.
- Make provisioning and repair idempotent, dry-runnable, diagnosable, and
  compatible with existing workspaces.
- Add upgrade impact reports for Contracts, Providers, permissions, events,
  schemas, and Extensions.
- Preserve the three customization tiers: workspace metadata, configurable
  process composition, and governed extension development.

Exit gate: upgrades and repairs preserve data and supported customization, and
incompatible changes fail before mutation.

### G. Data, reliability, performance, and observability

- Review transaction boundaries, indexes, pagination, retention, projection
  rebuilds, and reconciliation queries for the v0.5 feature set.
- Establish correlation across request, Command, audit, event, webhook, Outbox,
  provider, job, and projection records.
- Run empty-database and previous-release migration tests.
- Run tenant-isolation, concurrency, recovery, replay, and performance
  scenarios against production-shaped data.
- Define operator-visible diagnostics and repair procedures for each durable
  subsystem.

Exit gate: the active performance budgets pass and every durable asynchronous
flow has detection, retry, reconciliation, and support evidence.

### H. Developer and Agent experience

- Publish concise extension guides for Commands, Providers, integrations,
  projections, and custom objects.
- Generate test fixtures and architecture checks from declared contracts.
- Expose Agent-readable discovery for objects, allowed actions, effects,
  customization tiers, and constraints.
- Keep physical SQL, secrets, and internal manifests out of ordinary Agent and
  end-user customization flows.

Exit gate: a developer or authorized Agent can extend a supported boundary
without copying internal implementation or bypassing governance.

## 5. Execution Slices

```text
Slice 0  Freeze v0.5 behavior; generate inventory and compatibility fixtures
Slice 1  Lock package ownership, dependency rules, and architecture tests
Slice 2  Stabilize Command, Contract, Provider, and transaction boundaries
Slice 3  Unify integration ingress, Outbox, replay, and reconciliation primitives
Slice 4  Consolidate Principal, authorization, tenancy, and audit attribution
Slice 5  Productize provisioning, upgrades, repair, uninstall, and Extensions
Slice 6  Close observability, recovery, migration, and performance gates
Slice 7  Publish SDK/Agent guidance and complete the architecture review
```

Slices may overlap only when their ownership and compatibility fixtures are
already explicit. New v0.6 business features may begin after the boundaries
they depend on have passed their slice gates; the entire product roadmap does
not need to wait for unrelated cleanup.

## 6. v0.5 Closure Carried Into v0.6

The former v0.5 Commercial Benchmark and Pilot Gate becomes v0.6 validation
input:

- XLink-derived waterproof-repair scenario;
- neutral HVAC, plumbing, landscaping, pool, and installation-shaped
  scenarios;
- role-separated browser, mobile, voice, and payment acceptance;
- concurrency, recovery, isolation, webhook, and performance evidence.

These scenarios first establish behavior-preservation fixtures for v0.6.0 and
then contribute to the v0.6 real-customer validation gate. They do not justify
changing v0.5 behavior during the stabilization refactor.

## 7. Change Rules During v0.6.0

Allowed:

- behavior-preserving boundary and dependency refactors;
- security, isolation, consistency, migration, and reliability corrections;
- executable enforcement of already-declared invariants;
- diagnostics, compatibility fixtures, architecture tests, and repair tooling;
- the minimum migration required by an approved boundary correction.

Requires an ADR and explicit product review:

- changing an authoritative aggregate or lifecycle owner;
- moving data between Organization and Workspace scope;
- changing a public Command, event, Extension, or provider contract
  incompatibly;
- replacing the database, transaction model, deployment posture, or Catalog
  lifecycle;
- introducing a new generic abstraction that subsumes multiple bounded
  contexts.

Not allowed inside stabilization:

- a big-bang rewrite;
- combining architecture slices with broad FSM UI redesign;
- turning Workflow, Automation, or Agents into business truth owners;
- provider-specific fields in canonical business aggregates;
- speculative abstractions without two implemented consumers and a current
  product need;
- breaking existing Workspace Extensions without a migration and impact
  report.

## 8. Definition of Done

v0.6.0 is complete when:

1. the generated architecture inventory has no unexplained ownership or
   implementation gaps;
2. package and dependency rules are protected by architecture tests;
3. existing workspace snapshots and previous-release schemas pass upgrade and
   repair fixtures;
4. governed transitions cannot be bypassed through UI, generic CRUD, Workflow,
   Automation, MCP, Agent, webhook, or job paths;
5. provider callbacks and asynchronous effects pass duplicate, replay,
   reordering, recovery, and reconciliation tests;
6. Principal, tenant scope, permission, row visibility, and audit attribution
   pass the cross-channel negative matrix;
7. performance and projection-freshness budgets pass on production-shaped
   scenarios;
8. v0.5 canonical journeys pass without observable regression;
9. architecture authorities, ADRs, SDK guidance, runbooks, and release evidence
   reflect the implemented boundaries;
10. the architecture review records remaining debt with an owner, target
    milestone, and reason it does not block v0.6 product work.
